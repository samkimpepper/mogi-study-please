# 폰 노이만 구조가 혁신적이었던 이유

> [!question]
> 내가 물어본 질문:
>
> - 폰 노이만 구조가 왜 혁신적인가?
> - 프로그램도 데이터처럼 다룬다는 말은 함수 자체를 파라미터로 넘기는 것과 비슷한가?
> - 조건분기를 무제한 허용한다는 말은 무슨 뜻인가?
> - 변수에 값을 지정하듯이 다음 실행 위치를 지정할 수 있다는 뜻인가?

> [!summary]
> 결론:
>
> - 폰 노이만 구조의 혁신은 하드웨어를 바꾸지 않고, 메모리에 저장된 프로그램만 바꿔서 같은 기계가 여러 일을 하게 만든 데 있다.
> - 프로그램도 데이터처럼 메모리에 저장되므로, CPU는 명령어를 메모리에서 가져와 실행한다.
> - 조건분기는 CPU의 `PC`가 가리키는 다음 명령어 주소를 바꾸는 방식으로 이해할 수 있다.
> - 그래서 실행 흐름은 일직선으로 고정되지 않고, 조건에 따라 앞으로도 뒤로도 이동할 수 있다.

## 먼저 잡을 한 줄 정의

> [!info]
> 폰 노이만 구조는 프로그램과 데이터를 같은 메모리에 저장하고, CPU가 메모리에서 명령어를 가져와 순서대로 실행하는 컴퓨터 구조다.

> [!info]
> 핵심은 "기계 자체를 바꾸는 것"이 아니라 "메모리에 들어 있는 프로그램을 바꾸는 것"으로 다른 일을 하게 만드는 데 있다.

## 왜 혁신적이었나

예전 계산 장치는 특정 작업을 하도록 하드웨어 구조가 강하게 정해진 경우가 많았다.

```text
특정 계산용 기계
-> 다른 일을 하려면 배선이나 구조를 바꿔야 함
```

폰 노이만 구조는 이 생각을 바꿨다.

```text
같은 하드웨어
+ 다른 프로그램
-> 다른 계산 수행
```

즉 컴퓨터가 단순한 고정 계산 장치가 아니라, 프로그램을 읽고 실행하는 범용 장치가 된 것이다.

> [!example]
> 백엔드 관점에서 보면:
>
> ```text
> 하드코딩된 서버
> -> 주문 처리만 가능
> -> 다른 기능을 하려면 코드를 고쳐 다시 배포해야 함
>
> 프로그램 저장식 컴퓨터
> -> 같은 CPU와 메모리
> -> 어떤 프로그램을 올리느냐에 따라 웹 서버, DB, 컴파일러, 게임이 될 수 있음
> ```

## 프로그램도 데이터처럼 다룬다는 뜻

폰 노이만 구조에서는 메모리에 데이터만 저장하는 것이 아니다.

```text
메모리
├─ 데이터
└─ 명령어, 즉 프로그램
```

CPU는 메모리의 특정 주소에서 명령어를 가져와 실행한다.

```text
100번지: 3
101번지: 4

200번지: LOAD 100
201번지: ADD 101
202번지: STORE 102
```

여기서 `100`, `101`번지는 데이터처럼 쓰이고, `200`, `201`, `202`번지는 명령어처럼 쓰인다.

하지만 둘 다 결국 메모리에 저장된 비트다. 차이는 CPU가 그것을 어떤 맥락에서 읽고 해석하느냐에 있다.

## 함수 파라미터 비유

> [!info]
> 함수를 파라미터로 넘기는 것은 "코드를 값처럼 다룬다"는 고수준 언어의 예시다.

예를 들어 JavaScript에서는 함수를 값처럼 넘길 수 있다.

```javascript
function runLater(fn) {
  fn();
}

runLater(() => {
  console.log("나중에 실행됨");
});
```

여기서 `fn`은 값처럼 전달되지만, 실제로는 실행 가능한 코드다.

```text
코드인데
값처럼 변수에 담고
인자로 넘기고
나중에 실행한다
```

이 감각은 폰 노이만 구조의 "프로그램도 데이터처럼 다룬다"와 닮아 있다.

> [!warning]
> 다만 함수 파라미터는 폰 노이만 구조 그 자체의 직접 예시는 아니다.
> 더 정확히는, 명령어가 메모리에 데이터와 같은 방식으로 저장될 수 있다는 하위 구조 위에서 가능한 고수준 프로그래밍 패턴에 가깝다.

## 임의 접근 메모리가 중요한 이유

임의 접근 메모리는 주소를 이용해서 원하는 위치에 바로 접근할 수 있는 메모리다.

```text
100번지 읽기
200번지 읽기
105번지 읽기
```

처럼 순서대로만 읽는 것이 아니라, 주소를 지정해서 필요한 위치를 읽을 수 있다.

이것이 중요한 이유는 CPU가 다음에 실행할 명령어 위치를 바꿀 수 있기 때문이다.

만약 명령어 저장소가 테이프처럼 앞으로만 순서대로 읽는 구조라면, 갑자기 `200번지`로 이동하기 어렵다. 하지만 주소 기반 메모리라면 다음처럼 할 수 있다.

```text
다음 명령어는 103번지가 아니라 200번지에서 읽어.
```

## 조건분기를 무제한 허용한다는 뜻

CPU에는 보통 `PC`라는 레지스터가 있다.

```text
PC, Program Counter
= 다음에 실행할 명령어의 메모리 주소
```

기본 실행 흐름은 다음과 같다.

```text
1. PC가 가리키는 주소에서 명령어를 가져온다.
2. 명령어를 실행한다.
3. 보통은 PC를 다음 주소로 증가시킨다.
```

예를 들면 다음과 같다.

```text
100: LOAD x
101: ADD y
102: STORE z
103: ...
```

보통은 `100 -> 101 -> 102 -> 103` 순서로 실행된다.

그런데 조건분기 명령어가 있으면 `PC` 값을 다른 주소로 바꿀 수 있다.

```text
100: LOAD x
101: CMP 0
102: JUMP_IF_ZERO 200
103: ...
200: ...
```

`x == 0`이면 `103`으로 가지 않고 `200`번지로 점프한다.

> [!summary]
> 조건분기는 조건에 따라 `PC` 값을 다른 명령어 주소로 바꾸는 것이다.

"조건분기를 무제한 허용한다"는 말은 물리적으로 무한한 메모리와 시간이 있다는 뜻이 아니다.

구조적으로 실행 흐름을 미리 정해진 직선 순서에 가두지 않는다는 뜻이다.

```text
앞으로 점프
뒤로 점프
같은 위치로 다시 점프
조건에 따라 다른 위치로 점프
```

이것이 가능하면 반복문, 함수 호출, 재귀, 무한 루프 같은 제어 흐름을 표현할 수 있다.

```text
100: 조건 검사
101: 거짓이면 105로 점프
102: 작업 수행
103: 100으로 점프
105: 반복문 끝
```

여기서 `103: 100으로 점프`는 고수준 언어의 `while` 반복문과 닮아 있다.

## 변수에 값을 지정하는 것처럼 볼 수 있나

감각적으로는 맞다.

일반 변수에 값을 넣는 것이 다음과 같다면:

```text
x = 200
```

조건분기는 다음 실행 위치를 가리키는 특수한 값인 `PC`를 바꾸는 것처럼 볼 수 있다.

```text
if (조건이 참이면) {
  PC = 200
}
```

그래서 다음처럼 이해하면 좋다.

> [!info]
> 조건분기는 변수에 값을 지정하듯이, 다음에 실행할 명령어 위치를 특정 주소로 지정하는 것에 가깝다.

단, `PC`는 일반 애플리케이션 코드에서 쓰는 변수는 아니다. CPU 내부의 특수한 레지스터다.

개발자가 직접 `PC = 200`이라고 쓰지는 않지만, `if`, `while`, `return`, 함수 호출 같은 문법은 기계어 수준에서 결국 실행 위치를 바꾸는 명령으로 내려간다.

## 튜링 머신과 이어서 이해하기

튜링 머신은 "계산 가능하다"는 말을 엄밀하게 정의하기 위한 이론적 모델이다.

폰 노이만 구조는 그 범용 계산의 감각을 현실 하드웨어에서 구현한 대표적인 구조다.

| 구분 | 튜링 머신 | 폰 노이만 구조 |
| --- | --- | --- |
| 관심사 | 계산 가능성의 기준 | 현실 컴퓨터의 구조 |
| 핵심 | 유한한 규칙으로 계산을 수행할 수 있는가 | 프로그램과 데이터를 메모리에 저장하고 CPU가 실행 |
| 혁신 | 알고리즘으로 풀 수 있는 것과 없는 것을 말할 기준 제공 | 같은 하드웨어가 프로그램만 바꿔 여러 일을 수행 |
| 연결점 | 범용 튜링 머신 | 저장 프로그램 방식의 범용 컴퓨터 |

## 코드와 데이터가 같은 메모리에 있으면 문제 없나

> [!question]
> 폰 노이만 구조에서는 실행 코드와 데이터가 따로 구분되지 않고 같은 메모리에 섞여 있다고 한다. 그러면 분리하지 않아도 문제가 안 생기나?

문제가 생길 수 있다.

폰 노이만 구조의 핵심은 코드와 데이터가 모두 메모리에 저장된 비트라는 점이다. CPU는 어떤 주소의 값을 읽고, 그것을 명령어로 해석할 수도 있고 데이터로 해석할 수도 있다.

하지만 아무 보호 없이 코드와 데이터를 섞어두면 보안과 안정성 문제가 생긴다.

그래서 현대 컴퓨터는 폰 노이만 구조의 큰 아이디어는 유지하되, 운영체제와 CPU 수준에서 실행 코드와 데이터를 논리적으로 구분하고 보호한다.

```text
프로세스 메모리
├─ text/code 영역 : 실행 코드
├─ data 영역      : 전역 변수
├─ heap 영역      : 동적 데이터
└─ stack 영역     : 함수 호출, 지역 변수
```

각 영역은 페이지 단위로 권한을 가진다.

```text
code page  : 읽기 + 실행 가능, 쓰기 불가
data page  : 읽기 + 쓰기 가능, 실행 불가
stack page : 읽기 + 쓰기 가능, 실행 불가
```

> [!warning]
> 즉 "같은 메모리에 있다"는 말은 "아무 구분 없이 마음대로 읽고 쓰고 실행한다"는 뜻이 아니다.
> 현대 시스템은 같은 메모리 모델 위에 권한 분리와 보호 장치를 얹는다.

### 보안 문제가 생기는 이유

데이터 영역에 들어간 값이 명령어처럼 실행될 수 있으면 공격자가 악성 코드를 데이터처럼 넣고 실행 흐름을 그쪽으로 돌릴 수 있다.

```text
데이터 영역에 악성 코드 주입
-> 버그로 return address 조작
-> CPU가 데이터 영역을 명령어처럼 실행
```

이런 감각이 버퍼 오버플로우, 코드 인젝션 같은 공격과 연결된다.

그래서 현대 시스템은 다음과 같은 방어 장치를 둔다.

| 방어 장치 | 의미 | 왜 필요한가 |
| --- | --- | --- |
| NX bit / DEP | 데이터 영역은 실행하지 못하게 막음 | 데이터로 들어온 악성 코드 실행 방지 |
| W^X | 쓰기 가능한 메모리는 실행 불가, 실행 가능한 메모리는 쓰기 불가 | 코드를 마음대로 수정하고 실행하는 상황 방지 |
| ASLR | 메모리 주소를 랜덤화 | 공격자가 점프할 주소를 예측하기 어렵게 함 |

### 성능 문제도 있다

코드와 데이터가 같은 메모리 통로를 공유하면 CPU가 명령어도 가져와야 하고 데이터도 읽고 써야 한다.

```text
CPU는 빠름
메모리 접근은 상대적으로 느림
명령어 fetch와 데이터 read/write가 같은 통로를 두고 경쟁
```

이 병목을 폰 노이만 병목이라고 부른다.

그래서 현대 CPU는 겉으로는 폰 노이만 구조처럼 같은 주소 공간을 쓰더라도, 내부적으로는 명령어 캐시와 데이터 캐시를 나누는 경우가 많다.

```text
L1 instruction cache
L1 data cache
```

이것은 하버드 구조의 아이디어를 일부 섞은 최적화라고 볼 수 있다.

> [!summary]
> 폰 노이만 구조는 코드와 데이터를 같은 메모리에 저장할 수 있게 해서 범용 컴퓨터를 가능하게 했다.
> 하지만 아무 보호 없이 섞으면 보안과 안정성 문제가 생기므로, 현대 시스템은 OS, MMU, 페이지 권한, NX bit, W^X, ASLR 같은 장치로 코드와 데이터를 논리적으로 구분한다.

## 현대 컴퓨터는 외부는 폰 노이만, 내부는 하버드라는 말

> [!question]
> 현대 컴퓨터는 외부는 폰 노이만 구조이고, CPU 내부는 하버드 아키텍처라는 말이 무슨 뜻인가?

한 줄로 잡으면 다음과 같다.

> [!summary]
> 프로그래머와 운영체제가 보는 메모리 모델은 코드와 데이터가 같은 주소 공간에 있는 폰 노이만 구조에 가깝지만, CPU 내부에서는 성능을 위해 명령어 경로와 데이터 경로를 하버드 구조처럼 나눠서 처리한다는 뜻이다.

폰 노이만 구조는 명령어와 데이터가 같은 메모리에 있다.

```text
CPU
 |
공통 메모리
├─ 코드
└─ 데이터
```

프로그래머 입장에서는 코드 주소든 데이터 주소든 하나의 주소 공간 안에서 관리된다.

하버드 아키텍처는 원래 명령어 메모리와 데이터 메모리를 분리한다.

```text
        ┌─ 명령어 메모리
CPU ────┤
        └─ 데이터 메모리
```

명령어를 가져오는 공간과 데이터를 읽고 쓰는 공간이 분리되어 있으므로, 명령어 fetch와 데이터 access가 서로 덜 부딪힌다.

현대 CPU가 "겉은 폰 노이만, 안은 하버드"라는 말은 다음과 같다.

```text
외부 관점, 프로그래머/OS 관점
-> 코드와 데이터가 같은 메모리 주소 공간에 있음
-> 폰 노이만처럼 보임

CPU 내부 관점
-> 명령어 캐시와 데이터 캐시를 따로 둠
-> 명령어 fetch 경로와 데이터 load/store 경로를 나눔
-> 하버드처럼 동작하는 부분이 있음
```

대표적인 예가 L1 캐시 분리다.

```text
L1 instruction cache : 명령어 가져오기용
L1 data cache        : 데이터 읽기/쓰기용
```

CPU는 실행 중에 두 가지 일을 계속 해야 한다.

```text
1. 다음에 실행할 명령어 가져오기
2. 현재 명령어가 필요한 데이터 읽기/쓰기
```

둘이 완전히 같은 통로만 쓰면 서로 기다릴 수 있다.

```text
명령어 가져오는 중
-> 데이터 읽기 기다림

데이터 읽는 중
-> 다음 명령어 가져오기 기다림
```

이것이 폰 노이만 병목의 감각이다.

그래서 현대 CPU 내부에서는 다음처럼 나눠서 성능을 높인다.

```text
명령어 fetch 경로
데이터 load/store 경로
```

> [!warning]
> 이것이 완전한 하버드 구조라는 뜻은 아니다.
> 보통은 수정된 하버드 구조, modified Harvard architecture라고 보는 편이 더 정확하다.

완전한 하버드 구조라면 명령어 메모리와 데이터 메모리가 프로그래머에게도 분리되어 보일 수 있다.

하지만 일반적인 현대 컴퓨터에서는 운영체제가 하나의 가상 주소 공간 안에서 코드, 데이터, 힙, 스택을 관리한다.

```text
프로세스 가상 주소 공간
├─ text/code
├─ data
├─ heap
└─ stack
```

즉 바깥에서 보는 프로그래밍 모델은 폰 노이만 구조에 가깝고, CPU 내부 구현은 성능을 위해 하버드 구조의 아이디어를 섞은 것이다.

| 관점 | 구조 | 의미 |
| --- | --- | --- |
| 프로그래머/OS가 보는 바깥 구조 | 폰 노이만 | 코드와 데이터가 같은 메모리 주소 공간에 있음 |
| CPU 내부 구현 | 하버드식 최적화 | 명령어 캐시와 데이터 캐시, fetch 경로와 load/store 경로를 분리 |
| 정확한 표현 | 수정된 하버드 구조 | 겉보기 메모리 모델은 통합, 내부 성능 구조는 분리 |

> [!example]
> 백엔드 관점에서 보면:
>
> ```text
> 외부 API는 하나처럼 보임
> 내부 구현은 읽기 경로, 쓰기 경로, 캐시를 나눠 최적화함
> ```
>
> 클라이언트는 하나의 API만 보지만, 내부에서는 read replica, primary DB, cache를 분리해서 성능을 높일 수 있다.
> 현대 CPU도 프로그래머에게는 하나의 메모리 주소 공간처럼 보이게 하면서, 내부에서는 명령어와 데이터 경로를 나눠 성능을 높인다.

## 면접에서 말하면

> [!tip]
> 폰 노이만 구조의 핵심은 프로그램과 데이터를 같은 메모리에 저장하고, CPU가 메모리에서 명령어를 가져와 실행한다는 점입니다. 이 덕분에 하드웨어를 매번 바꾸지 않고 프로그램만 바꿔서 같은 기계가 여러 작업을 수행할 수 있게 되었습니다. 또한 명령어가 주소 기반 메모리에 저장되므로 조건에 따라 다음 실행 주소를 바꾸는 분기와 반복이 가능해졌습니다.

## 내가 다시 헷갈릴 것 같은 부분

- "프로그램도 데이터처럼 다룬다"는 말은 함수 파라미터 비유와 닮았지만, 더 아래 수준에서는 명령어 자체가 메모리에 저장된다는 뜻이다.
- "조건분기를 무제한 허용한다"는 말은 물리적 무한이 아니라, 실행 흐름을 고정된 순서에 가두지 않는다는 뜻이다.
- 조건분기는 결국 `PC`가 가리키는 다음 명령어 주소를 바꾸는 것으로 이해하면 된다.
- 임의 접근 메모리가 있어야 원하는 명령어 주소로 이동하는 실행 흐름을 자연스럽게 만들 수 있다.
- 코드와 데이터가 같은 메모리에 있다는 말은 보호 없이 섞어도 된다는 뜻이 아니라, 현대 시스템에서는 실행 가능 여부와 쓰기 가능 여부를 권한으로 나눠서 관리한다는 점을 같이 기억해야 한다.
- 현대 컴퓨터는 프로그래머 관점에서는 폰 노이만 구조처럼 보이지만, CPU 내부에서는 명령어 캐시와 데이터 캐시를 나눠 하버드 구조의 장점을 일부 가져온다.

## 한 줄 회고

- 헷갈렸던 점: 프로그램을 데이터처럼 다룬다는 말을 고수준 함수 전달 비유로 이해할 수는 있지만, 폰 노이만 구조의 핵심은 더 아래에서 명령어 자체가 메모리에 저장되고 CPU가 그것을 읽어 실행한다는 점.
- 헷갈렸던 점: 조건분기는 마치 `PC`라는 특수한 변수에 다음 실행 주소를 대입하는 것처럼 이해할 수 있다는 점.
- 헷갈렸던 점: 코드와 데이터가 같은 메모리에 있다는 말은 현대 시스템에서 코드 영역과 데이터 영역을 아무 보호 없이 섞는다는 뜻이 아니라, 같은 메모리 모델 위에서 페이지 권한과 실행 권한으로 구분한다는 점.
- 헷갈렸던 점: 현대 컴퓨터는 바깥에서 볼 때는 코드와 데이터가 같은 주소 공간에 있는 폰 노이만 모델이지만, CPU 내부에서는 명령어와 데이터 경로를 나눠 병목을 줄이는 하버드식 최적화를 쓴다는 점.
