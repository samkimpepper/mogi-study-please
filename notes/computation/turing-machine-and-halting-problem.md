# 튜링 머신과 정지 문제 입문

> [!summary]
> 튜링 머신은 실제 컴퓨터 설계도가 아니라, 사람이 손으로 절차대로 하는 계산을 수학적으로 추상화한 모델이다.
>
> - 튜링 머신의 핵심은 `테이프`, `헤드`, `상태`, `규칙표`다.
> - 중요한 점은 복잡한 컴퓨터를 닮았는지가 아니라, "계산 가능하다"는 말을 엄밀하게 정의했다는 것이다.
> - 정지 문제는 이 계산 모델 위에서 "모든 프로그램이 멈출지 판별하는 일반 알고리즘은 없다"는 한계를 보여준다.

## 내가 물어본 질문

> [!question]
> 튜링 머신이 뭔지 잘 모르겠다.
>
> - 폰 노이만 구조처럼 명령어를 순서대로 하나씩 수행하는 구조인가?
> - 너무 당연한 말처럼 느껴지는데, 왜 당시에는 중요하고 혁신적인 개념이었을까?

## 먼저 잡을 한 줄 정의

> [!info]
> 튜링 머신은 "컴퓨터가 계산한다"는 말을 수학적으로 설명하기 위해 만든 아주 단순한 가상의 계산 모델이다.

폰 노이만 구조는 현실 컴퓨터를 어떻게 만들 것인지에 가까운 구조다.

반면 튜링 머신은 현실 컴퓨터의 부품 배치보다 더 근본적인 질문을 다룬다.

```text
계산한다는 것은 무엇인가?
어떤 문제는 절차적으로 풀 수 있는가?
어떤 문제는 아무리 알고리즘을 만들어도 풀 수 없는가?
```

그래서 둘은 비슷해 보이지만 층위가 다르다.

| 구분 | 튜링 머신 | 폰 노이만 구조 |
| --- | --- | --- |
| 성격 | 수학적 계산 모델 | 실제 컴퓨터 구조 모델 |
| 관심사 | 무엇이 계산 가능한가 | 명령어와 데이터를 어떻게 저장하고 실행하는가 |
| 구성 | 테이프, 헤드, 상태, 규칙표 | CPU, 메모리, 저장장치, 입출력 |
| 목적 | 계산의 본질 정의 | 범용 컴퓨터 구현 |

## 튜링 머신과 폰 노이만 구조의 관계

> [!warning]
> 폰 노이만 구조를 "튜링 머신을 그대로 실체화한 것"이라고 말하면 조금 부정확하다.

튜링 머신과 폰 노이만 구조는 생김새가 다르다.

```text
튜링 머신:
무한 테이프 + 헤드 + 상태 + 규칙표

폰 노이만 구조:
CPU + 메모리 + 프로그램 카운터 + 명령어 + 데이터 + 입출력 장치
```

튜링 머신은 계산 가능성을 설명하기 위한 이론적 모델이고, 폰 노이만 구조는 현실 컴퓨터를 만들기 위한 하드웨어 구조다.

하지만 둘은 **범용 계산**이라는 점에서 강하게 연결된다.

> [!summary]
> 폰 노이만 구조는 튜링 머신을 부품 그대로 물리적으로 만든 것은 아니지만, 튜링 머신이 정식화한 범용 계산 개념을 현실 하드웨어로 구현한 대표적인 구조로 볼 수 있다.

특히 중요한 연결고리는 **범용 튜링 머신**이다.

일반 튜링 머신은 특정 계산을 수행하는 기계로 볼 수 있다.

```text
덧셈하는 튜링 머신
곱셈하는 튜링 머신
문자열을 비교하는 튜링 머신
```

범용 튜링 머신은 다른 튜링 머신의 설명과 입력을 받아서, 그 튜링 머신의 실행을 흉내 낼 수 있다.

```text
범용 튜링 머신
+ 다른 튜링 머신의 설명
+ 입력
-> 해당 튜링 머신의 실행을 시뮬레이션
```

이 감각이 현대 컴퓨터와 닮아 있다.

```text
컴퓨터
+ 프로그램 파일
+ 입력 데이터
-> 프로그램 실행
```

폰 노이만 구조에서는 프로그램도 데이터처럼 메모리에 저장된다. CPU는 메모리에서 명령어를 가져와 실행한다.

그래서 같은 하드웨어가 프로그램만 바꿔서 여러 일을 할 수 있다.

```text
같은 컴퓨터
-> 브라우저 실행
-> IDE 실행
-> 서버 실행
-> 게임 실행
```

백엔드 관점으로 비유하면 다음과 같다.

```java
interface ComputationModel {
    Output run(Program program, Input input);
}
```

튜링 머신은 이런 계산 모델의 이론적 기준에 가깝다.

```java
class VonNeumannComputer implements ComputationModel {
    Memory memory;
    CPU cpu;
    ProgramCounter pc;
}
```

폰 노이만 구조는 그 기준을 만족하는 현실적인 구현 방식 중 하나에 가깝다.

> [!tip]
> 짧게 말하면 튜링 머신은 이론이고, 폰 노이만 구조는 그 이론이 말하는 범용 계산을 현실 컴퓨터로 구현한 대표적인 설계다.

## 튜링 머신의 구성

튜링 머신은 다음 네 가지로 계산을 표현한다.

| 구성 | 의미 |
| --- | --- |
| 테이프 | 무한히 긴 메모리 |
| 헤드 | 현재 한 칸을 읽고 쓰는 장치 |
| 상태 | 지금 머신이 어떤 단계에 있는지 나타내는 값 |
| 규칙표 | 현재 상태와 읽은 값에 따라 다음 행동을 정한 표 |

```text
... _ _ 1 0 1 _ _ ...
          ^
         헤드
```

동작은 단순하다.

```text
현재 상태 + 현재 칸의 값
-> 새 값 쓰기
-> 왼쪽 또는 오른쪽으로 한 칸 이동
-> 다음 상태로 변경
```

백엔드 코드 느낌으로 보면 다음과 비슷하다.

```java
while (true) {
    Symbol current = tape.read(head);
    Rule rule = rules.get(state, current);

    tape.write(head, rule.newSymbol);
    head.move(rule.direction);
    state = rule.nextState;

    if (state == HALT) {
        break;
    }
}
```

> [!info]
> 튜링 머신은 "절차대로 하나씩 명령을 수행한다"는 감각과 맞닿아 있다.
>
> 다만 핵심은 실제 CPU를 닮았다는 점이 아니라, 이렇게 단순한 모델로도 일반적인 계산을 표현할 수 있다는 점이다.

## 왜 헷갈렸나

튜링 머신 설명은 지금 보면 너무 당연하게 느껴진다.

```text
읽는다.
쓴다.
이동한다.
상태를 바꾼다.
반복한다.
```

현대 개발자에게는 프로그램, 메모리, 루프, 상태, 입력, 출력이 이미 익숙하기 때문이다.

하지만 튜링이 이 개념을 제시한 1936년에는 현대적인 컴퓨터가 없었다. 그때는 "계산 가능하다"는 말 자체가 지금처럼 명확하지 않았다.

```text
계산으로 풀 수 있다
= 사람이 공식을 안다?
= 유한한 절차가 있다?
= 손으로 따라 하면 언젠가 답이 나온다?
= 기계가 해도 된다?
```

튜링 머신은 이 애매한 말을 다음처럼 정리했다.

> [!summary]
> 어떤 문제가 계산 가능하다는 것은, 튜링 머신이 유한한 규칙을 따라가서 답을 낼 수 있다는 뜻으로 볼 수 있다.

## 왜 중요한가

튜링 머신의 혁신은 "컴퓨터처럼 생긴 모델"을 만든 데 있지 않다.

진짜 중요한 점은 계산 가능성의 기준을 세웠다는 것이다.

> [!info]
> 튜링 머신 덕분에 "아직 방법을 못 찾았다"와 "원리적으로 어떤 알고리즘으로도 풀 수 없다"를 구분할 수 있게 되었다.

이 기준이 있어야 정지 문제 같은 말을 정확히 할 수 있다.

```text
HALT(machine, input)
-> 이 기계가 이 입력에서 언젠가 멈추는가?
```

튜링은 모든 튜링 머신과 모든 입력에 대해 이 질문에 항상 정확히 답하는 일반 알고리즘은 없다는 것을 보였다.

즉 정지 문제는 단순히 "무한루프를 찾는 도구를 만들기 어렵다"가 아니다.

```text
임의의 프로그램과 임의의 입력에 대해,
그 프로그램이 멈출지 여부를 항상 정확히 판별하는 일반 알고리즘은 존재하지 않는다.
```

## 정지 문제에서 멈춘다는 뜻

> [!info]
> 정지 문제에서 "멈춘다"는 것은 프로그램이 유한한 계산 단계 후 종료 상태에 도달한다는 뜻이다.

멈춘다는 말은 정상 종료만 뜻하지 않는다.

```text
정상 종료
에러 종료
명시적인 halt 상태
```

처럼 더 이상 다음 계산 단계를 실행하지 않으면 멈춘 것으로 볼 수 있다.

반대로 멈추지 않는다는 것은 어떤 유한한 단계 후에도 종료 상태에 도달하지 않는다는 뜻이다.

```text
1번째 단계
2번째 단계
3번째 단계
...
```

이렇게 계속 진행해도 끝나는 지점이 없는 경우다.

> [!warning]
> 지금까지 안 끝났다는 사실은, 앞으로도 절대 안 끝난다는 증거가 아니다.

예를 들어 어떤 프로그램이 `10^100`번째 단계에서 종료한다고 하자.

사람이 보기에는 사실상 영원히 도는 것처럼 보일 수 있지만, 이론적으로는 멈추는 프로그램이다.

```text
오래 걸림
!=
안 멈춤
```

그래서 "언제까지 돌아야 무한인가?"라는 질문에 대한 답은 다음과 같다.

```text
아무리 오래 관찰해도,
관찰만으로는 진짜 무한 실행이라고 확정할 수 없다.
```

## 타임아웃은 정지 문제의 답이 아니다

실무에서는 타임아웃이 매우 중요하다.

```text
3초 안에 응답이 없으면 실패 처리
1시간 넘게 도는 배치는 중단
락 대기는 5초까지만 허용
```

하지만 타임아웃은 수학적 판별이 아니라 운영 정책이다.

```text
3초 안에 안 끝났으니 중단하겠다.
```

라고 말할 뿐이다.

이 말은 다음을 증명하지 않는다.

```text
이 프로그램은 절대 안 끝난다.
```

예를 들어 5초 뒤 종료하는 프로그램에 3초 타임아웃을 걸면, 실무적으로는 실패 처리할 수 있지만 이론적으로는 멈추는 프로그램을 중단한 것이다.

> [!summary]
> 타임아웃은 서비스를 보호하는 장치이지, 프로그램이 영원히 멈추지 않는다는 증명은 아니다.

## N번째 단계와 귀납법

정지 문제를 이해할 때는 "시간"보다 "계산 단계"로 생각하는 편이 좋다.

```text
어떤 자연수 N이 있어서
N번째 단계 안에 종료하면 멈춘다.
```

반대로 그런 `N`이 아예 없으면 멈추지 않는다.

이 관점에서 보면 귀납법이나 종료 증명이 떠오를 수 있다.

실제로 어떤 프로그램은 귀납적으로 종료를 증명할 수 있다.

```java
for (int i = 0; i < n; i++) {
    work();
}
return;
```

이 코드는 `i`가 0에서 시작해서 매번 1씩 증가하고, 결국 `n`에 도달하면 반복문이 끝난다.

재귀도 비슷하게 볼 수 있다.

```java
int factorial(int n) {
    if (n == 0) return 1;
    return n * factorial(n - 1);
}
```

여기서는 `n`이 매 호출마다 줄어든다.

```text
n
n - 1
n - 2
...
0
```

그래서 언젠가 base case에 도달한다.

이런 종료 증명을 보통 termination proof라고 볼 수 있다.

> [!info]
> 귀납법은 특정 프로그램의 종료를 증명하는 데 쓸 수 있다.
>
> 하지만 모든 프로그램에 대해 그런 증명을 자동으로 찾아주는 만능 알고리즘은 만들 수 없다.

## 왜 단순 귀납으로는 전체를 못 푸나

귀납법으로 종료를 보이려면 보통 다음 구조가 필요하다.

```text
1. 어떤 값이 매번 줄어든다.
2. 그 값은 더 이상 내려갈 수 없는 바닥이 있다.
3. 그래서 언젠가 끝난다.
```

예를 들면 다음과 같은 값들이 있다.

```text
남은 반복 횟수
남은 리스트 길이
남은 트리 높이
남은 재귀 깊이
```

하지만 일반 프로그램은 상태가 훨씬 자유롭다.

```text
변수 여러 개
힙 메모리
외부 입력
파일
네트워크
랜덤
재귀
동적 dispatch
예외
스레드
락
```

무엇이 "줄어드는 값"인지 자동으로 찾기 어렵고, 아예 그런 값이 없는 프로그램도 있다.

> [!warning]
> 정지 문제의 핵심은 특정 프로그램의 종료 증명을 못 찾을 수 있다는 정도가 아니다.
>
> 모든 프로그램의 종료 여부를 항상 맞히는 일반 알고리즘 자체가 불가능하다는 점이다.

정리하면 다음과 같다.

| 접근 | 가능한가 |
| --- | --- |
| 특정 프로그램이 끝난다는 것을 귀납법으로 증명 | 가능할 수 있음 |
| 특정 프로그램이 안 끝난다는 것을 증명 | 가능할 수 있음 |
| 모든 프로그램에 대해 끝남/안 끝남을 자동 판별 | 불가능 |

## 백엔드 관점에서 보면

튜링 머신은 현실 서버 구조와 직접 대응되는 설계도는 아니다.

하지만 백엔드 개발자가 만나는 문제와 감각적으로 이어진다.

```text
요청 처리 로직
-> 상태 조회
-> 조건 분기
-> 반복 또는 재귀
-> 외부 입력에 따라 다른 경로 실행
-> 언젠가 응답하거나, 끝나지 않거나, 블로킹됨
```

간단한 코드는 종료 여부를 알 수 있다.

```java
return "ok";
```

이 코드는 바로 끝난다.

```java
while (true) {
}
```

이 코드는 명백히 끝나지 않는다.

하지만 모든 가능한 프로그램과 입력에 대해 일반적으로 판별하는 것은 불가능하다.

그래서 실무에서는 완벽한 정지 판별기 대신 다음 장치를 둔다.

```text
- 타임아웃
- watchdog
- circuit breaker
- 리소스 제한
- 정적 분석
- 테스트
- 장애 격리
```

> [!tip]
> 실무에서는 "이 프로그램이 반드시 끝나는지 완벽히 증명한다"보다, 끝나지 않거나 오래 걸릴 때 피해를 제한하는 설계가 더 현실적이다.

## P, NP와의 위치 차이

정지 문제와 P/NP는 연결해서 이야기할 수 있지만 같은 문제는 아니다.

| 주제 | 핵심 질문 |
| --- | --- |
| 정지 문제 | 이 문제는 알고리즘으로 항상 풀 수 있는가? |
| P vs NP | 풀 수 있는 문제라면 빠르게 풀 수 있는가? |
| 괴델 불완전성 | 수학 체계 안에서 모든 참인 명제를 증명할 수 있는가? |

정지 문제는 계산 가능성의 한계를 다룬다.

P/NP는 계산 가능한 문제들 안에서 효율성의 한계를 다룬다.

```text
정지 문제:
풀 수 있냐?

P/NP:
풀 수 있다고 할 때, 빠르게 풀 수 있냐?
```

## P/NP에서 빠른 풀이와 빠른 검증

> [!question]
> 헷갈렸던 질문:
>
> - 빠른 검증과 빠른 풀이가 무슨 차이인가?
> - "어떤 답이 정답인지 돌려보는 것"이 검증이라면, 풀이는 무엇인가?
> - P/NP는 정지 문제와 같은 핀트의 문제인가?

> [!info]
> 풀이는 답을 직접 찾아내는 일이고, 검증은 누가 준 답 후보가 진짜 맞는지 확인하는 일이다.

검증에는 문제와 답 후보가 같이 주어진다.

```text
문제 + 답 후보
-> 이 후보가 진짜 정답인지 확인
```

풀이는 답 후보 없이 문제만 보고 시작한다.

```text
문제
-> 정답을 직접 찾아내기
```

예를 들어 비밀번호 자물쇠가 있다고 하자.

```text
0000 ~ 9999
```

풀이는 맞는 비밀번호를 직접 찾는 것이다.

```text
0000?
0001?
0002?
...
```

검증은 누가 준 후보가 맞는지 확인하는 것이다.

```text
후보: 3842
-> 입력해보기
-> 열림 / 안 열림
```

스도쿠도 비슷하다.

```text
풀이:
빈칸을 채워서 완성된 정답판을 만들어라.

검증:
누가 완성된 판을 줬다.
이 판이 규칙을 만족하는지 확인해라.
```

검증은 다음을 확인하면 된다.

```text
- 원래 주어진 숫자를 바꾸지 않았는가?
- 각 행이 조건을 만족하는가?
- 각 열이 조건을 만족하는가?
- 각 박스가 조건을 만족하는가?
```

백엔드 감각으로 비유하면 다음과 같다.

```text
풀이:
빈 저장소에서 요구사항만 보고 동작하는 코드를 직접 작성하기

검증:
PR diff를 보고 요구사항을 만족하는지 확인하기
```

검증도 어려울 수 있지만, 답 후보가 이미 있다는 점에서 풀이와 다르다.

## P와 NP의 정확한 질문

> [!info]
> P는 빠르게 풀 수 있는 결정 문제들의 집합이고, NP는 답 후보가 주어졌을 때 빠르게 검증할 수 있는 결정 문제들의 집합이다.

여기서 "빠르게"는 보통 다항 시간 안에 가능하다는 뜻이다.

```text
O(n)
O(n log n)
O(n^2)
O(n^3)
```

P/NP 문제의 핵심 질문은 다음과 같다.

```text
P = NP 인가?
```

풀어서 말하면 다음과 같다.

```text
빠르게 검증할 수 있는 문제는
전부 빠르게 풀 수도 있는가?
```

이 질문은 아직 증명되지 않았다.

> [!summary]
> 2026년 현재, P vs NP는 미해결 문제다.
>
> 많은 컴퓨터과학자들은 `P != NP`일 것이라고 예상하지만, 아직 증명된 것은 아니다.

만약 `P = NP`라면, 답 후보를 빠르게 확인할 수 있는 많은 문제에서 답 자체도 빠르게 찾을 수 있다는 뜻이 된다.

```text
빠른 검증 가능
-> 빠른 풀이도 가능
```

이 경우 암호학, 최적화, 스케줄링, 물류, 회로 설계, AI 탐색, 증명 자동화 등에 큰 영향이 생길 수 있다.

반대로 `P != NP`가 증명되면, 빠르게 검증은 되지만 본질적으로 빠르게 풀 수는 없는 문제가 존재한다는 뜻이 된다.

## 정지 문제와 P/NP의 핀트 차이

정지 문제와 P/NP는 둘 다 계산의 한계를 다루지만, 묻는 질문이 다르다.

```text
정지 문제:
이 문제를 푸는 알고리즘이 애초에 존재하는가?

P/NP:
알고리즘으로 풀 수 있는 문제들 중에서,
빠르게 풀 수 있는가?
```

정지 문제는 계산 가능성의 문제다.

```text
Can we solve it at all?
```

P/NP는 계산 복잡도와 효율성의 문제다.

```text
Can we solve it efficiently?
```

층위로 보면 다음과 같다.

```text
1층: 계산 가능성
     알고리즘이 존재하는가?
     정지 문제는 여기서 "없다"가 증명됨.

2층: 계산 복잡도
     알고리즘이 있다면 얼마나 빠른가?
     P/NP는 여기서 "빠른 검증 = 빠른 풀이인가?"를 묻는다.
```

> [!warning]
> 정지 문제와 P/NP는 "컴퓨터도 못 하는 게 있다"는 감각 때문에 비슷해 보이지만, 못 한다는 말의 의미가 다르다.

| 주제 | 못 한다는 말의 의미 |
| --- | --- |
| 정지 문제 | 모든 경우를 맞히는 알고리즘 자체가 없음 |
| P/NP | 알고리즘은 있지만 빠른 풀이가 항상 있는지는 모름 |

정리하면 다음 한 문장이다.

> [!summary]
> 정지 문제는 가능/불가능의 문제이고, P/NP는 효율/비효율의 문제라 핀트가 다르다.

## 정지 문제, P/NP, 괴델 불완전성 비교

세 주제는 모두 "규칙과 계산으로 모든 것을 해결할 수 있는가?"라는 기대에 한계를 보여준다.

하지만 한계가 생기는 층위는 서로 다르다.

| 주제 | 묻는 질문 | 결론 | 한계의 종류 |
| --- | --- | --- | --- |
| 정지 문제 | 모든 프로그램의 정지 여부를 알고리즘으로 항상 판별할 수 있는가? | 불가능이 증명됨 | 계산 가능성의 한계 |
| P/NP | 빠르게 검증할 수 있는 문제는 빠르게 풀 수도 있는가? | 아직 미해결 | 계산 효율성의 한계 |
| 괴델 불완전성 | 충분히 강한 형식 체계에서 모든 참인 명제를 증명할 수 있는가? | 불가능이 증명됨 | 형식 증명의 한계 |

각 주제의 핵심 문장은 다음과 같다.

```text
정지 문제:
계산으로 풀 수 없는 문제가 있다.

P/NP:
풀 수 있는 문제라도 효율적으로 풀 수 있는지는 별개의 문제다.

괴델 불완전성:
참이지만 그 체계 안에서는 증명할 수 없는 명제가 있다.
```

> [!summary]
> 정지 문제는 알고리즘 자체가 없는 경우를 말하고, P/NP는 알고리즘이 있는 문제들 안에서 빠른 풀이가 가능한지를 묻고, 괴델 불완전성은 참인 명제가 특정 형식 체계 안에서 증명 가능한지를 묻는다.

## 괴델 불완전성과의 감각적 연결

괴델 불완전성과 정지 문제는 완전히 같은 말은 아니지만 비슷한 메시지를 준다.

> [!summary]
> 충분히 강력한 형식 시스템이나 계산 모델 안에는, 그 시스템 내부의 규칙만으로는 해결할 수 없는 한계가 생긴다.

괴델은 수학에서 다음을 보였다.

```text
참이지만 그 체계 안에서는 증명할 수 없는 명제가 있다.
```

튜링은 계산에서 다음을 보였다.

```text
정답은 정해져 있어도, 모든 경우를 판별하는 알고리즘은 없다.
```

그래서 정지 문제는 괴델 불완전성을 컴퓨터와 알고리즘의 언어로 만나는 입구처럼 볼 수 있다.

## 면접에서 말하면

> [!tip]
> 튜링 머신은 실제 컴퓨터 구조라기보다, 계산 가능하다는 말을 엄밀하게 정의하기 위한 수학적 모델입니다. 무한 테이프, 헤드, 상태, 규칙표만으로 계산을 표현하고, 이 모델을 기준으로 어떤 문제가 알고리즘으로 풀 수 있는지 논의할 수 있습니다. 정지 문제는 이 모델 위에서 임의의 프로그램이 멈출지 여부를 항상 판별하는 일반 알고리즘은 없다는 것을 보여주는 대표적인 예입니다.

## 나중에 다시 볼 포인트

- 튜링 머신은 폰 노이만 구조보다 추상적인 계산 모델이다.
- 폰 노이만 구조는 튜링 머신을 그대로 실체화한 것이 아니라, 범용 계산 개념을 현실 하드웨어로 구현한 대표적인 구조다.
- 지금 당연해 보이는 이유는 현대 프로그래밍의 개념을 이미 알고 있기 때문이다.
- 튜링 머신의 중요성은 계산의 기준을 만든 데 있다.
- 정지 문제는 "어렵다"가 아니라 "일반 알고리즘으로는 불가능하다"는 종류의 한계다.
- 오래 걸리는 프로그램과 영원히 안 끝나는 프로그램은 다르다.
- 타임아웃은 실무 정책이지 정지 여부의 수학적 증명이 아니다.
- 귀납법은 특정 프로그램의 종료 증명에 쓸 수 있지만, 모든 프로그램에 대한 만능 판별기는 될 수 없다.
- P/NP는 계산 가능성보다 한 단계 뒤의 효율성 문제다.
- P/NP에서 풀이는 답을 직접 찾는 것이고, 검증은 주어진 답 후보가 맞는지 확인하는 것이다.
- P vs NP는 "빠른 검증이 가능하면 빠른 풀이도 가능한가?"를 묻는 미해결 문제다.
- 정지 문제, P/NP, 괴델 불완전성은 모두 한계를 다루지만 각각 계산 가능성, 계산 효율성, 형식 증명의 한계로 구분된다.

## 한 줄 회고

- 헷갈렸던 점: 튜링 머신은 실제 컴퓨터 구조를 설명하려는 것이 아니라, 사람이 손으로 하는 절차적 계산을 추상화해서 "계산 가능하다"는 말의 기준을 세우려는 모델이라는 점.
- 헷갈렸던 점: 폰 노이만 구조는 튜링 머신의 부품을 그대로 현실화한 것이 아니라, 범용 튜링 머신과 닮은 "프로그램을 저장하고 실행하는 범용 계산"을 현실 하드웨어로 구현한 구조라는 점.
- 헷갈렸던 점: 아무리 오래 실행되어도 그것만으로 무한루프라고 확정할 수 없고, 어떤 `N`번째 단계에서라도 종료하면 이론적으로는 멈추는 프로그램이라는 점.
- 헷갈렸던 점: 귀납법은 특정 종료 증명에는 쓸 수 있지만, 모든 프로그램의 종료 여부를 자동으로 판별하는 일반 해법은 될 수 없다는 점.
- 헷갈렸던 점: P/NP에서 검증은 주어진 답 후보를 확인하는 것이고, 풀이는 답 후보 없이 정답을 직접 찾아내는 것이라는 점.
- 헷갈렸던 점: 정지 문제는 알고리즘 존재 여부의 문제이고, P/NP는 알고리즘이 있는 문제들 안에서 빠른 풀이와 빠른 검증의 관계를 묻는 문제라는 점.
- 헷갈렸던 점: 정지 문제, P/NP, 괴델 불완전성은 모두 "한계"를 말하지만 각각 계산 가능성, 계산 효율성, 형식 증명이라는 다른 층위를 다룬다는 점.
