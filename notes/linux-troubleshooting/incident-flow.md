# 실전 장애 확인 흐름

리눅스로 서버 상태를 본다는 것은 `CPU/메모리`, `디스크`, `프로세스`, `포트/연결`, `로그`를 순서대로 확인해서 병목 위치를 좁히는 것이다.

장애 상황에서 바로 코드부터 보면 원인을 늦게 찾을 수 있다.

먼저 서버 자체가 힘든지, 특정 프로세스가 문제인지, 애플리케이션 포트와 TCP 연결 상태는 어떤지 확인해야 한다.

```text
1. 서버가 진짜 힘든가?
   top
   free -h
   df -h

2. 어떤 프로세스가 문제인가?
   top
   ps -fp <PID>

3. 애플리케이션 포트가 열려 있는가?
   ss -ltnp | grep ':8080'

4. 연결이 어떤 상태로 쌓였는가?
   ss -ant | grep ':8080'
   ss -ant | grep ':8080' | awk '{print $1}' | sort | uniq -c

5. 앱 로그에는 뭐가 찍히는가?
   tail -f app.log
   journalctl -u <service-name> -f
```

## API가 느릴 때 예시 흐름

예를 들어 "API 응답이 느리다"는 장애가 났다고 하자.

바로 코드를 보기 전에 먼저 서버 상태를 본다.

```bash
top
```

여기서 CPU가 95% 이상이면 다음처럼 본다.

```text
서버가 바쁨
-> 어떤 PID가 CPU를 많이 쓰는지 확인
-> ps -fp <PID>
-> 그게 내 Spring/Node 애플리케이션인지, DB인지, 다른 배치인지 확인
```

메모리가 부족한지는 다음으로 확인한다.

```bash
free -h
```

디스크가 꽉 찼는지도 확인한다.

```bash
df -h
```

디스크가 100%라면 로그를 못 쓰거나 임시 파일 생성이 실패해서 애플리케이션이 이상하게 동작할 수 있다.

그 다음 애플리케이션 포트가 열려 있는지 본다.

```bash
ss -ltnp | grep ':8080'
```

이 명령어는 "내 서버가 8080 포트에서 요청 받을 준비가 되어 있는가?"를 확인하는 용도다.

그리고 연결 상태를 본다.

```bash
ss -ant | grep ':8080' | awk '{print $1}' | sort | uniq -c
```

여기서 상태별로 해석한다.

```text
ESTABLISHED 많음:
요청 처리 중인 연결이 많음
앱 처리 지연, DB 지연, 외부 API 지연 가능성

CLOSE_WAIT 많음:
상대방은 연결을 닫았는데 내 앱이 정리를 못 함
코드에서 connection, stream, response body close 누락 가능성

TIME_WAIT 많음:
짧은 연결을 너무 많이 만들고 끊는 중
커넥션 풀, keep-alive 설정 확인

SYN_RECV 많음:
TCP 연결 수립 단계에서 막힘
트래픽 폭증, 네트워크 문제, 공격 가능성
```

마지막으로 앱 로그를 본다.

```bash
tail -f app.log
journalctl -u <service-name> -f
```

정리하면, "서버가 느리다"는 말을 들었을 때 바로 하나의 원인으로 단정하지 않는다.

다음 질문으로 좁혀간다.

```text
서버 리소스 문제인가?
특정 프로세스 문제인가?
포트가 열려 있는가?
연결이 어떤 상태로 쌓였는가?
앱 로그에는 어떤 에러가 있는가?
DB, Redis, 외부 API 같은 뒤쪽 의존성이 느린가?
```
