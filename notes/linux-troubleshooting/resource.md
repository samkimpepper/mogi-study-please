# 서버 리소스 확인

서버 리소스 확인은 CPU, 메모리, 디스크가 부족한지 확인하는 작업이다.

서버가 느리다고 하면 먼저 다음 세 가지를 확인한다.

```bash
top
free -h
df -h
```

## 리소스 확인 3종 세트

```text
top:
CPU, 메모리, load average, 프로세스별 사용량 확인

free -h:
전체 메모리와 swap 사용량 확인

df -h:
디스크 용량 확인
```

## CPU가 높은 경우

`top`에서 CPU 사용률이 높다면 어떤 PID가 CPU를 많이 쓰는지 본다.

```bash
top
ps -fp <PID>
```

흐름은 보통 이렇다.

```text
서버가 느림
-> top 실행
-> CPU를 많이 쓰는 PID 확인
-> ps -fp <PID>로 어떤 프로세스인지 확인
-> 내 애플리케이션인지, 배치인지, DB인지 확인
-> 앱 로그, 쿼리, 최근 배포, 트래픽 증가를 이어서 확인
```

백엔드 서버라면 CPU가 높은 원인은 대략 다음처럼 좁혀볼 수 있다.

```text
- 트래픽 증가
- 무거운 쿼리나 반복문
- 대량 데이터 처리
- 배치 작업
- 무한 루프 또는 비정상적인 재시도
```

## 메모리가 부족한 경우

메모리 상태는 `free -h`로 본다.

```bash
free -h
```

메모리가 부족하면 애플리케이션이 느려지거나, swap을 많이 쓰거나, 심하면 OOM으로 프로세스가 죽을 수 있다.

```text
메모리 부족
-> free -h 확인
-> top에서 어떤 프로세스가 메모리를 많이 쓰는지 확인
-> ps -fp <PID>로 정체 확인
-> 애플리케이션 메모리 누수, 캐시 증가, 대량 조회, JVM heap 설정 등을 의심
```

백엔드에서는 다음 상황을 의심해볼 수 있다.

```text
- 너무 큰 데이터를 한 번에 메모리에 올림
- 캐시가 계속 커짐
- 커넥션이나 객체가 해제되지 않음
- JVM heap 설정이 서버 크기에 비해 부적절함
- 트래픽 증가로 동시 요청이 많아짐
```

## 디스크가 꽉 찬 경우

디스크 용량은 `df -h`로 본다.

```bash
df -h
```

디스크가 100%에 가까우면 애플리케이션이 로그를 쓰지 못하거나, 임시 파일을 만들지 못하거나, DB가 정상 동작하지 못할 수 있다.

```text
디스크 부족
-> df -h 확인
-> 어떤 마운트가 찼는지 확인
-> 로그 파일, 업로드 파일, 임시 파일, DB 데이터 파일 확인
-> 로그 로테이션이나 보관 정책 확인
```

디스크 문제는 코드 에러처럼 보이지 않을 수 있다.

예를 들어 애플리케이션 로그에는 단순히 파일 쓰기 실패, DB 쓰기 실패, 임시 파일 생성 실패처럼 찍힐 수 있다.
