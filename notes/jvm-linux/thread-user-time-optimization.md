# OpenJDK 스레드 CPU 시간 조회 최적화

## 한 줄 정리

숫자 하나를 얻기 위해 `/proc` 파일을 열고 문자열을 파싱하던 구현을, `clock_gettime()`으로 커널에 바로 물어보는 구현으로 바꿔 성능이 크게 좋아진 사례다.

## 헷갈렸던 점

처음에는 `/proc`이 가상메모리와 비슷한 것처럼 느껴질 수 있다.

하지만 둘은 다르다.

```text
/proc
커널 내부 상태를 파일처럼 읽게 해주는 가상 파일 시스템

가상메모리
프로세스마다 독립된 메모리 공간처럼 보이게 하는 메모리 관리 방식
```

`/proc/self/task/<tid>/stat`는 디스크에 저장된 일반 파일이라기보다, 읽는 순간 커널이 해당 스레드의 상태 정보를 문자열로 만들어 보여주는 인터페이스에 가깝다.

> [!note]
> `/proc`은 "커널 상태를 파일처럼 조회하는 창구"라고 보면 된다. 가상메모리와는 다른 개념이다.

## 이 API가 보려는 시간

`ThreadMXBean.getCurrentThreadUserTime()`은 현재 Java 스레드가 CPU에서 실제로 실행된 시간을 보려는 API다.

여기서 중요한 점은 "스레드가 살아 있던 시간"이 아니라는 것이다.

```text
스레드가 10초 동안 존재함
→ 9초는 DB 응답을 기다림
→ 0.8초는 Java 애플리케이션 코드 실행
→ 0.2초는 커널 코드 실행
```

이 경우 시간은 대략 이렇게 나뉜다.

```text
wall-clock time = 10초
user CPU time = 0.8초
system CPU time = 0.2초
total CPU time = 1.0초
```

`getCurrentThreadUserTime()`이 관심 있는 값은 이 중 `user CPU time`이다.

즉 현재 스레드가 애플리케이션 코드 쪽에서 CPU를 실제로 쓴 시간이다.

## 기존 구현이 느렸던 이유

기존 OpenJDK 구현은 Linux에서 `/proc/self/task/<tid>/stat`를 읽어서 현재 스레드의 user time을 구했다.

흐름은 대략 이렇다.

```text
현재 스레드 tid 확인
→ /proc/self/task/<tid>/stat 경로 만들기
→ 파일 열기
→ 커널이 만든 긴 문자열 읽기
→ 여러 필드 중 user time 필드 찾기
→ 문자열을 숫자로 변환
→ 단위 변환
→ Java API 결과로 반환
```

숫자 하나를 얻기 위해 파일 I/O 형태의 경로와 문자열 파싱을 모두 거친다.

> [!warning]
> `/proc`은 진짜 디스크 파일은 아니지만, 애플리케이션 입장에서는 파일을 열고 읽고 파싱하는 경로를 타기 때문에 숫자 하나를 얻는 방법으로는 무겁다.

## 기존 구현의 구체적인 문제

기존 `getCurrentThreadUserTime()`은 `/proc/self/task/<tid>/stat` 파일을 열어 CPU 시간을 계산했다.

이 파일에는 스레드 상태 정보가 한 줄 문자열로 들어 있고, 그 안의 특정 필드가 CPU 시간 정보를 담고 있다.

```text
utime = user mode에서 쓴 CPU time
stime = kernel mode에서 쓴 CPU time
```

즉 기존 구현은 user time 숫자 하나를 얻기 위해 다음 과정을 거쳤다.

```text
파일 경로 문자열 만들기
→ open()
→ read()
→ 커널이 /proc용 상태 문자열 생성
→ 버퍼에서 필요한 필드 위치 찾기
→ sscanf()로 숫자 변환
→ close()
```

여기서 까다로운 점은 `/proc/.../stat` 포맷이 단순히 공백으로 나누기 좋은 형태가 아니라는 것이다.

예를 들어 두 번째 필드에는 명령어 이름이 괄호로 감싸져 들어간다.

```text
12345 (java) S ...
```

그런데 명령어 이름 자체에 괄호가 포함될 수 있다.

그래서 단순히 첫 번째 `)`를 찾거나 공백으로 split하면 필드 위치가 틀어질 수 있다. 기존 구현에는 문자열에서 마지막 `)`를 찾기 위해 `strrchr()`를 사용하는 로직까지 들어갔다.

> [!note]
> 숫자 하나를 얻는 코드인데, 파일 포맷의 예외까지 신경 써야 했다. 이 자체가 `/proc` 경로가 너무 우회적이라는 신호다.

반면 `getCurrentThreadCpuTime()`은 이미 훨씬 단순한 경로를 사용했다.

```c
clock_gettime(CLOCK_THREAD_CPUTIME_ID, &ts);
```

이 방식은 현재 스레드의 CPU time을 전용 시스템 호출 한 번으로 가져온다.

그래서 두 API는 비슷한 정보를 다루지만 내부 비용이 크게 달랐다.

```text
getCurrentThreadUserTime()
→ /proc 파일 읽기 + 문자열 파싱

getCurrentThreadCpuTime()
→ clock_gettime() 한 번 호출
```

2018년 버그 리포트인 `JDK-8210452`에서는 두 메서드 사이의 속도 차이가 30배에서 400배까지 날 수 있다고 보고됐다.

## /proc 방식과 clock_gettime 방식 비교

`/proc` 방식은 단순히 파일을 읽는 것만이 아니다.

커널은 내부 상태를 `/proc` 포맷에 맞는 문자열로 만들어야 하고, 애플리케이션은 다시 그 문자열을 파싱해서 숫자로 바꿔야 한다.

```text
/proc 방식
커널 내부 상태
→ /proc용 문자열 생성
→ 애플리케이션이 문자열 읽기
→ 필드 위치 찾기
→ 숫자 파싱
```

`clock_gettime()` 방식은 목적이 더 직접적이다.

```text
clock_gettime 방식
커널에 특정 clock 값 요청
→ 커널이 시간 값 조회
→ timespec으로 반환
```

문단에서 말하는 `sched_entity`는 Linux 스케줄러가 태스크의 실행 시간 같은 정보를 관리할 때 사용하는 내부 구조체라고 보면 된다. `clock_gettime()` 경로는 이런 스케줄링/시간 accounting 정보에서 필요한 값을 더 직접적으로 조회하는 쪽에 가깝다.

또 하나 중요한 차이는 병렬 부하 상황이다.

많은 스레드가 동시에 `/proc`을 읽으면 커널도 여러 스레드의 상태 정보를 문자열로 만들어줘야 한다. 이 과정에서 커널 내부 자료구조 접근과 락 경합이 생길 수 있다.

```text
스레드 A → /proc stat 읽기
스레드 B → /proc stat 읽기
스레드 C → /proc stat 읽기
...
```

그래서 평균 호출 시간이 느린 것뿐 아니라, 부하가 걸릴수록 지연이 더 커질 수 있다.

## 백엔드 관점 비유

둘 다 결과는 "숫자 하나"지만 경로가 완전히 다르다.

```text
/proc 방식
관리자 페이지 HTML을 HTTP로 받아온 뒤,
문자열 파싱해서 주문 상태 숫자 하나 뽑기

clock_gettime 방식
메트릭 API나 DB 쿼리로 주문 상태 숫자 하나만 바로 조회하기
```

HTML을 파싱하는 방식은 사람이 보기 좋은 보고서를 기계가 다시 뜯어 읽는 느낌이다.

반면 메트릭 API나 DB 쿼리는 처음부터 기계가 필요한 값을 바로 받는 느낌이다.

`/proc`과 `clock_gettime()`의 차이도 비슷하다.

```text
/proc
커널 상태 보고서를 문자열로 받아서 필요한 필드를 찾음

clock_gettime()
커널에 원하는 시간 값을 바로 요청함
```

> [!summary]
> 느린 이유는 단순히 "파일을 읽어서"가 아니다. 커널 상태를 문자열 보고서로 만들고, 애플리케이션이 다시 그 문자열을 파싱하는 우회 경로를 탔기 때문이다.

## 새 구현이 빨라진 이유

새 구현은 `/proc`을 읽지 않고 `clock_gettime()`을 사용한다.

```c
clock_gettime(clockid, &timespec);
```

이 함수는 커널에 특정 clock의 시간을 바로 물어보는 API다.

개선된 흐름은 대략 이렇다.

```text
현재 스레드의 clock id 준비
→ Linux의 clockid_t 인코딩 규칙에 맞게 일부 비트 조정
→ user CPU time용 clock id 생성
→ clock_gettime(clockid) 호출
→ 결과 반환
```

즉 기존 방식이 "문자열 보고서에서 숫자 하나를 찾아 읽는 방식"이었다면, 새 방식은 "커널에게 숫자 하나를 바로 물어보는 방식"이다.

> [!summary]
> 성능 차이의 핵심은 `/proc` 파일 파싱 경로를 없애고, 커널의 시간 조회 API를 직접 사용했다는 점이다.

## 내부 변수를 직접 읽은 것은 아니다

처음에는 "Linux 내부 변수값을 직접 가져온 것"처럼 이해할 수 있다.

하지만 정확히는 커널 내부 메모리 변수를 직접 읽은 것이 아니다.

더 정확한 표현은 다음과 같다.

```text
Linux가 제공하는 clock_gettime()이라는 공식 시스템 호출을 사용했다.
다만 여기에 넘기는 clockid_t 값을 Linux 내부 ABI 규칙에 맞게 만들어,
현재 스레드의 user CPU time을 바로 조회하게 했다.
```

`clockid_t`는 겉으로는 단순한 clock 종류처럼 보일 수 있다.

```text
CLOCK_REALTIME
CLOCK_MONOTONIC
CLOCK_THREAD_CPUTIME_ID
```

하지만 Linux에서는 특정 프로세스나 스레드의 CPU clock을 표현하기 위해 `clockid_t` 안에 대상과 clock 종류를 비트로 인코딩할 수 있다.

이번 최적화는 이 Linux 내부 ABI를 이해하고, `pthread_getcpuclockid()`로 얻은 값의 일부 비트를 조정해서 user time 전용 clock처럼 사용한 것이다.

## 커널 fast-path 추가 최적화

`/proc` 경로를 없애고 `clock_gettime()`을 쓰는 것만으로도 큰 개선이 있었다.

그런데 여기서 한 번 더 최적화할 수 있는 지점이 있었다.

일반적인 흐름은 다음과 같다.

```text
pthread_getcpuclockid()로 현재 스레드의 clockid 얻기
→ clockid 일부 비트 조정
→ clock_gettime(clockid) 호출
```

추가 최적화는 이 중 `pthread_getcpuclockid()`에 기대지 않고, JVM이 직접 `clockid`를 구성하는 방식이다.

Linux의 `clockid_t`는 내부적으로 대상과 clock 종류를 비트로 인코딩할 수 있다.

```text
대상이 프로세스인지 스레드인지
어떤 시간인지
대상 pid/tid가 무엇인지
```

여기서 중요한 규칙이 있다.

`clockid`에 PID=0이 인코딩된 경우, Linux 커널은 이것을 "현재 스레드"로 보고 빠른 경로를 탈 수 있다.

일반적인 clockid는 대상 스레드를 찾아야 한다.

```text
clockid 안의 tid 확인
→ 이 tid에 해당하는 task_struct 찾기
→ radix tree 같은 자료구조 탐색
→ 해당 스레드의 시간 정보 조회
```

하지만 PID=0이면 현재 실행 중인 스레드를 의미하므로 커널이 바로 접근할 수 있다.

```text
PID=0 확인
→ 현재 스레드로 바로 판단
→ current 사용
→ 시간 정보 조회
```

이것이 fast-path다.

> [!note]
> fast-path는 자주 쓰이는 특수 케이스를 더 짧은 경로로 처리하는 최적화다. 여기서는 "현재 스레드"라는 특수 케이스를 별도 탐색 없이 바로 처리한다.

## PID=0을 /me로 이해하기

백엔드 관점으로 보면 PID=0은 `/me` 같은 느낌이다.

```http
GET /users/123/profile
```

이 요청은 서버가 `123`번 사용자를 찾아야 한다.

반면 다음 요청은 의미가 다르다.

```http
GET /me/profile
```

`/me`는 "현재 로그인한 사용자"를 뜻하므로, 서버는 요청 컨텍스트나 세션에서 현재 사용자를 바로 꺼낼 수 있다.

`clockid`도 비슷하게 볼 수 있다.

```text
일반 clockid
"tid가 12345인 스레드를 찾아서 그 CPU time을 줘"

PID=0 clockid
"현재 실행 중인 이 스레드의 CPU time을 줘"
```

그래서 PID=0을 넣은 clockid는 커널이 별도 탐색을 생략하고 현재 스레드 정보에 바로 접근할 수 있다.

## 추가 개선의 크기

이 fast-path를 사용하면 이미 빠른 `clock_gettime()` 경로에서 조금 더 빨라진다.

```text
pthread_getcpuclockid()로 얻은 clockid 사용
→ 평균 81.7ns

PID=0을 넣어 직접 만든 clockid 사용
→ 평균 70.8ns
```

약 13% 정도 추가 개선이다.

절대값으로는 약 10ns 정도 차이라 작아 보일 수 있다. 하지만 이 API가 매우 자주 호출되는 환경에서는 의미가 있을 수 있다.

## 왜 조심해야 하는가

이 최적화는 성능상 이점이 있지만 대가가 있다.

`pthread_getcpuclockid()`를 쓰면 표준 라이브러리나 OS가 clockid 구성을 대신 해준다.

```text
JVM
→ "현재 스레드용 clockid 주세요"
→ pthread_getcpuclockid()
→ clockid 반환
```

반면 JVM이 직접 clockid를 구성하면 Linux 내부 규칙을 코드가 직접 알아야 한다.

```text
clockid_t에서 pid는 어느 비트에 들어가는가?
PID=0은 현재 스레드를 의미하는가?
user time을 나타내는 비트는 무엇인가?
clockid_t 크기는 몇 비트인가?
아키텍처마다 같은가?
커널 버전이 바뀌어도 유지되는가?
```

그래서 가독성과 호환성 손실 우려가 생긴다.

백엔드로 비유하면 공식 SDK를 쓰는 대신, SDK 내부에서 쓰는 비공개 URL과 토큰 포맷을 보고 직접 요청을 조립하는 것과 비슷하다.

```text
안전한 방식
client.getCurrentUserProfile()

더 빠를 수 있지만 위험한 방식
SDK 내부 URL과 토큰 포맷을 보고 직접 HTTP 요청 조립
```

지금은 더 빠를 수 있다. 하지만 내부 규칙이 바뀌거나 플랫폼 차이가 생기면 깨질 수 있고, 나중에 코드를 읽는 사람도 왜 이런 비트 조작을 하는지 이해하기 어려워진다.

> [!warning]
> `/proc`에서 `clock_gettime()`으로 바꾼 것은 큰 우회 경로를 제거한 구조적 개선이다. 반면 PID=0 fast-path는 이미 빠른 경로에서 Linux 내부 ABI에 더 기대어 몇 ns를 줄이는 미세 최적화에 가깝다.

## 가져갈 포인트

성능 문제는 알고리즘만의 문제가 아니다.

같은 값을 얻더라도 어떤 경로로 얻는지에 따라 비용이 크게 달라질 수 있다.

```text
무거운 경로
파일 열기 + 문자열 읽기 + 필드 파싱 + 숫자 변환

가벼운 경로
전용 시스템 호출로 값 직접 조회
```

평소에는 Java API라는 추상화만 보고 써도 된다. 하지만 저수준 성능 병목을 잡을 때는 그 API가 내부에서 실제로 어떤 OS 경로를 타는지 확인하면 큰 최적화 포인트를 찾을 수 있다.

또한 모든 최적화의 성격이 같지는 않다.

```text
/proc → clock_gettime()
복잡한 문자열 파싱 경로를 없앤 구조적 개선

pthread_getcpuclockid() → 직접 clockid 구성
더 빠르지만 Linux 내부 ABI 의존이 커지는 미세 최적화
```

성능 이득이 크고 안정성 손실이 작으면 좋은 최적화다. 반대로 이득이 작고 내부 구현 의존이 커지면, 실제로 그 비용을 감수할 만큼 호출 빈도와 성능 요구가 큰지 따져야 한다.

## 분포가 넓다는 것의 의미

성능 측정에서 분포가 넓게 퍼진다는 것은 같은 코드가 매번 같은 비용으로 실행되지 않는다는 뜻이다.

예를 들어 어떤 함수를 여러 번 측정했을 때 다음처럼 나온다면 분포가 좁다.

```text
70ns
72ns
71ns
73ns
70ns
72ns
```

이 경우에는 "이 함수는 대체로 70ns 근처에서 안정적으로 실행된다"고 볼 수 있다.

반대로 다음처럼 나오면 이야기가 달라진다.

```text
70ns
75ns
72ns
1,200ns
74ns
15,000ns
73ns
2,500ns
```

평균을 하나 낼 수는 있지만, 더 중요한 질문은 이것이다.

```text
왜 어떤 호출은 70ns인데,
어떤 호출은 15,000ns까지 튀는가?
```

분포가 넓다는 것은 평균 하나로 설명되지 않는 다른 원인이 섞여 있을 수 있다는 신호다.

가능한 원인은 여러 가지다.

- 커널 락 경합
- 스레드 스케줄링 지연
- 캐시 미스
- 인터럽트
- CPU 코어 이동
- CPU 전원 상태 변화
- 느린 경로로 빠지는 조건

`/proc` 방식에서도 비슷하게 볼 수 있다.

```text
대부분은 빠르게 /proc을 읽음
→ 병렬 부하가 걸릴 때 가끔 크게 느려짐
→ 커널 내부 락 경합이나 문자열 생성 경로를 의심할 수 있음
```

즉 분포가 넓다는 것은 단순히 "평균이 높다"는 뜻이 아니다. 특정 조건에서 느린 경로가 섞이거나, 경합이 생기거나, 시스템이 개입하고 있다는 단서일 수 있다.

> [!summary]
> 분포가 넓다는 것은 성능 문제가 "항상 조금 느림"이 아니라 "대부분 괜찮지만 특정 조건에서 확 느려짐"일 수 있다는 신호다.

## 백엔드 API 성능과의 연결

백엔드 API에서도 같은 관점이 필요하다.

```text
대부분 요청: 30ms
가끔 요청: 2,000ms
```

이런 경우 평균만 보면 괜찮아 보일 수 있다. 하지만 실무에서는 다음 질문을 해야 한다.

```text
왜 가끔 2초가 걸리지?
DB 커넥션 풀 대기인가?
락 경합인가?
캐시 미스인가?
GC인가?
외부 API 지연인가?
특정 쿼리 플랜 문제인가?
```

그래서 평균뿐 아니라 p95, p99, max, outlier를 같이 봐야 한다.

```text
평균 응답 시간: 50ms
p99 응답 시간: 3000ms
```

이 경우 평균은 좋아 보여도, 100명 중 1명 정도는 3초짜리 요청을 겪을 수 있다. 사용자는 이것을 "가끔 사이트가 멈춘다"고 느낀다.

성능 최적화는 평균을 낮추는 것뿐 아니라 튀는 값을 줄여 예측 가능성을 높이는 일이기도 하다.

## 한 줄 회고

- 헷갈렸던 점: `/proc`을 가상메모리처럼 생각한 점, `clock_gettime()` 최적화를 Linux 내부 변수를 직접 읽는 것으로 이해한 점, 숫자 하나를 얻기 위해 문자열 보고서를 파싱하는 비용이 왜 큰지 감이 잘 안 왔던 점, PID=0 fast-path가 왜 빠르지만 조심해야 하는지 헷갈렸던 점, 그리고 성능 분포가 넓게 퍼지는 것 자체가 왜 병목의 단서가 되는지 헷갈렸던 점.
