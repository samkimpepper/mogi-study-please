# 모바일과 데스크탑 개발환경 차이

> [!summary]
> 모바일과 데스크탑은 화면 크기만 다른 컴퓨터가 아니다.
>
> 데스크탑은 사용자가 시스템을 직접 조작하는 환경에 가깝고, 모바일은 앱이 정해진 공간 안에서 안전하게 동작하는 환경에 가깝다.

## 처음 헷갈렸던 질문

> [!question]
> 아이폰에서도 터미널을 열고 Claude Code를 실행하면 되는 것 아닌가?
>
> 왜 맥북이나 홈서버에 원격 접속해서 써야 할까?

Claude Code는 단순한 채팅앱이 아니라 개발자 CLI 도구에 가깝다.

맥북에서는 다음 흐름이 자연스럽다.

```text
터미널 실행
-> 프로젝트 폴더 이동
-> Claude Code CLI 실행
-> 파일 검색
-> 코드 수정
-> git diff 확인
-> 테스트 실행
```

아이폰에서는 이 흐름이 자연스럽지 않다. iOS는 앱마다 강하게 분리된 샌드박스 환경을 기본으로 하기 때문이다.

## 데스크탑의 기본 감각

데스크탑 OS에서는 사용자가 시스템을 직접 조작할 수 있다.

```text
~/projects/my-app
~/Downloads
/usr/bin
/var/log
```

터미널에서 파일을 찾고, 명령어를 실행하고, 개발 서버를 띄우는 일이 자연스럽다.

```bash
cd ~/projects/backend
rg "UserService"
git diff
./gradlew test
pnpm dev
```

백엔드 개발자 입장에서는 데스크탑이 이런 작업을 하기 좋은 환경이다.

- 프로젝트 폴더를 자유롭게 다룰 수 있다.
- `git`, `node`, `npm`, `gradle`, `docker` 같은 도구를 설치할 수 있다.
- 터미널 프로세스를 오래 띄워둘 수 있다.
- 개발 서버와 테스트 프로세스를 로컬에서 실행할 수 있다.
- 여러 도구가 같은 프로젝트 파일을 함께 읽고 쓸 수 있다.

## 모바일의 기본 감각

모바일 OS는 앱 격리와 권한 제한을 강하게 둔다.

```text
Claude 앱 공간
Termius 앱 공간
Working Copy 앱 공간
Files 앱 공간
Photos 앱 공간
```

각 앱은 기본적으로 자기 앱 공간 안에서 동작한다. 다른 앱의 내부 파일을 마음대로 읽거나 수정할 수 없다.

사용자가 파일 선택 창에서 명시적으로 골라주거나, iOS가 허용한 공유 방식으로 넘겨줘야 접근할 수 있는 경우가 많다.

> [!info]
> 모바일의 기본 철학은 "사용자가 시스템 전체를 마음대로 다룬다"보다 "각 앱이 허용된 권한 안에서 안전하게 동작한다"에 가깝다.

그래서 아이폰 로컬에서 다음 흐름을 만들기 어렵다.

```bash
cd ~/projects/backend
rg "UserService"
git diff
./gradlew test
```

공통 프로젝트 폴더에 여러 개발 도구가 붙어서 작업하는 방식이 모바일의 기본 모델과 잘 맞지 않는다.

## 파일시스템 차이

데스크탑에서는 하나의 프로젝트 폴더를 여러 도구가 같이 다룰 수 있다.

```text
VS Code
터미널
git
Claude Code
테스트 러너
빌드 도구
```

모두 같은 파일을 읽고 수정한다.

```text
~/projects/my-app
```

모바일에서는 앱별 샌드박스가 기본이라 이런 식의 자유로운 공유 작업이 어렵다.

| 구분 | 데스크탑 | 모바일 |
| --- | --- | --- |
| 파일 접근 | 사용자가 넓게 접근 | 앱별 샌드박스 중심 |
| 프로젝트 폴더 | 여러 도구가 같이 접근 | 앱 간 접근 제한이 큼 |
| 파일 수정 | 터미널/에디터/git이 같이 작업 | 앱이 허용한 방식 중심 |

## 프로세스와 백그라운드 실행 차이

데스크탑에서는 터미널 프로세스를 오래 띄워둘 수 있다.

```bash
pnpm dev
docker compose up
tail -f app.log
claude
```

개발 서버를 계속 켜두고, 로그를 보고, 테스트를 돌리는 일이 자연스럽다.

모바일은 배터리와 보안 때문에 백그라운드 실행이 제한된다. 앱을 내리거나 화면이 꺼지면 OS가 실행을 멈추거나 제한할 수 있다.

> [!warning]
> 긴 빌드, 테스트, 개발 서버, 파일 watch 작업은 모바일에서 안정적으로 돌리기 어렵다.
>
> 모바일은 이런 프로세스를 직접 실행하는 머신보다, 이미 켜져 있는 개발 머신에 접속하는 클라이언트 역할이 더 잘 맞는다.

## Claude Code가 아이폰 로컬에 안 맞는 이유

Claude Code는 내 프로젝트 파일을 직접 읽고 수정하는 CLI 도구다.

```text
내 컴퓨터 터미널에서 실행
-> 프로젝트 파일 읽기
-> 명령어 실행
-> 코드 수정
-> git diff 확인
```

그래서 실행 환경에 다음 요소들이 필요하다.

- 프로젝트 파일
- shell
- `git`
- 빌드 도구
- 테스트 명령어
- 패키지 매니저
- 오래 살아 있는 프로세스

아이폰은 이런 환경을 로컬에서 구성하기에 제약이 많다.

Claude 앱으로 질문을 하는 것은 가능하지만, Claude Code처럼 로컬 프로젝트를 직접 조작하는 개발 CLI를 돌리는 것은 맥북/PC/서버 쪽이 훨씬 자연스럽다.

## 현실적인 사용 구조

아이폰에서 직접 Claude Code를 돌리기보다, 맥북이나 홈서버에 접속해서 거기서 실행하는 방식이 현실적이다.

```text
아이폰
-> SSH 또는 원격 터미널 앱
-> 맥북/홈서버
-> Claude Code 실행
-> 프로젝트 파일 수정
```

Tailscale을 같이 쓰면 외부에서도 안전하게 접속할 수 있다.

```text
아이폰
-> Tailscale 사설망
-> 맥북 SSH
-> Claude Code
```

이 구조에서 아이폰은 개발 머신이 아니라 화면과 키보드 역할을 한다. 실제 개발환경은 맥북이나 서버가 담당한다.

## 비교해서 이해하기

| 구분 | 데스크탑 | 모바일 |
| --- | --- | --- |
| OS 철학 | 사용자 제어권이 큼 | 앱 격리와 권한 제한 중심 |
| 터미널 | 시스템 조작의 핵심 도구 | 제한적이거나 원격 접속용 |
| 파일시스템 | 프로젝트 폴더를 자유롭게 다룸 | 앱별 샌드박스 중심 |
| 프로세스 | 오래 실행 가능 | 백그라운드 제한 강함 |
| 개발 서버 | 로컬에서 자연스럽게 실행 | 직접 운영하기 부적합 |
| 개발 도구 | 설치와 실행이 쉬움 | 설치/실행 제약이 큼 |
| 개발에서 역할 | 실제 개발환경 | 원격 접속 클라이언트 |

## 나중에 다시 볼 포인트

- 모바일과 데스크탑은 화면 크기 차이가 아니라 OS 사용 모델 차이가 크다.
- 데스크탑은 사용자가 파일, 프로세스, 네트워크를 직접 조작하기 쉽다.
- 모바일은 앱별 샌드박스와 권한 제한이 강하다.
- Claude Code는 채팅앱이 아니라 로컬 프로젝트를 조작하는 CLI 도구에 가깝다.
- 아이폰은 Claude Code 실행 머신보다 맥북/홈서버에 접속하는 리모컨 역할이 더 잘 맞는다.
