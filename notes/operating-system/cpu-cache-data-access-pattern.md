# CPU를 화나게 만드는 데이터 접근 패턴

원문 링크: https://news.hada.io/topic?id=30910

관련 개념 노트: [캐시 라인과 페이지는 뭐가 다른가](cache-line-vs-page.md)

> [!summary]
> 같은 정수 합산 루프라도 데이터를 어떤 순서로 읽느냐에 따라 실행 시간이 크게 달라진다.
>
> CPU는 가까운 데이터를 예측 가능한 순서로 읽는 패턴에 강하고, 캐시 라인, 페이지, TLB, DRAM 지역성을 계속 깨는 패턴에는 약하다.

## 처음 헷갈렸던 질문

> [!question]
> 같은 `sum += data[positions[i]]` 루프인데 왜 접근 순서만 바꿨다고 실행 시간이 10배 이상 차이 날까?
>
> 그리고 랜덤 접근보다 더 나쁜 접근 패턴은 어떻게 만들 수 있을까?

## 한 줄 정의

*==CPU 캐시 관점에서 좋은 데이터 접근 패턴은 "가까운 데이터를, 예측 가능한 순서로, 한 번 가져온 캐시 라인을 최대한 재사용하는 패턴"이다.==*

반대로 CPU를 괴롭히는 패턴은 메모리 지역성을 일부러 깨는 패턴이다. 데이터는 같은 배열 안에 있어도, 읽는 순서가 계속 튀면 CPU는 다음 데이터를 미리 가져오기 어렵고 캐시도 제대로 써먹지 못한다.

## 실험의 핵심

실험의 목표는 단순했다.

```cpp
sum += data[positions[i]];
```

`data` 배열의 정수를 더하는 코드는 고정하고, `positions` 순열만 바꿔서 가장 느린 실행 시간을 만드는 것이다.

데이터 크기는 `2^26`개의 `uint32_t` 정수였고, 4KB 페이지 65,536개를 사용했다. 측정은 `positions` 생성 시간이 아니라 실제 누산 루프 실행 시간만 `rdtsc` 사이클 카운트로 비교했다.

## 선형 접근이 빠른 이유

가장 단순한 패턴은 배열을 앞에서부터 차례대로 읽는 것이다.

```text
positions = 0, 1, 2, 3, 4, ...
```

이 경우 CPU 입장에서는 다음에 읽을 위치가 거의 뻔하다.

> 아, 이 코드는 배열을 앞에서부터 쭉 읽고 있구나. 다음 캐시 라인을 미리 가져오면 되겠다.

그래서 프리페처가 잘 맞고, 캐시 라인 안의 가까운 데이터도 재사용하기 쉽다. 원문 실험에서는 선형 접근이 약 `1.33억` 사이클로 가장 빠른 축에 속했다.

## 무작위 접근이 느린 이유

무작위 순열은 이런 식이다.

```text
positions = 38192, 7, 999999, 42, ...
```

이제 CPU는 다음에 어디를 읽을지 예측하기 어렵다. 이미 가져온 캐시 라인 주변 데이터를 다시 쓸 가능성도 낮다.

> 다음 데이터가 어디 있지?
> 캐시에 없네.
> 메모리까지 가야겠네.
> 또 다음 위치도 완전히 다르네.

이렇게 되면 같은 덧셈 루프라도 대부분의 시간이 메모리 접근을 기다리는 데 쓰인다. 원문 실험에서는 무작위 접근이 약 `15.7억` 사이클로, 선형 접근보다 10배 이상 느려졌다.

## 랜덤보다 더 나쁜 패턴

이 글의 재미있는 지점은 "랜덤 접근이 최악인가?"를 묻는 데 있다. 결론은 아니었다.

랜덤보다 더 CPU를 괴롭히려면, 단순히 아무 데나 읽는 것보다 CPU의 여러 계층 최적화를 동시에 망가뜨리는 순서를 만들면 된다.

| 괴롭히는 방식 | 무슨 뜻인가 | 왜 느려지는가 |
| --- | --- | --- |
| 캐시 라인 안의 가까운 데이터를 재사용하지 못하게 함 | 한 번 64바이트 정도를 가져와도 그 안의 다른 값을 거의 안 씀 | 메모리에서 가져온 묶음을 낭비함 |
| 페이지 경계를 계속 넘게 함 | 4KB 페이지를 하나씩 순서대로 쓰지 않고 계속 다른 페이지로 튐 | 주소 변환과 페이지 지역성이 나빠짐 |
| 같은 캐시 set에 몰리게 함 | 서로 다른 주소들이 캐시의 같은 자리 후보에 몰림 | 캐시에 넣은 데이터가 자꾸 밀려남 |
| TLB/PTE 캐시 지역성을 깨버림 | 가상 주소를 물리 주소로 바꾸는 정보도 계속 새로 필요해짐 | 주소 변환 쪽 캐시도 잘 못 씀 |
| DRAM bank/row 충돌 가능성을 노림 | 메모리 내부에서도 가까운 row를 활용하지 못하게 함 | DRAM 접근 자체도 더 비싸질 수 있음 |

결과적으로 CPU는 이런 상태가 된다.

> 미리 가져왔는데 틀림.
> 캐시에 넣었는데 곧바로 밀림.
> 주소 변환 정보도 없음.
> 메모리 내부에서도 가까운 위치가 아님.

그래서 원문 실험에서는 단순 무작위 접근보다 더 느린 `20.6억~20.8억` 사이클 수준의 패턴도 나왔다.

## 캐시 라인 단위로 접근을 벌리면

CPU는 메모리에서 값을 4바이트씩 딱 하나만 가져오는 것이 아니라, 보통 64바이트 캐시 라인 단위로 묶어서 가져온다. `uint32_t` 정수 하나가 4바이트라면, 캐시 라인 하나에는 정수 16개가 들어간다.

```text
캐시 라인 1
data[0], data[1], data[2], ... data[15]

캐시 라인 2
data[16], data[17], data[18], ... data[31]
```

선형 접근은 이 캐시 라인을 잘 써먹는다.

```text
data[0]
data[1]
data[2]
data[3]
...
```

`data[0]`을 읽으려고 캐시 라인 하나를 가져오면, 그 안에 `data[1]`부터 `data[15]`까지 같이 들어온다. 그래서 다음 값들은 이미 캐시에 있는 데이터를 읽게 된다.

반대로 캐시 라인 하나씩 떨어져 접근하면 이런 식이 된다.

```text
data[0]
data[16]
data[32]
data[48]
...
```

매번 다른 캐시 라인으로 넘어가는 것이다.

```text
data[0]  -> 캐시 라인 1에서 정수 하나만 씀
data[16] -> 캐시 라인 2에서 정수 하나만 씀
data[32] -> 캐시 라인 3에서 정수 하나만 씀
```

이러면 CPU가 64바이트를 가져와도 그중 4바이트만 쓰고 다음 캐시 라인으로 넘어가는 꼴이 된다.

> 편의점에서 물 한 병만 마시려고 16개들이 박스를 매번 사 오는데, 한 병만 마시고 다음 박스로 넘어가는 느낌이다.

그래서 원문 실험에서 캐시 라인 간격 접근은 약 `7.18억` 사이클이 걸렸다. 선형 접근의 약 `1.33억` 사이클보다 4배 이상 느리다.

다만 이 패턴은 아직 완전한 최악은 아니다. 접근 순서가 `0, 16, 32, 48...`처럼 일정하기 때문이다. 하드웨어 프리페처는 이런 단순 stride 패턴을 감지해서 다음 캐시 라인을 미리 가져올 수 있다.

> 얘가 계속 64바이트씩 건너뛰면서 읽네. 그럼 다음 캐시 라인을 미리 가져와야겠다.

즉 캐시 라인 재사용은 망가졌지만, 프리페처가 어느 정도 도와줄 여지는 남아 있다.

## 페이지 단위로 접근을 벌리면

다음 악화 패턴은 캐시 라인보다 더 크게, 4KB 페이지 단위로 접근을 벌리는 것이다.

페이지 하나는 보통 4,096바이트다. `uint32_t` 정수는 4바이트이므로 한 페이지에는 정수 1,024개가 들어간다.

```text
페이지 0: data[0]    ~ data[1023]
페이지 1: data[1024] ~ data[2047]
페이지 2: data[2048] ~ data[3071]
```

페이지 간격 접근은 대략 이런 식이다.

```text
data[0]
data[1024]
data[2048]
data[3072]
...
```

각 페이지의 같은 오프셋을 하나씩 찍고 넘어가는 것이다.

```text
page 0의 첫 번째 정수
page 1의 첫 번째 정수
page 2의 첫 번째 정수
page 3의 첫 번째 정수
```

이 패턴이 더 나쁜 이유는 많은 Intel CPU의 하드웨어 데이터 프리페처가 4KB 페이지 경계를 넘어서는 프리페치를 조심하기 때문이다. 페이지가 바뀌면 가상 주소를 물리 주소로 변환하는 정보도 달라질 수 있고, 권한이나 매핑도 달라질 수 있다.

캐시 라인 간격 접근은 그래도 일정한 stride라 프리페처가 따라갈 여지가 있었다.

```text
data[0]
data[16]
data[32]
data[48]
```

하지만 페이지 간격 접근은 계속 페이지를 바꾼다.

```text
data[0]       page 0
data[1024]    page 1
data[2048]    page 2
data[3072]    page 3
```

이러면 프리페처가 덜 도와주고, TLB나 페이지 테이블 관련 캐시에도 부담이 커진다. 원문 실험에서 페이지 간격 접근은 약 `14.11억` 사이클이 걸렸다. 캐시 라인 간격 접근보다도 거의 두 배 느려진 것이다.

| 접근 방식 | 예시 | CPU 입장 |
| --- | --- | --- |
| 선형 접근 | `0, 1, 2, 3...` | 캐시 라인 재사용 좋고 프리페처도 잘 맞음 |
| 캐시 라인 간격 | `0, 16, 32, 48...` | 캐시 라인마다 하나만 써서 낭비, 그래도 stride 예측 가능 |
| 페이지 간격 | `0, 1024, 2048...` | 페이지를 계속 넘어서 프리페처와 TLB 효율이 더 나빠짐 |

백엔드 감각으로 보면 선형 접근은 DB 페이지 하나를 읽어왔을 때 그 안의 row들을 쭉 처리하는 느낌이다.

```text
page 1 row 1
page 1 row 2
page 1 row 3
```

반대로 페이지 간격 접근은 매번 다른 데이터 페이지에서 row 하나씩만 찍고 넘어가는 느낌이다.

```text
page 1에서 하나
page 500에서 하나
page 999에서 하나
```

> [!summary]
> 캐시 라인 단위로 떨어져 읽으면 한 번 가져온 64바이트 중 4바이트만 써서 낭비가 크다. 그래도 일정한 간격이면 프리페처가 어느 정도 따라올 수 있다. 하지만 접근 간격을 4KB 페이지 단위로 벌리면 프리페처가 페이지 경계를 잘 넘지 못하고, 주소 변환 지역성도 깨져서 훨씬 느려진다.

## 같은 캐시 set에 몰린다는 뜻

L1 캐시는 그냥 큰 바구니 하나가 아니다. 내부가 여러 개의 set으로 나뉘어 있고, 각 메모리 주소는 들어갈 수 있는 set이 정해져 있다.

실험 머신의 L1d 캐시는 대략 이런 구조였다.

```text
L1d = 48KB
64 set
12-way
cache line = 64B
```

단순화하면 이런 모양이다.

```text
set 0  : cache line 최대 12개
set 1  : cache line 최대 12개
set 2  : cache line 최대 12개
...
set 63 : cache line 최대 12개
```

중요한 점은 아무 데이터나 아무 set에 들어가는 것이 아니라는 점이다. CPU는 주소의 일부를 보고 set 번호를 정한다.

캐시 라인은 64B이고 set은 64개다.

```text
64B * 64 set = 4096B
```

그래서 주소가 4,096바이트 차이나면 같은 set으로 돌아올 수 있다.

```text
주소 A          -> set 7
주소 A + 4096   -> set 7
주소 A + 8192   -> set 7
주소 A + 12288  -> set 7
```

페이지 크기도 보통 4,096B다. 그래서 각 페이지의 같은 오프셋만 계속 찍으면 같은 set에 몰리기 쉽다.

```text
page 0의 offset 0 -> set 0
page 1의 offset 0 -> set 0
page 2의 offset 0 -> set 0
page 3의 offset 0 -> set 0
```

한 set에는 12개 way, 즉 캐시 라인 12개까지만 들어갈 수 있다. 같은 set으로 13번째 캐시 라인이 들어오면 기존 라인 하나를 내쫓아야 한다.

```text
set 0에 12개가 이미 있음
-> 새 라인도 set 0에 들어와야 함
-> 기존 라인 하나 퇴출
```

이렇게 되면 L1 전체 용량은 48KB여도 실제로는 한 set만 괴롭히는 꼴이 된다.

```text
12 way * 64B = 768B
```

즉 48KB짜리 캐시를 가지고도, 특정 접근 패턴에서는 768B짜리 캐시처럼 동작할 수 있다.

> 큰 주차장 전체는 비어 있는데, 모든 차가 0번 구역에만 주차하려고 해서 0번 구역만 계속 만차가 되는 느낌이다.

## 같은 set 몰림을 피하는 방법

실무에서 set 번호를 직접 계산하면서 코딩하는 경우는 드물다. 대신 원칙은 단순하다.

*==같은 간격으로 멀리 뛰며 같은 위치만 찍지 말고, 가까운 데이터를 충분히 쓰고 넘어가야 한다.==*

좋은 기본값은 순차 접근이다.

```text
data[0]
data[1]
data[2]
data[3]
...
```

나쁜 패턴은 페이지마다 같은 오프셋만 찍는 것이다.

```text
page 0 offset 0 -> set 0
page 1 offset 0 -> set 0
page 2 offset 0 -> set 0
page 3 offset 0 -> set 0
```

반대로 같은 페이지 안에서 옆으로 쭉 읽으면 set을 고르게 지나가고, 캐시 라인도 잘 재사용한다.

```text
page 0 offset 0    -> set 0
page 0 offset 64B  -> set 1
page 0 offset 128B -> set 2
page 0 offset 192B -> set 3
```

피하는 방법은 다음과 같다.

| 안 좋은 패턴 | 피하는 방향 |
| --- | --- |
| 페이지마다 하나씩 찍기 | 한 페이지나 한 청크를 충분히 처리하고 넘어가기 |
| `A`, `A + 4096`, `A + 8192`처럼 고정 stride 접근 | 접근 순서를 바꾸거나 stride를 비틀기 |
| row-major 배열을 세로로 읽기 | 가능하면 row 방향으로 순차 접근하기 |
| 캐시 set 주기와 딱 맞는 크기의 row/구조 | 성능 민감 코드에서는 padding으로 간격 비틀기 |
| 포인터를 따라 흩어진 객체 순회 | 배열이나 연속 자료구조 우선 고려하기 |

면접식으로 말하면 다음처럼 정리할 수 있다.

> 집합 연관 캐시는 주소에 따라 들어갈 set이 정해져 있기 때문에, 4KB처럼 캐시 set 주기와 맞는 stride로 접근하면 여러 캐시 라인이 같은 set에 몰려 충돌할 수 있습니다. 이를 피하려면 순차 접근을 우선하고, 페이지나 row를 건너뛰며 같은 오프셋만 찍는 패턴을 줄이며, 필요한 경우 접근 순서 변경이나 padding으로 stride를 비틀 수 있습니다.

## 페이지 stride 8과 PTE까지 괴롭히기

다음 변형은 연속 페이지가 아니라 `N`개 페이지 간격으로 접근하는 패턴이다.

예를 들어 stride가 8이면 다음처럼 접근한다.

```text
page 0
page 8
page 16
page 24
page 32
...
```

페이지 하나가 4KB이므로, stride 8은 매번 32KB씩 떨어진 위치를 읽는 것이다.

```text
4KB * 8 = 32KB
```

원문에서는 여러 stride를 측정했고, 페이지 stride가 8일 때 가장 나쁜 결과가 나왔다. `-DSTRIDE=8`로 단독 실행했을 때 약 `20.58억` 사이클이 걸렸고, 단순 무작위 접근보다도 느렸다.

가능한 이유 중 하나는 주소 변환이다.

프로그램은 메모리에 접근할 때 가상 주소를 사용한다. CPU의 MMU는 이 가상 주소를 실제 물리 주소로 바꿔야 한다.

```text
프로그램: 이 가상 주소를 읽어줘
MMU: 이 가상 주소는 실제 물리 메모리 어디지?
```

이때 사용하는 주소 변환 기록이 page table entry, 즉 PTE다.

> PTE는 "이 가상 페이지는 실제 물리 메모리의 어디에 매핑되어 있다"를 적어둔 작은 기록이다.

PTE 하나는 8바이트이고, 캐시 라인 하나는 64바이트다. 그래서 캐시 라인 하나에는 PTE 8개가 들어간다.

```text
PTE 하나 = 8B
캐시 라인 하나 = 64B

64B / 8B = PTE 8개
```

즉 페이지 8개에 대한 주소 변환 정보가 PTE 캐시 라인 하나에 같이 들어갈 수 있다.

```text
PTE cache line
[PTE page0][PTE page1][PTE page2]...[PTE page7]
```

그런데 stride가 8이면 접근 페이지가 이렇게 된다.

```text
page 0
page 8
page 16
page 24
```

이 경우 데이터 캐시 라인뿐 아니라 PTE 캐시 라인도 계속 새로 찍을 가능성이 커진다.

```text
page 0  -> PTE 라인 A
page 8  -> PTE 라인 B
page 16 -> PTE 라인 C
page 24 -> PTE 라인 D
```

쉽게 말하면 데이터도 멀리 있어서 캐시에 없는데, 그 데이터가 실제 메모리 어디 있는지 알려주는 지도 조각도 매번 새로 찾아야 하는 상황이 된다.

### TLB와 PTE를 구분하면

| 개념 | 느낌 |
| --- | --- |
| PTE | 페이지 테이블에 저장된 원본 주소 변환 기록 |
| TLB | 최근 주소 변환 결과를 저장하는 빠른 캐시 |
| PTE cache line | PTE들을 메모리나 캐시에 가져올 때의 묶음 |

흐름은 대략 이렇다.

```text
가상 주소 접근
-> TLB에 변환 결과가 있는지 확인
-> 없으면 page table을 따라가며 PTE를 읽음
-> PTE를 보고 물리 주소 계산
-> 실제 데이터 접근
```

TLB에 있으면 빠르지만, 없으면 PTE를 읽으러 가야 한다. 이 실험의 stride 8 패턴은 데이터 접근 패턴뿐 아니라 주소 변환에 필요한 PTE 접근 패턴까지 나쁘게 만들었을 가능성이 있다.

## DRAM row와 bank까지 괴롭히기

마지막 단계는 실제 RAM 내부 구조까지 방해하려는 시도다.

DRAM은 그냥 큰 배열 하나가 아니라 여러 층으로 구성된다.

```text
channel
rank
chip
bank
row
column
```

여기서 중요한 것은 `bank`와 `row`다. DRAM bank 안에는 여러 row가 있고, 특정 row를 접근하려면 먼저 그 row를 열어 row buffer에 올려야 한다.

```text
row 열기
-> row buffer에 올림
-> column 읽기
```

이미 열린 row를 다시 읽으면 빠르다. 이것을 row-buffer hit라고 한다.

```text
bank 0 row 10 열림
-> 또 row 10 읽음
-> 빠름
```

반대로 같은 bank에서 다른 row를 읽으려면 기존 row를 닫고 새 row를 열어야 한다.

```text
bank 0 row 10 열림
-> bank 0 row 99 읽고 싶음
-> row 10 닫기
-> row 99 열기
-> 읽기
```

이것이 row-buffer conflict다. 같은 bank 안에서 서로 다른 row를 번갈아 접근하면, 거의 매번 precharge로 기존 row를 닫고 새 row를 activate해야 해서 느려진다.

느린 패턴을 만들려는 전략은 다음과 같다.

```text
같은 bank의 row A 읽기
같은 bank의 row B 읽기
같은 bank의 row A 읽기
같은 bank의 row B 읽기
```

그러면 매번 다음 흐름이 반복된다.

```text
기존 row 닫기
새 row 열기
읽기
```

반대로 이미 열린 row를 계속 접근하면 row-buffer hit가 되고, 여러 bank를 동시에 잘 쓰면 memory controller가 작업을 겹쳐 처리할 수 있다.

## DRAM 패턴을 정확히 만들기 어려운 이유

문제는 물리 주소가 DRAM의 channel, rank, bank, row, column으로 어떻게 매핑되는지가 깔끔하게 문서화되어 있지 않다는 점이다.

```text
물리 주소 0x12345678
-> channel 몇 번?
-> bank 몇 번?
-> row 몇 번?
```

이 매핑은 CPU, 메모리 타입, BIOS/firmware 설정, channel/rank 구성, address hashing에 따라 달라질 수 있다.

DRAMA나 Sudoku 같은 도구로 역추적할 수 있지만, 원문 실험 머신에서는 제대로 동작시키지 못했다. 그래서 DRAMA 논문과 로컬 실험을 바탕으로 bank group 4개, group당 bank 4개, `DRAM_ROW_SHIFT = 18` 같은 값을 사용해 근사했다.

DRAM bank/row 충돌까지 고려한 패턴은 약 `20.82억` 사이클이 걸렸다. stride 8의 약 `20.58억` 사이클보다 일관되게 조금 느렸지만, 차이는 크지 않았다.

완전한 row conflict를 만들기 어려웠던 이유는 다음과 같다.

- bank group hash, bank hash, row shift 추정이 정확하지 않을 수 있다.
- 8페이지 stride는 약 32KB 간격 접근이라 같은 DRAM row에 있기 어렵다.
- Intel의 bank hashing 때문에 실제로는 여러 bank를 동시에 쓰게 될 수 있다.

## 메모리 계층 전체를 괴롭히는 흐름

이 실험은 점점 아래 계층으로 내려가며 병목을 만든다.

| 단계 | 괴롭힌 대상 | 무슨 일이 생김 |
| --- | --- | --- |
| 선형 접근 깨기 | 캐시 라인 | 64B를 가져와도 4B만 씀 |
| 페이지 단위 접근 | 프리페처, TLB | 4KB 경계를 계속 넘음 |
| stride 8 | PTE 캐시 라인 | 주소 변환용 정보도 계속 새로 읽음 |
| DRAM row/bank | 실제 RAM 내부 | 같은 bank의 다른 row를 번갈아 열게 만듦 |

쉽게 말하면 다음 흐름이다.

```text
1단계: 데이터 가까운 것을 안 읽게 만들기
2단계: 페이지를 계속 바꾸게 만들기
3단계: 주소 변환 지도도 계속 새로 보게 만들기
4단계: RAM 내부에서도 row를 열고 닫게 만들기
```

백엔드 감각으로 비유하면, 데이터 하나를 찾을 때마다 인덱스 페이지도 새로 보고, 테이블 페이지도 새로 보고, 다음 row는 또 완전히 다른 곳에 있는 패턴과 비슷하다.

> [!summary]
> stride 8 패턴은 데이터 접근뿐 아니라 페이지 주소 변환에 필요한 PTE 접근까지 캐시 친화적이지 않게 만들어 더 느려질 수 있다. 이후 DRAM bank/row 충돌까지 노리면 실제 RAM 내부에서도 이미 열린 row를 재사용하지 못하고 계속 닫고 여는 비용이 생긴다. 다만 물리 주소가 DRAM 구조에 어떻게 매핑되는지는 플랫폼마다 달라서 완전히 통제하기 어렵고, 그래서 DRAM 최적화와 악화는 근사적으로만 실험할 수 있다.

## 이 안티패턴을 피하는 방법

핵심은 간단하다.

*==데이터를 가까운 것끼리, 예측 가능한 순서로, 한 번 가져온 캐시 라인을 최대한 써먹게 읽어야 한다.==*

### 1. 배열과 리스트는 가능하면 순서대로 돈다

좋은 패턴은 앞에서부터 차례대로 읽는 것이다.

```java
for (int i = 0; i < arr.length; i++) {
    sum += arr[i];
}
```

나쁜 패턴은 인덱스가 계속 튀는 것이다.

```java
for (int i : randomIndexes) {
    sum += arr[i];
}
```

순차 접근은 CPU 프리페처가 맞히기 쉽다. 반면 랜덤 인덱스 접근은 캐시 라인과 페이지 지역성을 깨기 쉽다.

### 2. 랜덤 조회가 필요하면 모아서 정렬한다

ID 목록이 이런 식으로 들어왔다고 해보자.

```text
900, 3, 12000, 8, 901, 4
```

가능하면 가까운 값끼리 접근하도록 정렬해서 처리하는 편이 낫다.

```text
3, 4, 8, 900, 901, 12000
```

백엔드에서도 비슷하다. 랜덤한 ID를 하나씩 조회하는 것보다, 가능한 경우 한 번에 묶어서 DB가 더 나은 접근 계획을 잡게 하는 편이 좋다.

```sql
SELECT *
FROM users
WHERE id IN (3, 4, 8, 900, 901, 12000);
```

### 3. 객체를 너무 흩뿌리지 않는다

`List<User>`는 리스트 자체는 순서가 있어도, 실제 `User` 객체들이 힙 여기저기에 흩어져 있을 수 있다. 반면 `int[]`, `long[]` 같은 배열은 값이 연속적으로 배치되기 쉽다.

성능이 중요한 코드에서는 객체 묶음보다 필요한 필드만 연속 배열로 두는 구조가 유리할 때가 있다.

```text
User 객체 100만 개
-> id, age, score가 객체마다 흩어질 수 있음

id[] age[] score[]
-> 필요한 값만 연속으로 읽기 쉬움
```

이런 관점을 data-oriented design이라고 부르기도 한다.

### 4. 큰 데이터는 페이지 단위로 튀지 않게 처리한다

안 좋은 느낌은 이런 식이다.

```text
page 1에서 하나 읽기
page 5000에서 하나 읽기
page 2에서 하나 읽기
page 9000에서 하나 읽기
```

좋은 느낌은 한 덩어리를 충분히 쓰고 다음 덩어리로 넘어가는 것이다.

```text
page 1을 쭉 읽기
page 2를 쭉 읽기
page 3을 쭉 읽기
```

파일 처리, 로그 처리, 대량 배치 작업에서도 이 감각이 중요하다. 여기저기 seek 하면서 읽는 것보다, 청크 단위로 순차 처리하는 편이 보통 더 좋다.

### 5. 포인터 추적 구조를 조심한다

`LinkedList` 같은 구조는 논리적으로는 순서가 있어도 실제 메모리에서는 노드가 흩어져 있을 수 있다.

```text
node1 -> node2 -> node3 -> node4
```

반면 배열은 실제 메모리에서도 가까이 붙어 있는 경우가 많다.

```text
arr[0], arr[1], arr[2], arr[3]
```

그래서 단순 순회가 많으면 `LinkedList`보다 `ArrayList`가 유리한 경우가 많다. Java에서 대부분의 일반적인 상황에 `ArrayList`를 먼저 고려하는 이유 중 하나도 캐시 지역성이다.

## 백엔드 관점에서 보면

CPU 캐시 이야기는 DB나 서버 코드에서도 비슷한 감각으로 이어진다.

| 피하고 싶은 것 | 대신 할 것 |
| --- | --- |
| 랜덤 인덱스 접근 | 정렬해서 가까운 순서로 접근 |
| 객체가 힙에 흩어진 구조 | 배열이나 연속 구조 활용 |
| 하나씩 DB 조회 | batch, `IN`, join으로 묶기 |
| `LinkedList` 순회 | `ArrayList` 우선 고려 |
| 필요한 필드 몇 개 때문에 큰 객체 전체 읽기 | projection으로 필요한 필드만 읽기 |
| 페이지를 계속 넘나드는 처리 | 청크 단위로 순차 처리 |

DB에서는 특히 이런 패턴이 중요하다.

- N+1 쿼리 줄이기
- 랜덤 ID 반복 조회 줄이기
- range scan이 가능하게 조건 잡기
- batch size 조절하기
- covering index로 테이블 랜덤 접근 줄이기
- 필요한 컬럼만 projection 하기

단순히 Big-O만 보는 게 아니라, 실제 데이터가 메모리와 디스크에서 어떤 순서로 읽히는지도 성능에 영향을 준다.

## 백엔드 개발자가 외우면 좋은 정도

이 주제의 저수준 숫자를 전부 외울 필요는 없다. 백엔드 개발자라면 다음 감각을 기억하면 충분하다.

| 개념 | 외울 정도 |
| --- | --- |
| 캐시 라인 | 보통 `64B` 단위로 메모리를 가져온다 |
| 페이지 | 보통 `4KB` 단위로 가상 메모리를 관리한다 |
| 순차 접근 | CPU가 제일 좋아한다 |
| 랜덤 접근 | 캐시, 프리페처, TLB 효율이 떨어진다 |
| 재사용 거리 | 다시 쓰기 전까지 너무 많이 돌아다니면 캐시에서 밀린다 |
| 같은 set 충돌 | 특정 간격으로 뛰면 캐시 일부 칸만 계속 때릴 수 있다 |
| 실무 결론 | 가까운 데이터끼리 묶고, batch/range/순차 처리하자 |

외울 문장으로 줄이면 다음과 같다.

> CPU는 메모리를 64B 캐시 라인 단위로 가져오고, OS는 보통 4KB 페이지 단위로 메모리를 관리한다. 그래서 데이터를 순차적으로 읽으면 캐시와 프리페처가 잘 동작하지만, 랜덤 접근이나 페이지를 계속 넘나드는 접근은 캐시 재사용과 주소 변환 효율을 떨어뜨린다.

백엔드 실무에서는 다음 연결을 떠올리면 된다.

### 1. N+1 쿼리는 랜덤 접근의 DB 버전처럼 생각하기

N+1 쿼리는 필요한 데이터를 한 번에 가져오지 않고, 작은 조회를 반복해서 왔다 갔다 하는 패턴이다.

```text
유저 1명 조회
-> 주문 따로 조회
-> 댓글 따로 조회
-> 프로필 따로 조회
```

이런 방식은 DB 버퍼풀, 네트워크, 쿼리 실행 비용을 낭비하기 쉽다.

좋은 방향은 필요한 데이터를 묶어서 가져오는 것이다.

```text
batch query
join
fetch join
IN query
```

### 2. Range scan은 순차 접근에 가깝다

인덱스에서 가까운 범위를 쭉 읽을 수 있으면 순차 접근에 가까운 이점을 얻을 수 있다.

```sql
SELECT id, status, created_at
FROM orders
WHERE created_at >= ?
AND created_at < ?
ORDER BY created_at;
```

반대로 랜덤한 ID를 하나씩 찍는 조회는 접근 위치가 계속 튀기 쉽다.

### 3. 순회가 많으면 `ArrayList`를 먼저 고려하기

`LinkedList`는 논리적으로는 순서가 있어도, 실제 노드가 메모리 여기저기에 흩어져 있을 수 있다. 반면 `ArrayList`는 내부 배열 기반이라 가까운 데이터를 연속으로 읽기 좋다.

> 순회가 많으면 `LinkedList`보다 `ArrayList`가 캐시 친화적이다.

### 4. 필요한 컬럼만 읽기

큰 객체 전체를 가져오면 DB I/O, 메모리 사용량, 네트워크 응답 크기, JSON 직렬화 비용이 모두 늘 수 있다.

```sql
SELECT *
FROM orders;
```

보다 필요한 컬럼만 읽는 것이 나을 때가 많다.

```sql
SELECT id, status, created_at
FROM orders;
```

### 5. 배치와 청크 단위로 처리하기

대량 데이터를 처리할 때는 하나씩 랜덤하게 찍기보다 일정한 순서로 청크 단위 처리하는 편이 안정적이다.

```sql
SELECT id, status
FROM orders
WHERE id > ?
ORDER BY id
LIMIT 1000;
```

이런 식의 커서 기반 순차 처리는 메모리와 DB 접근 패턴을 예측 가능하게 만든다.

## 숫자보다 중요한 감각

숫자는 이 정도만 기억해도 된다.

```text
캐시 라인: 보통 64B
페이지: 보통 4KB
```

그리고 진짜 중요한 감각은 다음이다.

```text
64B 안에 들어온 데이터는 최대한 같이 써먹자.
4KB 페이지를 계속 넘나들면 부담이 커질 수 있다.
다시 쓸 데이터는 너무 오래 돌아다니기 전에 쓰자.
```

## 면접에서 말하면

> [!tip]
> CPU 캐시 관점에서 안 좋은 데이터 접근 패턴은 메모리 지역성을 깨는 패턴입니다. 이를 피하려면 가능한 한 연속된 메모리를 순차적으로 접근하고, 랜덤 접근이 필요할 때는 접근 대상을 모아서 정렬하거나 배치 처리해 캐시 라인과 페이지 지역성을 살리는 방식으로 설계합니다. 백엔드에서는 N+1 쿼리, 랜덤 ID 반복 조회, 과도한 객체 탐색을 줄이고 batch query, range scan, projection, `ArrayList` 같은 연속 자료구조를 우선 고려할 수 있습니다.

> [!tip]
> 저수준 성능에서는 메모리 접근 패턴이 중요합니다. CPU는 보통 64바이트 캐시 라인 단위로 데이터를 가져오고, OS는 보통 4KB 페이지 단위로 메모리를 관리하기 때문에, 순차 접근은 캐시와 프리페처에 유리하고 랜덤 접근은 불리합니다. 백엔드에서는 이 감각을 N+1 쿼리 줄이기, batch 처리, range scan 활용, 필요한 컬럼만 조회, `ArrayList` 같은 연속 자료구조 선택으로 연결해서 볼 수 있습니다.

## 나중에 다시 볼 포인트

- 같은 덧셈 루프라도 데이터 접근 순서에 따라 실행 시간이 크게 달라질 수 있다.
- CPU는 순차 접근에 강하다.
- 랜덤 접근은 프리페처, 캐시 라인 재사용, 페이지 지역성을 깨기 쉽다.
- 캐시 라인 간격 접근은 한 번 가져온 64바이트 중 4바이트만 쓰는 낭비를 만든다.
- 페이지 간격 접근은 프리페처가 4KB 페이지 경계를 넘기 어렵다는 점까지 건드려 더 느려질 수 있다.
- 주소 간격이 4KB처럼 캐시 set 주기와 맞으면 같은 set에 몰려 캐시 충돌이 생길 수 있다.
- stride 8처럼 PTE 캐시 라인까지 계속 바꾸는 패턴은 주소 변환 비용까지 악화시킬 수 있다.
- DRAM에서는 같은 bank의 다른 row를 번갈아 접근하면 row-buffer conflict로 느려질 수 있다.
- 물리 주소와 DRAM channel, bank, row 매핑은 플랫폼 의존적이라 완전히 통제하기 어렵다.
- 랜덤보다 더 나쁜 패턴은 캐시 set 충돌, TLB/PTE 캐시 지역성 붕괴, DRAM row/bank 충돌까지 노린다.
- 실무에서는 연속 접근, batch 처리, range scan, projection, 청크 단위 처리를 먼저 떠올리면 된다.
- 백엔드 개발자는 `64B 캐시 라인`, `4KB 페이지`, `순차 접근`, `랜덤 접근`, `재사용 거리` 감각만 먼저 잡아도 충분하다.
