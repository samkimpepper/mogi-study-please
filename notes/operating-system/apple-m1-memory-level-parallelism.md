# Apple M1의 메모리 병렬성 이해하기

관련 개념 노트: [캐시 라인과 페이지는 뭐가 다른가](cache-line-vs-page.md)

참고 링크:

- [Memory access on the Apple M1 processor - Daniel Lemire](https://lemire.me/blog/2021/01/06/memory-access-on-the-apple-m1-processor/)
- [testingmlp - GitHub](https://github.com/lemire/testingmlp)
- [More M1 fun: Hardware Information - Jim Cownie](https://cpufun.substack.com/p/more-m1-fun-hardware-information)

> [!summary]
> M1 싱글코어가 빠르다는 말은 메모리 접근 하나하나가 무조건 엄청 빠르다는 뜻만은 아니다.
>
> 이 문맥에서는 독립적인 메모리 요청 여러 개를 동시에 걸어두고, 메모리 대기 시간을 겹쳐 숨길 수 있다는 뜻에 가깝다. 이것을 memory-level parallelism, 줄여서 MLP라고 한다.

## 처음 헷갈렸던 질문

> [!question]
> 4096바이트 단위로 읽어오는 것이 캐시 라인 얘기인가?
>
> M1이 28 lane으로 메모리를 처리한다는 말은 무슨 뜻인가?
>
> 메모리 접근 7개면 1개보다 7배 느려져야 하는 것 아닌가?

## 페이지와 캐시 라인은 다르다

먼저 4096바이트 단위라는 감각은 대부분 페이지 이야기다. 캐시 라인과 페이지는 목적이 다르다.

| 개념 | 느낌 | 보통 크기 |
| --- | --- | --- |
| 페이지 | OS/MMU가 가상 메모리를 관리하는 단위 | x86에서 흔히 4KB, M1/macOS에서는 16KB로 보고됨 |
| 캐시 라인 | CPU 캐시에 데이터를 가져오는 최소 블록 | x86에서 흔히 64B, M1에서는 `sysctl` 기준 128B로 보고됨 |

Jim Cownie가 M1에서 `sysctl`로 확인한 값은 다음과 같이 보고된다.

```text
hw.pagesize: 16384
hw.cachelinesize: 128
```

즉 "4096바이트씩 읽어온다"는 표현은 캐시 라인보다는 페이지나 TLB 쪽 이해에 더 가깝다. CPU가 `arr[100]`을 읽을 때는 1바이트나 4바이트만 딱 가져오는 것이 아니라, 그 주소가 들어 있는 캐시 라인 한 덩어리를 가져온다.

```text
arr[100] 읽기
-> 같은 캐시 라인 안의 arr[101], arr[102] 근처도 같이 들어올 수 있음
```

그래서 `arr[100]` 다음에 `arr[101]`, `arr[102]`를 읽으면 이미 캐시 라인 안에 있어서 거의 공짜처럼 느껴질 수 있다.

## 순진한 메모리 접근 모델

처음에는 이런 모델이 그럴듯하다.

> x를 읽으면 x+1 근처는 공짜다. 그러니 서로 다른 랜덤 캐시 라인 수만 세면 성능을 대충 예측할 수 있지 않을까?

예를 들어:

```text
arr[random1]     = 비쌈
arr[random1 + 1] = 같은 캐시 라인이면 거의 공짜

arr[random2]     = 비쌈
arr[random2 + 1] = 같은 캐시 라인이면 거의 공짜
```

이 모델은 많은 상황에서 꽤 쓸모 있다. 하지만 실제 현대 CPU에서는 이 모델이 깨질 수 있다.

## 왜 깨지는가: CPU는 하나 읽고 멍때리지 않는다

순진한 모델은 메모리 접근을 이렇게 생각한다.

```text
메모리 접근 1개 = 100ns
메모리 접근 7개 = 700ns
```

하지만 현대 CPU는 독립적인 메모리 접근을 여러 개 동시에 걸어둘 수 있다.

```text
1번 주소 읽어와!
2번 주소 읽어와!
3번 주소 읽어와!
...
7번 주소 읽어와!

다 같이 오면 계산하자.
```

이것이 memory-level parallelism, MLP다.

백엔드식으로 보면 동기 HTTP 요청과 비동기 HTTP 요청 차이와 비슷하다.

```text
동기 HTTP 요청 7개:
요청1 끝남 -> 요청2 끝남 -> 요청3 끝남 ...
총 7배 가까이 걸림

비동기 HTTP 요청 7개:
요청1~7 동시에 날림 -> 제일 늦은 요청을 기다림
7배까지는 안 걸림
```

CPU 메모리 접근도 독립적인 요청이라면 각각의 latency를 전부 더하는 대신, 여러 요청의 대기 시간을 겹쳐 숨길 수 있다.

그래서 질문은 이렇게 바뀐다.

> 7개의 메모리 접근을 동시에 한다면 1개에 접근하는 것보다는 느리겠지만, 7배가 느릴까?

답은 아니다. 독립적인 접근이면 7배보다 훨씬 덜 느릴 수 있다.

## 28 lane은 무슨 뜻인가

여기서 lane은 코어 안에 실제 물리 도로 28개가 있다는 뜻이 아니다. 벤치마크에서 만든 독립적인 pointer chasing 루틴의 개수라고 보면 된다.

포인터 추적이 하나만 있으면 다음처럼 된다.

```c
p = p->next;
p = p->next;
p = p->next;
```

이 코드는 병렬화가 어렵다. 다음 주소를 알려면 이전 노드를 먼저 읽어야 하기 때문이다.

```text
p->next를 읽어야 다음 p 주소를 알 수 있음
-> 이전 메모리 접근이 끝날 때까지 다음 접근을 시작하기 어려움
-> 메모리 latency를 그대로 맞음
```

반대로 독립적인 리스트를 여러 개 두면 다르다.

```c
p1 = p1->next;
p2 = p2->next;
p3 = p3->next;
p4 = p4->next;
```

`p1`, `p2`, `p3`, `p4`는 서로 독립이다. CPU는 여러 메모리 요청을 동시에 걸 수 있다.

Lemire의 `testingmlp` 벤치마크도 여러 개의 독립 pointer chasing lane을 만들고, lane 수를 늘려가며 성능 증가가 멈추는 지점을 본다. 그 지점으로 프로세서의 memory-level parallelism을 추정한다.

```text
lane 1개  : 메모리 요청 1개씩 기다림
lane 2개  : 2개를 겹쳐 기다림
lane 4개  : 4개를 겹쳐 기다림
...
lane 28개쯤: M1 코어가 꽤 잘 버팀
그 이상   : 더 늘려도 효율이 떨어짐
```

따라서 "M1의 싱글코어는 28단계 이상으로 메모리를 병렬 처리한다"는 말은 다음처럼 이해하는 편이 정확하다.

> M1의 한 성능 코어가 독립적인 캐시 미스나 메모리 요청을 상당히 많이 outstanding 상태로 들고 있을 수 있는 것으로 보인다. 벤치마크상 약 28개 안팎까지 효율이 유지되는 듯하다.

이것은 공식 스펙이라기보다 벤치마크 기반 추정치에 가깝다.

## 왜 x+1은 공짜 모델이 실패하는가

순진한 모델은 근처 주소가 같은 캐시 라인 안에 있으면 거의 공짜라고 본다.

```text
arr[random1]     = 비쌈
arr[random1 + 1] = 거의 공짜
arr[random2]     = 비쌈
arr[random2 + 1] = 거의 공짜
```

그래서 서로 다른 랜덤 캐시 라인 수만 세면 성능을 예측할 수 있을 것처럼 보인다.

하지만 M1 같은 CPU는 독립적인 랜덤 접근을 동시에 여러 개 날릴 수 있다.

```text
arr[random1]
arr[random2]
arr[random3]
arr[random4]
...
```

그러면 비용이 단순히 다음처럼 늘지 않는다.

```text
랜덤 접근 수 * 메모리 latency
```

대충 이런 감각에 가까워진다.

```text
랜덤 접근 수 / 동시에 처리 가능한 개수 * 메모리 latency
```

물론 실제 CPU에서는 캐시, TLB, 프리페처, reorder buffer, load queue, 메모리 bandwidth가 같이 영향을 주기 때문에 훨씬 복잡하다.

그래도 감각적으로는 다음처럼 볼 수 있다.

```text
독립 랜덤 접근 1개  -> 1번 기다림
독립 랜덤 접근 7개  -> 7번 따로 기다리는 게 아니라 1묶음처럼 기다릴 수 있음
독립 랜덤 접근 28개 -> 여전히 꽤 잘 겹쳐질 수 있음
독립 랜덤 접근 50개 -> 내부 큐, 대역폭, ROB 등이 막혀 효율이 떨어질 수 있음
```

그래서 "7개 접근이면 7배 느릴 것"이라는 예측이 깨진다.

## 왜 `N / 동시처리개수 * latency`처럼 보이는가

이 식이 나오는 이유는 한 번의 메모리 접근 latency를 여러 접근이 같이 맞을 수 있기 때문이다.

백엔드식으로 보면 순차 `await`와 `Promise.all` 차이에 가깝다.

```js
// 느린 모델: 하나 끝나야 다음 요청
await fetch(a);
await fetch(b);
await fetch(c);

// 빠른 모델: 요청 여러 개 먼저 던져놓고 같이 기다림
await Promise.all([fetch(a), fetch(b), fetch(c)]);
```

CPU 메모리 접근도 서로 의존성이 없는 load라면 여러 개를 먼저 던져놓을 수 있다.

메모리 접근 1번이 100ns 걸린다고 해보자.

### 서로 의존적인 경우

```c
p = p->next;
p = p->next;
p = p->next;
p = p->next;
```

이 코드는 다음 주소를 알려면 이전 메모리 접근 결과가 필요하다.

```text
1번째 load 끝남 -> 다음 주소 알게 됨
2번째 load 끝남 -> 다음 주소 알게 됨
3번째 load 끝남 -> 다음 주소 알게 됨
...
```

그래서 대략 다음처럼 된다.

```text
4번 접근 * 100ns = 400ns
```

이것이 linked list, tree 탐색, pointer chasing이 느린 대표적인 이유다.

### 서로 독립적인 경우

```c
a = arr[random1];
b = arr[random2];
c = arr[random3];
d = arr[random4];
```

여기서는 `a`를 읽어야 `b` 주소를 알 수 있는 것이 아니다. `random1`, `random2`, `random3`, `random4` 주소가 이미 준비되어 있으면 CPU는 여러 load를 거의 동시에 걸 수 있다.

```text
arr[random1] 읽어와
arr[random2] 읽어와
arr[random3] 읽어와
arr[random4] 읽어와
```

그래서 결과가 항상 다음처럼 되지는 않는다.

```text
4번 * 100ns = 400ns
```

오히려 다음에 가까워질 수 있다.

```text
4개를 한 묶음으로 요청 -> 대략 100ns + 약간의 오버헤드
```

물론 완전히 공짜는 아니다. 내부 큐, 메모리 대역폭, 캐시 상태, TLB, 명령어 수 때문에 추가 비용은 생긴다.

## 동시 처리 개수로 나눠지는 직관

CPU가 메모리 요청을 동시에 4개까지 잘 걸 수 있다고 단순화해보자.

메모리 접근이 1개면 다음과 같다.

```text
[요청1] -> 100ns
```

메모리 접근이 4개면 다음처럼 묶일 수 있다.

```text
[요청1 요청2 요청3 요청4] -> 100ns
```

메모리 접근이 8개면 4개씩 두 묶음이 된다.

```text
[요청1 요청2 요청3 요청4] -> 100ns
[요청5 요청6 요청7 요청8] -> 100ns
```

그래서 대충 이런 식으로 보인다.

```text
8개 접근 / 4개 동시처리 * 100ns = 200ns
```

조금 더 정확한 감각은 다음에 가깝다.

```text
ceil(접근 수 / 동시에 outstanding 가능한 수) * latency
```

하지만 이것도 아주 거친 이해용 모델이다. 실제 성능 공식은 아니다.

M1에서 말하는 `28 lane`은 이 동시에 outstanding 가능한 메모리 요청 수가 꽤 크게 관측되었다는 뜻이다. Lemire의 M1 실험에서는 단일 코어가 약 28 levels of memory parallelism, possibly more로 보인다고 설명한다.

## CPU는 어떻게 동시에 날리는가

코드에는 순서가 있다.

```c
a = arr[random1];
b = arr[random2];
c = arr[random3];
d = arr[random4];
```

하지만 현대 CPU는 내부에서 다음을 확인한다.

```text
1. 이 명령어들 사이에 데이터 의존성이 있는가?
2. 없네?
3. 그러면 먼저 실행 가능한 load부터 여러 개 실행하자.
4. 결과는 프로그램이 보기에 순서가 맞게 정리하자.
```

이것이 out-of-order execution이다.

CPU 내부에서는 대략 이런 일이 일어난다.

```text
명령어를 미리 많이 가져옴
-> 의존성 분석함
-> 실행 가능한 load를 먼저 보냄
-> cache miss가 나도 바로 멈추지 않음
-> 다른 독립 load도 계속 보냄
-> 결과가 오면 원래 프로그램 순서처럼 정리해서 완료 처리
```

중요한 점은 CPU가 코드 순서를 무조건 한 줄씩 멍하게 실행하지 않는다는 것이다. 결과만 같게 유지할 수 있으면 내부적으로 순서를 바꿔 실행할 수 있다.

그래서 독립적인 메모리 접근은 병렬로 겹칠 수 있다.

## M1만의 개념은 아니다

MLP는 M1만의 개념이 아니다. 현대 고성능 CPU들은 대체로 다음과 같은 구조를 갖는다.

```text
out-of-order execution
non-blocking cache
load/store queue
miss buffer / fill buffer / MSHR 비슷한 구조
prefetcher
```

다만 M1은 이 능력이 꽤 강하게 관측된 편이다. Lemire의 `testingmlp` 벤치마크는 여러 개의 독립 pointer chasing lane을 늘려가며 성능 향상이 멈추는 지점으로 MLP를 추정한다.

정리하면 다음과 같다.

```text
개념: 현대 CPU 공통
M1: 그 폭이 큰 편으로 관측됨
28: Apple 공식 스펙이라기보다 벤치마크 기반 추정
```

## 독립이면 가능하고 의존이면 어려운 이유

이 차이가 MLP의 핵심이다.

독립 접근은 주소가 이미 준비되어 있다.

```c
x = arr[i];
y = arr[j];
z = arr[k];
```

CPU는 세 요청을 같이 날릴 수 있다.

```text
arr[i] 요청 ┐
arr[j] 요청 ├ 동시에 기다림
arr[k] 요청 ┘
```

의존 접근은 다르다.

```c
p = p->next;
q = p->next;
r = q->next;
```

`p`를 읽어야 `q` 주소를 알고, `q`를 읽어야 `r` 주소를 안다.

```text
p 요청 -> 결과 옴 -> q 주소 알게 됨
q 요청 -> 결과 옴 -> r 주소 알게 됨
r 요청
```

CPU가 아무리 좋아도 아직 모르는 미래 주소를 미리 요청할 수는 없다. 그래서 linked list 순회는 MLP가 잘 나오기 어렵다.

## 이 공식이 항상 맞지는 않다

`랜덤 접근 수 / 동시처리개수 * latency`는 이해용 모델이지 실제 예측 공식이 아니다.

처음에는 latency가 병목이다.

```text
메모리 요청 1개씩 기다리느라 느림
```

lane을 늘리면 latency가 숨겨진다.

```text
여러 요청을 같이 기다려서 빨라짐
```

하지만 너무 많이 늘리면 다른 병목이 생긴다.

```text
load queue 꽉 참
miss buffer 꽉 참
ROB 꽉 참
메모리 대역폭 부족
TLB miss 증가
캐시 라인 낭비
프리페처 실패
명령어 수 증가
```

그래서 어느 순간부터는 lane을 늘려도 성능이 좋아지지 않거나 오히려 떨어질 수 있다.

## 주니어 백엔드 개발자가 기억할 정도

`28 lane`, `ROB`, `MSHR`, `load queue` 같은 이름을 전부 외울 필요는 없다.

다음 정도면 충분하다.

```text
1. 메모리 접근은 계산보다 비쌀 수 있다.
2. 배열처럼 연속된 데이터는 CPU가 잘 먹는다.
3. linked list, tree, 객체 포인터 따라가기, HashMap 랜덤 접근은 느릴 수 있다.
4. 독립적인 메모리 접근은 CPU가 어느 정도 겹쳐서 처리할 수 있다.
5. 그래서 "메모리 접근 10번 = 10배 느림"은 항상 틀릴 수 있다.
6. 성능은 감으로 확정하지 말고 JMH, profiler, flame graph로 봐야 한다.
```

실무 감각으로는 다음 연결이 중요하다.

```text
ArrayList 순회가 LinkedList 순회보다 보통 빠른 이유
HashMap이 빠르지만 메모리 locality는 별로일 수 있는 이유
객체를 너무 잘게 쪼개면 캐시 효율이 나빠지는 이유
DTO/엔티티/컬렉션 구조가 성능에 영향을 줄 수 있는 이유
```

진짜 한 문장으로 줄이면 다음과 같다.

> CPU는 메모리 요청을 어느 정도 async처럼 겹칠 수 있지만, 포인터 따라가기는 의존성이 있어서 잘 못 겹친다.

## 캐시 충돌 문제와 MLP 문제는 다르다

앞에서 공부한 같은 set 몰림 문제는 cache conflict miss나 cache associativity 문제다.

```text
주소 A, B, C, D가 우연히 같은 캐시 set에 매핑됨
그 set에 들어갈 수 있는 칸은 제한됨
-> 서로 밀어냄
-> 계속 cache miss
```

이 문제의 질문은 다음이다.

> 캐시에 잘 들어가느냐, 자꾸 쫓겨나느냐?

M1의 28 lane 이야기는 조금 다른 질문이다.

> 캐시 미스가 여러 개 났을 때 CPU가 하나씩 멈춰 기다리느냐, 아니면 여러 미스를 동시에 걸어두고 기다림을 겹치느냐?

둘을 나란히 보면 다음과 같다.

| 주제 | 핵심 질문 |
| --- | --- |
| 캐시 충돌 | 캐시에 들어간 데이터가 자꾸 쫓겨나는가? |
| MLP | 캐시 미스가 났을 때 기다림을 얼마나 겹칠 수 있는가? |

둘 다 성능에 중요하지만 포인트가 다르다.

## 백엔드 개발자 감각으로 보면

자바나 코틀린 서버 코드에서 이런 패턴은 CPU에게 불리하다.

```java
Node cur = head;
while (cur != null) {
    sum += cur.value;
    cur = cur.next;
}
```

`cur.next` 주소는 `cur`를 읽어야 알 수 있다.

```text
다음 주소를 미리 모름
-> CPU가 다음 메모리 요청을 미리 못 날림
-> 메모리 latency를 그대로 맞음
```

반면 배열 순회는 훨씬 낫다.

```java
for (int i = 0; i < arr.length; i++) {
    sum += arr[i];
}
```

연속 메모리라 캐시 라인과 프리페처가 잘 동작한다.

그리고 독립적인 접근이 여러 개 있으면 CPU가 메모리 요청을 겹칠 여지가 생긴다.

```java
int a = arr[idx1];
int b = arr[idx2];
int c = arr[idx3];
int d = arr[idx4];
```

다만 Java 코드에서 실제로 얼마나 잘 겹치는지는 JIT, 객체 배치, 분기, GC, escape analysis, CPU 종류에 따라 달라진다. 그래서 저수준 성능은 감으로 단정하기보다 JMH 같은 도구로 측정하는 것이 맞다.

## 면접에서 말하면

> [!tip]
> Apple M1의 메모리 접근 성능을 볼 때 중요한 점은 단일 메모리 접근 latency만이 아니라 memory-level parallelism입니다. 독립적인 캐시 미스가 여러 개 있으면 CPU가 요청을 동시에 outstanding 상태로 두고 대기 시간을 겹쳐 숨길 수 있습니다. 그래서 메모리 접근 7개가 항상 1개보다 7배 느린 것은 아닙니다. 반대로 포인터 추적처럼 다음 주소가 이전 로드 결과에 의존하면 MLP를 활용하기 어렵고 메모리 latency를 그대로 맞기 쉽습니다.

## 나중에 다시 볼 포인트

- 4096바이트 감각은 보통 캐시 라인보다 페이지 쪽에 가깝다.
- M1/macOS에서는 `hw.pagesize: 16384`, `hw.cachelinesize: 128`로 보고된 사례가 있다.
- 캐시 라인은 CPU가 데이터를 가져오는 단위이고, 페이지는 OS/MMU가 주소를 관리하는 단위다.
- MLP는 독립적인 메모리 요청 여러 개의 대기 시간을 겹쳐 숨기는 능력이다.
- `랜덤 접근 수 / 동시처리개수 * latency`는 CPU가 독립 load를 한꺼번에 걸어둘 수 있다는 점에서 나온 이해용 모델이다.
- `28 lane`은 공식 스펙이 아니라 독립 pointer chasing 벤치마크 기반 추정치로 이해해야 한다.
- 캐시 충돌은 "캐시에 남느냐"의 문제이고, MLP는 "캐시 미스를 얼마나 동시에 기다릴 수 있느냐"의 문제다.
- MLP는 M1만의 개념이 아니라 현대 고성능 CPU의 일반적인 능력이고, M1에서 크게 관측된 편이다.
- 백엔드 코드에서는 pointer chasing 구조보다 배열/연속 자료구조가 CPU에 유리한 경우가 많다.
