# 캐시 라인과 페이지는 뭐가 다른가

> [!summary]
> 캐시 라인과 페이지는 서로 모여서 만들어지는 관계가 아니다.
>
> 둘 다 메모리 접근과 관련 있지만, 캐시 라인은 CPU 캐시가 RAM 데이터를 빠르게 가져오기 위한 단위이고, 페이지는 OS와 MMU가 가상 메모리를 관리하기 위한 단위다.

## 처음 헷갈렸던 질문

> [!question]
> MMU가 모여서 캐시가 되는 걸까?
>
> 캐시가 모여서 페이지가 되는 걸까?
>
> 페이지가 모여서 캐시가 되는 걸까?

결론부터 말하면 전부 아니다.

```text
MMU가 모여서 캐시가 됨 ❌
캐시가 모여서 페이지가 됨 ❌
페이지가 모여서 캐시가 됨 ❌
```

캐시 라인과 페이지는 같은 메모리 주소를 바라보지만, 목적이 완전히 다르다.

## 캐시 라인은 CPU 속도 문제

캐시 라인은 CPU 캐시가 RAM 데이터를 가져올 때 쓰는 단위다.

```text
캐시 라인 = 보통 64B
```

CPU가 `int` 하나, 즉 4바이트만 읽고 싶어도 RAM에서 딱 4바이트만 가져오는 것이 아니다. 보통 주변 데이터까지 64바이트 단위로 묶어서 가져온다.

```text
data[0] 읽기
-> data[0] ~ data[15] 정도가 같이 캐시에 들어옴
```

`uint32_t`나 Java의 `int`처럼 4바이트 정수라면, 64바이트 캐시 라인 하나에 정수 16개가 들어간다.

캐시 라인의 목적은 단순하다.

> 어차피 근처 데이터도 곧 읽을 가능성이 높으니, 미리 같이 가져와서 빠르게 하자.

즉 캐시 라인은 성능 최적화 단위다.

## 페이지는 OS와 MMU의 주소 관리 문제

페이지는 운영체제와 MMU가 가상 메모리를 관리하는 단위다.

```text
페이지 = 보통 4KB
```

프로그램은 실제 RAM 주소를 직접 쓰는 것이 아니라 가상 주소를 쓴다. OS와 MMU는 이 가상 주소를 실제 물리 주소와 연결한다.

```text
가상 주소 0x1000
-> MMU와 페이지 테이블
-> 물리 주소 0xABC000
```

이때 "주소를 어느 크기로 잘라서 관리할 것인가"의 기본 단위가 페이지다.

페이지의 목적은 다음과 같다.

- 프로세스마다 독립된 가상 주소 공간을 제공한다.
- 가상 주소와 물리 주소를 연결한다.
- 페이지 단위로 읽기, 쓰기, 실행 권한을 관리한다.
- 필요한 경우 디스크와 RAM 사이에서 메모리를 관리한다.

즉 페이지는 메모리 관리, 보안, 주소 변환 단위다.

## 둘을 나란히 비교하면

| 구분 | 캐시 라인 | 페이지 |
| --- | --- | --- |
| 보통 크기 | 64B | 4KB |
| 담당 | CPU 캐시 | OS와 MMU |
| 목적 | RAM 접근을 빠르게 하기 | 가상 메모리 관리하기 |
| 핵심 이유 | 근처 데이터 재사용 | 주소 변환, 권한, 격리 |
| 개발자가 느끼는 문제 | 캐시 미스, 지역성 문제 | page fault, TLB miss, 권한 오류 |
| 예시 | `data[0]`을 읽으면 주변 값도 같이 캐시에 들어옴 | 프로세스마다 같은 가상 주소도 다른 물리 메모리로 갈 수 있음 |

## 관계가 아예 없지는 않다

캐시 라인과 페이지는 구성 목적은 다르지만, 크기상으로는 겹쳐 볼 수 있다.

보통 페이지 하나는 4KB이고 캐시 라인 하나는 64B다.

```text
1 page = 4KB = 4096B
1 cache line = 64B

4096 / 64 = 64개 캐시 라인
```

그래서 페이지 하나 안에는 캐시 라인 64개가 들어갈 수 있다.

```text
페이지 1개, 4KB
┌──────────────────────────────┐
│ cache line 0  64B            │
│ cache line 1  64B            │
│ cache line 2  64B            │
│ ...                          │
│ cache line 63 64B            │
└──────────────────────────────┘
```

하지만 이것은 "캐시 라인이 모여서 페이지가 된다"는 뜻이 아니다. 페이지는 OS/MMU가 정한 주소 관리 단위이고, 캐시 라인은 CPU 캐시가 정한 데이터 이동 단위다.

> 같은 메모리를 두고, OS/MMU는 페이지 단위로 주소를 관리하고, CPU 캐시는 캐시 라인 단위로 데이터를 가져온다고 보면 된다.

## MMU는 어디에 끼는가

MMU는 캐시가 아니다. MMU는 주소 번역기다.

```text
가상 주소
-> MMU
-> 물리 주소
```

MMU는 페이지 테이블을 보고 "이 가상 페이지는 실제 RAM의 어느 물리 페이지로 가야 하는지"를 바꿔준다.

그리고 이 주소 변환 결과를 빠르게 재사용하려고 TLB라는 캐시 비슷한 구조가 있다.

```text
MMU = 주소 번역 장치
TLB = 주소 번역 결과 캐시
CPU cache = 실제 데이터 캐시
```

여기서 헷갈리기 쉬운 점은 TLB도 넓은 의미에서는 캐시라는 점이다. 하지만 일반적으로 `L1`, `L2`, `L3 캐시`라고 말할 때는 실제 데이터나 명령어를 담는 CPU 캐시를 말하는 경우가 많다.

## 메모리 읽기 흐름에서 보면

CPU가 `data[i]`를 읽는다고 해보자.

```text
1. 프로그램은 가상 주소를 낸다.
2. MMU가 TLB를 확인한다.
3. TLB에 없으면 페이지 테이블과 PTE를 보고 물리 주소로 바꾼다.
4. 물리 주소가 정해지면 CPU 캐시를 확인한다.
5. L1/L2/L3 캐시에 데이터가 있으면 가져온다.
6. 없으면 RAM에서 캐시 라인 단위로 가져온다.
```

여기서 페이지와 캐시 라인이 등장하는 위치가 다르다.

```text
페이지: 2~3번 주소 변환에서 중요
캐시 라인: 4~6번 실제 데이터 가져올 때 중요
```

## 기억할 문장

> 페이지는 "이 가상 주소가 실제 어디 RAM에 연결되는가"를 관리하는 단위이고, 캐시 라인은 "RAM에서 데이터를 가져올 때 주변까지 얼마나 묶어 가져올까"의 단위다.

둘은 계층적으로 만들어지는 관계가 아니다. 같은 메모리 접근을 서로 다른 목적에서 나눠 보는 단위다.
