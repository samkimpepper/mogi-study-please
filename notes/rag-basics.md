# RAG 기초 정리

> [!summary]
> RAG는 검색 알고리즘이 아니라, 검색한 자료를 LLM에게 근거로 넣어 답변하게 만드는 전체 구조다.
>
> - 검색 방식은 `rg`, BM25, FTS, 벡터 검색 등으로 바뀔 수 있다.
> - 벡터DB는 RAG의 필수 조건이 아니라 선택지 중 하나다.
> - 코드 검색은 의미 검색보다 정확한 키워드, 경로, 함수명 매칭이 더 중요할 수 있다.

## RAG는 검색 알고리즘이 아니다

RAG는 Retrieval-Augmented Generation의 줄임말이다.

이름만 보면 어려운데, 백엔드 관점에서는 다음처럼 보면 된다.

```text
RAG = 검색 시스템 + LLM 답변 생성
```

즉, RAG 자체가 BM25, grep, 벡터 검색 같은 검색 알고리즘인 것은 아니다.

> [!info]
> 백엔드 관점에서 보면 RAG는 "검색 API + LLM 호출"로 이루어진 파이프라인에 가깝다.
>
> ```text
> 사용자 질문
> -> 관련 문서 검색
> -> 검색 결과를 프롬프트에 넣음
> -> LLM 답변 생성
> ```

검색 알고리즘은 관련 문서를 찾는 역할을 한다.

```text
검색 알고리즘:
문서 중에서 관련 있는 것을 찾는다.
```

RAG는 검색한 결과를 LLM에게 참고자료로 넘기고, LLM이 그 자료를 바탕으로 답변하게 만드는 전체 구조다.

```text
RAG:
문서를 검색한다.
-> 검색 결과를 LLM에게 넣는다.
-> LLM이 답변을 생성한다.
```

## RAG 안에 들어갈 수 있는 검색 방식

RAG에서 검색 단계는 꼭 벡터DB일 필요가 없다.

> [!question]
> 헷갈리기 쉬운 질문:
>
> "RAG면 무조건 벡터DB를 써야 하나?"
>
> 답은 아니다. 핵심은 검색 결과를 LLM 답변의 근거로 넣는 것이다.

다음과 같은 방식도 RAG의 검색 단계로 사용할 수 있다.

```text
- grep / ripgrep
- SQL LIKE 검색
- SQLite FTS5
- Postgres Full Text Search
- Elasticsearch / OpenSearch
- BM25
- 벡터 검색
- BM25 + 벡터 검색 하이브리드
```

핵심은 검색 방식이 무엇이냐가 아니라, 검색한 자료를 LLM 답변의 근거로 넣는다는 점이다.

## RAG와 벡터DB의 관계

RAG를 만든다고 해서 벡터DB를 꼭 써야 하는 것은 아니다.

> [!warning]
> 벡터DB부터 고르면 구조가 커질 수 있다.
>
> 처음에는 `rg`, SQLite FTS, Postgres FTS처럼 단순한 검색으로 시작하고, 의미 검색이 실제로 필요할 때 벡터 검색을 붙이는 편이 현실적이다.

```text
RAG = 검색 + LLM 답변 생성
벡터DB = 검색을 구현하는 선택지 중 하나
```

벡터DB는 의미 기반 검색이 필요할 때 유용하다.

예를 들어 사용자가 이렇게 질문했다고 하자.

```text
로그인이 자꾸 풀려
```

문서에는 이렇게 적혀 있을 수 있다.

```text
세션 만료 정책
```

단어는 다르지만 의미가 비슷하다. 이런 경우 벡터 검색이 도움이 될 수 있다.

반대로 코드나 기술 문서는 정확한 키워드가 중요한 경우가 많다.

```text
@Transactional
N+1
JWT
ConnectionTimeoutException
userRepository.findByEmail
```

이런 것은 의미보다 정확한 토큰 매칭이 중요해서 BM25, FTS, `ripgrep` 같은 검색이 더 잘 맞을 수 있다.

## grep도 RAG에 쓸 수 있다

`grep`은 원래 터미널에서 쓰는 텍스트 검색 명령어다.

> [!example]
> 로컬 문서가 적다면 `rg`로 찾은 몇 줄을 LLM에게 넘기는 것도 넓은 의미의 RAG다.
>
> ```text
> 사용자 질문
> -> rg로 관련 문서 줄 검색
> -> 검색 결과를 LLM 프롬프트에 넣음
> -> LLM이 근거를 참고해서 답변
> ```

예를 들어 파일에서 특정 문자열을 찾을 수 있다.

```bash
grep "RAG" TIL.md
```

요즘 코드 검색에서는 `grep`보다 `ripgrep`, 즉 `rg`를 많이 쓴다.

```bash
rg "Transactional"
rg "findByEmail"
rg "ConnectionTimeoutException"
```

RAG 이야기에서 `grep`이나 `rg`가 나오는 이유는, RAG의 검색 단계가 꼭 거대한 검색엔진이나 벡터DB일 필요는 없기 때문이다.

로컬 문서가 적다면 다음 구조도 가능하다.

```text
사용자 질문
-> 서버가 grep/rg로 관련 문서 줄을 찾음
-> 찾은 문장 몇 개를 LLM 프롬프트에 넣음
-> LLM이 그걸 참고해서 답변함
```

이것도 넓은 의미에서는 RAG다.

## Postgres + pgvector

`pgvector`는 Postgres에서 벡터를 저장하고 검색할 수 있게 해주는 확장이다.

예를 들면 문서 조각과 임베딩을 같은 테이블에 저장할 수 있다.

```sql
CREATE EXTENSION vector;

CREATE TABLE documents (
  id bigserial PRIMARY KEY,
  content text,
  embedding vector(1536)
);
```

여기서 `embedding`은 문장이나 문단을 숫자 배열로 바꾼 값이다.

```text
"Spring 트랜잭션 전파 옵션"
-> [0.012, -0.331, 0.882, ...]
```

검색할 때는 사용자 질문도 임베딩으로 바꾼 뒤, 저장된 문서 임베딩과 거리를 비교한다.

```sql
SELECT content
FROM documents
ORDER BY embedding <-> '[0.01, -0.2, ...]'
LIMIT 5;
```

`Postgres + pgvector`가 가볍다고 말하는 경우는, 벡터 검색 자체가 가볍다는 뜻이 아니다.

별도 벡터DB를 운영하지 않고 기존 Postgres 안에서 처리할 수 있어서 운영 구성이 비교적 가볍다는 뜻이다.

```text
무거운 구성:
App + Postgres + 별도 Vector DB

비교적 가벼운 구성:
App + Postgres(pgvector)
```

하지만 벡터 검색 자체는 꽤 묵직한 기능이다.

```text
- 문서를 어떻게 자를지 결정해야 함
- 임베딩 생성 비용이 듦
- 벡터 컬럼 저장 공간이 큼
- 벡터 인덱스를 관리해야 함
- 질문할 때마다 질문 임베딩을 만들어야 함
- 검색 품질 튜닝이 필요함
```

## 임베딩

임베딩은 텍스트를 의미 검색용 숫자 배열로 바꾼 것이다.

중요한 점은 단어 하나하나를 단순히 숫자로 치환하는 것이 아니라, 보통 문장, 문단, 문서 조각 단위로 숫자 배열을 만든다는 것이다.

예를 들어 문서를 chunk로 나눌 수 있다.

```text
chunk 1:
Spring의 @Transactional은 메서드 실행 전후로 트랜잭션을 시작하고 커밋/롤백한다.

chunk 2:
전파 옵션에는 REQUIRED, REQUIRES_NEW, SUPPORTS 등이 있다.
```

각 chunk를 임베딩 모델에 넣으면 숫자 배열이 나온다.

```text
chunk 1 -> [0.12, -0.03, 0.77, ...]
chunk 2 -> [0.44, 0.18, -0.21, ...]
```

사용자 질문도 임베딩으로 바꾼다.

```text
질문:
Spring 트랜잭션 전파 옵션 뭐 있음?

질문 embedding:
[0.40, 0.15, -0.18, ...]
```

그 다음 질문 임베딩과 문서 chunk 임베딩의 거리를 비교해서 가까운 문서를 찾는다.

백엔드 개발자 입장에서는 임베딩 내부 원리까지 깊게 알 필요는 없다.

일단은 이렇게 이해하면 충분하다.

```text
임베딩 = 텍스트를 의미 기반 검색이 가능하도록 벡터로 변환한 값
```

RAG에서는 문서 조각과 사용자 질문을 같은 방식으로 임베딩해서 유사한 문서를 찾는 데 사용한다.

## 에이전트가 파일시스템이나 API를 직접 조회하는 방식

여기서 에이전트는 사람이 아니라, LLM이 도구를 써서 직접 찾아보는 방식을 말한다.

일반적인 인덱스 기반 RAG는 보통 이런 흐름이다.

```text
사용자 질문
-> 서버가 미리 만든 검색 인덱스에서 관련 문서 검색
-> 검색 결과를 LLM에게 넣음
-> 답변 생성
```

에이전트형 조회는 이런 흐름이다.

```text
사용자 질문
-> LLM이 어떤 파일을 봐야 할지 판단
-> 파일 목록 조회
-> 특정 파일 열기
-> 필요하면 grep 실행
-> API 호출
-> 결과 읽고 다시 판단
-> 답변 생성
```

에이전트가 직접 할 수 있는 일은 다음과 같다.

```text
- 파일 목록 보기
- 파일 열기
- grep / rg 실행하기
- DB 조회하기
- 내부 API 호출하기
- 로그 검색하기
```

이 방식은 미리 검색 인덱스를 만들 필요가 적어서 설정과 유지보수가 쉬울 수 있다.

파일이 바뀌면 인덱스를 다시 만들 필요 없이, 에이전트가 최신 파일을 직접 읽으면 된다.

하지만 느릴 수 있다.

예를 들어 사용자가 이렇게 물었다고 하자.

```text
로그인 로직 어디서 처리해?
```

에이전트는 한 번에 정답을 아는 것이 아니라 여러 단계를 거친다.

```text
1. 파일 목록 조회
2. auth 관련 파일 검색
3. controller 파일 읽기
4. service 파일 읽기
5. repository 파일 읽기
6. 테스트 파일 검색
7. 정리해서 답변
```

각 단계마다 도구 호출이 필요하고, 결과를 읽은 뒤 다시 판단해야 하므로 느릴 수 있다.

정리하면 다음과 같다.

```text
에이전트형 조회
장점: 최신 파일/API를 직접 봄, 인덱스 관리 부담이 적음
단점: 여러 번 조회해서 느릴 수 있음

인덱스 기반 RAG
장점: 검색이 빠름, 대량 문서에 유리함
단점: 인덱스 생성/갱신/튜닝이 필요함
```

## RAG가 LLM에게 넘기는 것은 짧은 텍스트 조각이다

RAG에서 LLM이 참고하는 재료는 결국 검색해서 가져온 짧은 문단들이다.

예를 들어 질문이 다음과 같다고 하자.

```text
Spring에서 REQUIRES_NEW는 언제 써?
```

RAG가 내부 문서에서 이런 조각을 찾는다.

```text
@Transactional(propagation = REQUIRES_NEW)는 기존 트랜잭션을 잠시 중단하고
새로운 트랜잭션을 시작한다.
```

LLM에게는 질문과 참고자료가 함께 넘어간다.

```text
질문:
Spring에서 REQUIRES_NEW는 언제 써?

참고자료:
@Transactional(propagation = REQUIRES_NEW)는 기존 트랜잭션을 잠시 중단하고
새로운 트랜잭션을 시작한다.
```

그러면 LLM은 이 참고자료를 바탕으로 답변한다.

그래서 RAG 답변 품질은 LLM만의 문제가 아니다.

검색이 좋은 문서 조각을 찾았는지가 매우 중요하다.

검색이 엉뚱한 조각을 가져오면 LLM도 이상한 답변을 할 수 있다.

```text
질문:
REQUIRES_NEW는 언제 써?

검색 결과:
JPA N+1 문제 설명

결과:
트랜잭션 질문인데 N+1 이야기를 섞어서 답변할 수 있음
```

따라서 RAG 성능 개선은 더 좋은 LLM 모델을 쓰는 것만이 아니라, 검색 품질을 개선하는 방향으로도 간다.

```text
- 문서를 잘 쪼갰는가?
- 검색어를 잘 만들었는가?
- 관련 문서를 잘 찾았는가?
- 상위 결과 순서가 적절한가?
- 너무 길거나 너무 짧은 조각을 넘기고 있지는 않은가?
```

## "벡터DB가 없어도 RAG인가?"에 대한 관점

RAG 정의를 넓게 보면 다음 두 가지가 있으면 RAG라고 볼 수 있다.

```text
retrieval = 필요한 정보 검색
generation = LLM이 답변 생성
```

그러면 검색 방식이 무엇이든 상관없다.

```text
grep으로 검색 + LLM 답변 생성
=> RAG라고 볼 수 있음

SQL 검색 + LLM 답변 생성
=> RAG라고 볼 수 있음

BM25 검색 + LLM 답변 생성
=> RAG라고 볼 수 있음

벡터 검색 + LLM 답변 생성
=> 당연히 RAG
```

하지만 현실에서 사람들이 RAG라고 하면 보통 벡터DB 기반 구조를 먼저 떠올리는 경우가 많다.

```text
문서 쪼개기
-> 임베딩 생성
-> 벡터DB 저장
-> 질문 임베딩
-> 유사도 검색
-> LLM 답변
```

정리하면 다음과 같다.

```text
정의상 RAG:
검색 + 생성이면 RAG

업계에서 흔히 떠올리는 RAG:
벡터DB 기반 검색 + 생성
```

오늘 기준으로 기억할 핵심은 이것이다.

```text
RAG의 본질은 벡터DB가 아니라,
검색한 자료를 LLM 답변의 근거로 넣는 것이다.
```

## 코드 검색에는 왜 BM25 + 트라이그램이 잘 맞는가

코드는 자연어 문서와 검색 성질이 다르다.

자연어 문서는 단어가 달라도 의미가 비슷하면 찾는 것이 중요하다.

```text
질문:
로그인이 자꾸 풀려

문서:
세션 만료 정책
```

반면 코드는 의미보다 정확한 식별자, 심볼, 문자열이 중요한 경우가 많다.

```text
findByEmail
@Transactional
REQUIRES_NEW
UserRepository
JWT_SECRET
NullPointerException
/order/{id}
```

이런 것은 "비슷한 의미"보다 "정확히 그 단어가 어디 나오는지"가 훨씬 중요하다.

그래서 코드 검색에서는 벡터 검색보다 BM25, 트라이그램, `ripgrep` 같은 키워드 기반 검색이 더 실용적일 수 있다.

### 임베딩이 코드에 애매할 수 있는 이유

코드에는 프로젝트 안에서만 의미가 있는 고유한 이름이 많다.

```java
calculateSettlementAmount()
findActiveUserByEmail()
JwtAuthenticationFilter
```

벡터 검색은 의미적으로 비슷한 것을 찾으려다 보니, 정확히 필요한 심볼 대신 그럴듯하게 관련 있어 보이는 코드를 가져올 수 있다.

예를 들어 `findByEmail`을 찾고 싶은데 다음 같은 결과가 섞일 수 있다.

```text
findUser
searchAccount
loadMember
```

자연어 검색에서는 이런 유사 결과가 도움이 될 수 있지만, 코드 추적이나 디버깅에서는 노이즈가 될 수 있다.

### BM25

BM25는 검색어가 문서에 얼마나 중요하게 등장하는지 점수화하는 키워드 검색 알고리즘이다.

코드에서는 다음 같은 것들이 강한 검색 힌트가 된다.

```text
- 메서드명
- 클래스명
- 패키지명
- 에러 메시지
- 설정 키
- API path
- SQL column
```

이런 검색에서는 의미 기반 검색보다 키워드 기반 검색이 더 빠르고 예측 가능하다.

### 트라이그램

트라이그램은 문자열을 3글자 단위로 쪼개서 비슷한 문자열을 찾는 방식이다.

예를 들어 `findByEmail`은 다음처럼 쪼갤 수 있다.

```text
fin, ind, ndB, dBy, ByE, yEm, Ema, mai, ail
```

이 방식은 오타, 부분 검색, 비슷한 이름 검색에 도움이 된다.

```text
findEmail
findByMail
findByEmailAddress
```

그래서 코드 검색에서는 BM25로 정확한 키워드 점수를 잡고, 트라이그램으로 유사 문자열까지 보완하는 조합이 좋을 수 있다.

## 리랭커와 재인덱싱 부담

벡터 검색은 보통 의미상 가까운 후보를 찾는 데 강하지만, 후보 중 진짜 정답을 고르는 것은 별도 문제다.

검색 품질을 높이려면 리랭커를 붙이는 경우가 많다.

```text
1차 검색:
후보 50개 가져오기

2차 리랭킹:
질문과 진짜 관련 높은 순서로 다시 정렬
```

하지만 리랭커를 붙이면 구성이 무거워진다.

```text
임베딩 모델
+ 벡터DB
+ 리랭커 모델
+ 재인덱싱
+ 튜닝
```

코드는 파일이 자주 바뀐다.

```text
- 메서드 이름 변경
- 파일 이동
- 클래스 분리
- 패키지명 변경
- 주석 수정
```

벡터 검색에서는 파일을 chunk로 나누고, 각 chunk마다 임베딩을 만들어 저장해야 한다.

파일이 바뀌면 관련 chunk를 다시 임베딩하고 인덱스를 갱신해야 한다.

반면 키워드 기반 검색은 상대적으로 단순하다.

```text
파일 변경
-> 텍스트 인덱스 갱신
```

또는 로컬 환경에서는 그냥 `rg`로 현재 파일을 바로 검색할 수도 있다.

## PostgreSQL에서 BM25와 벡터 검색을 같이 쓰는 방식

PostgreSQL에서도 BM25 기반 검색을 구현할 수 있다.

예를 들어 `plpgsql_bm25` 같은 확장을 쓰면 PostgreSQL 안에서 BM25 검색을 할 수 있다.

여기에 `pgvector`를 결합하면 다음과 같은 구성이 가능하다.

```text
PostgreSQL
+ BM25 검색
+ pgvector 벡터 검색
+ 둘을 합치는 하이브리드 검색
```

이렇게 하면 별도 Elasticsearch나 별도 벡터DB 없이 PostgreSQL 하나로 꽤 많은 검색 실험을 해볼 수 있다.

## RRF: Reciprocal Rank Fusion

RRF는 여러 검색 결과의 순위를 합치는 방식이다.

예를 들어 질문이 다음과 같다고 하자.

```text
로그인 세션 만료 처리 어디?
```

BM25 검색 결과:

```text
1등: session-policy.md
2등: auth-service.java
3등: jwt-filter.java
```

벡터 검색 결과:

```text
1등: auth-service.java
2등: login-flow.md
3등: session-policy.md
```

RRF는 여러 검색 방식에서 상위권에 자주 나온 결과를 좋게 평가해서 최종 순위를 만든다.

```text
최종:
1등: auth-service.java
2등: session-policy.md
3등: login-flow.md
```

정리하면 다음과 같다.

```text
BM25:
정확한 단어 매칭에 강함

벡터 검색:
의미가 비슷한 문서 찾기에 강함

RRF:
둘의 결과를 섞어서 최종 순위를 안정화함
```

## 코드 검색에서 파일 경로 + 시그니처 임베딩

코드 전체를 무작정 임베딩하면 노이즈가 많을 수 있다.

대신 파일 경로와 함수/클래스 시그니처처럼 정보 밀도가 높은 부분에 임베딩을 적용하면 좋은 결과를 얻을 수 있다.

예를 들어 다음 정보만 봐도 파일의 역할이 꽤 잘 드러난다.

```text
src/main/java/com/example/auth/JwtAuthenticationFilter.java
class JwtAuthenticationFilter
doFilterInternal(HttpServletRequest request, ...)
```

이 정보만으로도 "이 파일은 인증/JWT 필터와 관련 있다"는 것을 알 수 있다.

그래서 코드 검색에서는 다음 조합이 실용적일 수 있다.

```text
파일 경로
+ 클래스명
+ 메서드 시그니처
+ 짧은 주석
-> 임베딩

코드 본문
-> BM25 / 트라이그램 / ripgrep
```

즉, 코드 전체를 벡터화하기보다 구조적 힌트는 임베딩하고, 본문은 키워드 검색으로 찾는 방식이다.

## gpt-oss 20B + ripgrep + while 루프 방식

`gpt-oss 20B` 같은 로컬 LLM을 `ripgrep`과 함께 쓰는 에이전틱 방식도 가능하다.

대략적인 흐름은 다음과 같다.

```text
while 답을 못 찾았으면:
  1. LLM이 다음 검색어를 정함
  2. ripgrep으로 코드 검색
  3. 검색 결과를 LLM에게 다시 줌
  4. LLM이 다음 파일을 읽거나 추가 검색을 결정함
  5. 충분하면 답변함
```

이 방식은 미리 완성된 검색 인덱스를 만들어두는 방식이 아니다.

LLM이 `rg`를 도구처럼 쓰면서 직접 찾아가는 방식이다.

장점은 최신 파일을 바로 볼 수 있고, 인덱스 관리 부담이 적다는 점이다.

단점은 여러 번 검색하고 읽어야 해서 느릴 수 있다는 점이다.

## 맥미니 로컬 RAG를 만든다면

맥미니 로컬에 RAG 검색을 구축한다면 처음부터 벡터DB 중심으로 크게 시작하기보다, 단순한 검색부터 단계적으로 키우는 것이 현실적이다.

> [!tip]
> 면접이나 실무 대화에서 짧게 말하면:
>
> RAG는 벡터DB 자체가 아니라 검색 결과를 LLM의 근거로 넣는 구조입니다. 작은 시스템에서는 `ripgrep`이나 BM25로 시작하고, 의미 검색이 필요해질 때 벡터 검색과 RRF를 붙이는 식으로 확장할 수 있습니다.

추천 흐름은 다음과 같다.

```javascript
1단계:
Markdown/코드 파일을 ripgrep으로 검색

2단계:
SQLite FTS5나 PostgreSQL BM25로 인덱싱

3단계:
검색 결과를 로컬 LLM 또는 API LLM에 넣어서 답변

4단계:
부족하면 pgvector나 FAISS로 벡터 검색 추가

5단계:
BM25 + 벡터 검색 + RRF로 하이브리드 검색
```

처음부터 완성형 RAG를 만들기보다, 다음 순서로 가는 것이 백엔드 공부용으로도 좋고 실제로 쓸 수 있는 도구가 될 가능성도 높다.

```javascript
rg
-> BM25
-> 하이브리드 검색
-> 에이전틱 조회
```

오늘 기준으로 중요한 결론은 이것이다.

> [!summary]
> 로컬 RAG는 벡터DB부터 시작하지 않아도 된다.
>
> ```text
> rg
> -> BM25
> -> 하이브리드 검색
> -> 에이전틱 조회
> ```
>
> 작게 시작하고, 필요해질 때 `pgvector`, FAISS, RRF를 붙이면 된다.
