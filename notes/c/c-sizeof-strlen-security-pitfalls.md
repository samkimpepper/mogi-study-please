# C sizeof와 strlen에서 생기는 보안 함정

> 내가 물어본 질문:
>
> - C의 `sizeof`나 `strlen` 같은 것에 보안 약점이 있지 않은가?
> - 배열 길이, 문자열 길이, 버퍼 크기는 어떻게 다른가?
> - 왜 이런 착각이 버퍼 오버플로우나 정보 노출로 이어지는가?

## 한 줄 요약

`sizeof` 자체나 `strlen` 자체가 나쁜 것은 아니다. 문제는 `sizeof`를 실제 사용 길이처럼 착각하거나, `strlen`을 안전한 버퍼 크기 검사처럼 착각할 때 생긴다.

## `sizeof`는 실제 사용 길이가 아니다

`sizeof`는 타입이나 객체가 차지하는 바이트 크기를 알려준다.

```c
char name[10] = "mogi";
```

```c
sizeof(name); // 10
strlen(name); // 4
```

둘은 의미가 다르다.

```text
sizeof(name)
-> name 배열에 확보된 전체 바이트 수

strlen(name)
-> '\0' 전까지의 문자열 길이
```

즉 `sizeof(name)`은 "현재 의미 있게 들어 있는 글자 수"가 아니다.

## 함수에 넘기면 `sizeof`가 더 헷갈린다

배열을 함수 파라미터로 넘기면 대부분 포인터처럼 취급된다.

```c
void f(char name[]) {
    printf("%zu\n", sizeof(name));
}
```

이 함수 안에서 `name`은 배열 전체가 아니라 사실상 `char *`다.

```text
함수 밖:
sizeof(name) = 배열 전체 크기

함수 안:
sizeof(name) = 포인터 크기
```

그래서 다음 코드는 위험하다.

```c
void clear(char buf[]) {
    memset(buf, 0, sizeof(buf));
}
```

개발자는 버퍼 전체를 지운다고 생각했지만, 실제로는 포인터 크기만큼만 지울 수 있다.

```text
원한 것:
버퍼 전체 초기화

실제:
포인터 크기만큼만 초기화
```

보안적으로는 비밀번호, 토큰, 키 같은 민감 정보 일부가 메모리에 남을 수 있다.

## `strlen`은 `'\0'`을 만날 때까지 읽는다

C 문자열은 끝에 널 문자 `'\0'`이 있어야 한다.

```c
char s[] = "mogi";
```

메모리는 대충 이렇게 생긴다.

```text
'm' 'o' 'g' 'i' '\0'
```

`strlen(s)`는 배열 전체 크기를 아는 함수가 아니다. 앞에서부터 읽다가 `'\0'`을 만나면 멈춘다.

```c
strlen(s); // 4
```

문제는 널 종료 문자가 없을 때다.

```c
char buf[4] = {'m', 'o', 'g', 'i'};

strlen(buf); // 위험
```

`buf` 안에 `'\0'`이 없기 때문에 `strlen`은 버퍼 밖 메모리까지 계속 읽을 수 있다.

```text
[m][o][g][i][옆 메모리][옆 메모리] ... ['\0']
```

이런 상황은 다음 문제로 이어질 수 있다.

- 버퍼 밖 읽기
- 민감한 메모리 노출
- 프로그램 크래시
- 잘못된 길이 계산
- 이후 복사 로직과 결합된 버퍼 오버플로우

## 위험한 함수와 연결될 때 더 커진다

예를 들어 `strcpy`는 `'\0'`을 만날 때까지 복사한다.

```c
char src[4] = {'m', 'o', 'g', 'i'};
char dst[4];

strcpy(dst, src); // 위험
```

`src`에 널 종료 문자가 없으면 `strcpy`는 `src` 밖까지 읽고, `dst` 밖까지 쓸 수 있다.

입력에서도 비슷한 문제가 생긴다.

```c
char buf[8];

scanf("%s", buf); // 길이 제한 없음
```

사용자가 100글자를 입력하면 `buf` 8칸을 넘어 써버릴 수 있다.

## 셋을 구분해야 한다

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| 버퍼 크기 | 읽고 써도 되는 전체 공간 | `char buf[10]`의 10칸 |
| 문자열 길이 | `'\0'` 전까지의 글자 수 | `"mogi"`의 4 |
| 포인터 크기 | 주소값 자체의 크기 | `char *p`의 8바이트 등 |

이 셋을 섞으면 위험해진다.

```c
char buf[10] = "mogi";
```

```text
sizeof(buf) = 10
strlen(buf) = 4
```

하지만 함수에 넘기면:

```c
void f(char *buf) {
    sizeof(buf); // 포인터 크기
    strlen(buf); // '\0'을 만날 때까지 읽음
}
```

둘 다 "버퍼 전체 크기"를 안정적으로 알려주지 못한다.

## 백엔드 관점으로 보면

Java의 `String`이나 `byte[]`는 길이 정보를 가진 객체다.

```java
String s = "mogi";
s.length(); // 4

byte[] body = new byte[10];
body.length; // 10
```

반면 C의 `char *`는 그냥 시작 주소에 가깝다.

```c
char *s;
```

이 말은 대충 이런 계약에 가깝다.

```text
여기서부터 문자열일 것이다.
끝은 '\0'이 나오면 끝일 것이다.
버퍼 크기는 호출자가 따로 알고 있어야 한다.
```

그래서 C에서는 포인터만 넘기지 말고, 보통 크기나 길이를 같이 넘기는 습관이 중요하다.

```c
void handle(char *buf, size_t buf_size) {
    /* buf_size 안에서만 읽고 쓴다 */
}
```

## 결론

`sizeof`, `strlen`, 버퍼 크기는 서로 다르다.

```text
sizeof
-> 현재 스코프에서 보이는 객체/타입의 바이트 크기

strlen
-> '\0'을 만날 때까지 센 문자열 길이

buffer size
-> 실제로 읽고 써도 되는 최대 공간
```

보안 문제는 이 차이를 헷갈릴 때 생긴다. 특히 함수에 배열을 넘긴 뒤 `sizeof`를 배열 크기처럼 쓰거나, 널 종료가 보장되지 않은 데이터에 `strlen`/`strcpy`를 쓰는 패턴은 위험하다.
