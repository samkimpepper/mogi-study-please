# C 배열이 길이 메타데이터를 들고 있지 않는 이유

> [!question]
> 내가 물어본 질문:
>
> - C가 그냥 옛날 언어라서 배열 안에 자기 길이를 저장하지 않는 건가?
> - Java 배열도 결국 연속된 메모리 덩어리 아닌가?
> - 길이 하나 안 들고 있는 게 임베디드에서 그렇게 큰 차이가 나나?

> [!summary]
> 결론:
>
> - Java 배열도 실제 원소들은 연속된 메모리 기반으로 배치된다.
> - 차이는 C 배열은 메타데이터 없는 연속 메모리에 가깝고, Java 배열은 길이 정보를 가진 객체라는 점이다.
> - 길이 정보 하나가 항상 큰 성능 차이를 만드는 것은 아니다.
> - 하지만 C는 모든 배열에 고정 비용을 강제로 붙이지 않고, 필요한 경우 개발자가 직접 길이를 관리하게 설계된 언어다.
> - 특히 작은 버퍼가 많거나 메모리 배치가 중요한 임베디드, OS, 네트워크, 파일 포맷 영역에서는 이 차이가 의미 있다.

관련 노트: [[C 배열 길이와 Java 배열의 차이]]

## 먼저 잡을 한 줄 정의

> [!info]
> C 배열은 "값이 연속된 공간"이고, Java 배열은 "길이 메타데이터가 붙은 배열 객체"에 가깝다.

## Java 배열도 연속 메모리 아닌가

Java 배열도 원소 접근을 빠르게 하기 위해 실제 데이터는 연속된 메모리 기반으로 둔다고 보면 된다.

배열 인덱스 접근은 보통 이런 계산에 가깝다.

```text
arr[3] 위치 = 배열 시작 주소 + (3 * 원소 크기)
```

이 감각은 C 배열과 Java 배열이 비슷하다.

차이는 배열 앞이나 주변에 런타임이 관리하는 정보가 붙어 있느냐다.

```text
C 배열:
[값][값][값][값][값]
```

```text
Java 배열:
[객체 헤더][length][값][값][값][값][값]
```

> [!warning]
> "C만 연속 메모리이고 Java는 아니다"가 아니다.
> 더 정확히는 C 배열은 메타데이터 없는 연속 메모리이고, Java 배열은 메타데이터가 붙은 객체 안에 연속 데이터가 있는 구조에 가깝다.

## 길이 하나가 그렇게 큰 차이를 만드나

배열 하나만 보면 길이 정보는 별것 아닌 것처럼 보인다.

```c
int arr[5];
```

`int`가 4바이트라면 데이터 자체는 20바이트다.

```text
int 5개 = 20바이트
length  = 4바이트 정도
```

이 경우에는 길이 정보 하나가 엄청 커 보이지 않는다.

하지만 작은 배열이 많으면 이야기가 달라진다.

```c
char flags[2];
char code[3];
char name[8];
```

`char[2]`에 길이 4바이트가 강제로 붙는다고 생각하면, 실제 데이터보다 관리 정보가 더 커질 수 있다.

```text
데이터 2바이트
길이 4바이트
```

임베디드에서는 RAM이 몇 KB 또는 몇십 KB인 경우도 있어서, 이런 고정 오버헤드가 의미 있는 비용이 될 수 있다.

> [!info]
> 핵심은 "길이 하나가 언제나 엄청난 성능 차이를 만든다"가 아니다.
> C는 "필요하지 않은 비용을 언어가 모든 배열에 강제로 붙이지 않는다"는 쪽에 가깝다.

## 메모리 배치가 중요한 경우

C에서는 배열이 구조체 안에 들어가서 네트워크 패킷, 파일 포맷, 하드웨어 레지스터 같은 실제 메모리 구조를 표현하는 경우가 많다.

```c
struct Packet {
    char header[4];
    char payload[128];
    int checksum;
};
```

이런 구조에서는 필드가 메모리에 어떤 순서와 크기로 배치되는지가 중요하다.

만약 배열마다 자동으로 `length`가 끼어들면 구조체의 메모리 모양이 달라진다.

```text
원하는 배치:
[header 4바이트][payload 128바이트][checksum]

길이 정보가 강제로 붙는 배치:
[header length][header][payload length][payload][checksum]
```

네트워크로 받은 바이트를 그대로 구조체처럼 해석하거나, 하드웨어가 정해둔 주소에 맞춰 데이터를 써야 하는 상황에서는 이런 자동 메타데이터가 오히려 방해가 된다.

## C는 필요하면 직접 붙이는 방식

C에서도 길이가 필요하면 개발자가 직접 구조체를 만들 수 있다.

```c
struct Buffer {
    size_t len;
    char data[128];
};
```

이렇게 하면 Java 배열처럼 길이와 데이터를 같이 들고 다니는 구조를 만들 수 있다.

반대로 길이가 필요 없으면 그냥 배열만 둘 수 있다.

```c
char data[128];
```

> [!tip]
> C의 기본 선택은 "항상 안전하고 편하게"가 아니라 "필요한 만큼만 비용을 내게" 하는 쪽이다.

## 백엔드 관점으로 보면

백엔드에서 요청 바디나 파일 업로드 데이터를 다룰 때도 보통 실제 데이터와 길이를 함께 생각한다.

```text
pointer + length
buffer + size
byte[] + length
```

C에서는 이 조합을 언어가 자동으로 객체에 넣어주지 않는다. 그래서 함수에 버퍼를 넘길 때 보통 포인터와 길이를 같이 넘긴다.

```c
void handle_packet(const char *buf, size_t len) {
    /* buf는 데이터 시작 위치, len은 읽어도 되는 바이트 수 */
}
```

이 패턴을 쓰면 함수 안에서 어디까지 읽어도 되는지 알 수 있다.

## 면접에서 말하면

> [!tip]
> C 배열이 길이를 들고 있지 않은 이유는 단순히 오래된 언어라서라기보다, C 배열이 메타데이터 없는 연속 메모리로 설계되었기 때문입니다. Java 배열도 원소 데이터는 연속 메모리 기반으로 배치될 수 있지만, Java 배열은 객체라서 `length` 같은 런타임 정보를 함께 가집니다. C는 모든 배열에 길이 정보를 강제로 붙이지 않고, 필요한 경우 개발자가 `struct`나 별도 변수로 길이를 관리하게 합니다. 이 방식은 작은 버퍼가 많거나 정확한 메모리 배치가 중요한 임베디드, OS, 네트워크 코드에서 의미가 있습니다.

## 내가 다시 헷갈릴 것 같은 부분

- Java 배열도 원소 접근을 위해 연속 메모리 기반으로 배치될 수 있다.
- C 배열과 Java 배열의 차이는 "연속 메모리냐 아니냐"가 아니라 "길이 같은 메타데이터를 객체가 들고 있느냐"다.
- 길이 정보 하나는 배열 하나만 보면 작지만, 작은 배열이 많으면 비율상 큰 비용이 될 수 있다.
- C는 비용을 자동으로 강제하지 않고, 필요하면 개발자가 `len`을 직접 같이 관리한다.
- 네트워크 패킷, 파일 포맷, 하드웨어 레지스터처럼 메모리 배치가 중요한 영역에서는 자동 메타데이터가 방해가 될 수 있다.
