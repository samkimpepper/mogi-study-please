# C 포인터 산술과 이식성

> 내가 물어본 질문:
>
> - 예전에는 주소를 몇 바이트 올릴지 개발자가 직접 계산했는가?
> - `pointer++`가 왜 고마운 기능이라고 하는가?
> - CPU 레지스터 길이가 다르면 지금과 무엇이 달랐는가?

## 한 줄 요약

C의 `pointer++`는 주소를 무조건 1바이트 올리는 문법이 아니라, 포인터가 가리키는 타입의 실제 크기만큼 다음 원소로 이동하게 해주는 추상화다.

## 주소도 결국 숫자다

CPU 입장에서 주소는 메모리 위치를 나타내는 숫자다.

그래서 어셈블리처럼 더 낮은 수준에서는 다음 원소로 이동하기 위해 주소 레지스터에 직접 값을 더할 수 있다.

```asm
add rax, 4   ; rax에 들어 있는 주소를 4바이트 증가
```

만약 `rax`가 `int` 배열의 현재 원소 주소를 들고 있고, 이 환경에서 `int`가 4바이트라면 다음 `int`로 이동하는 코드가 된다.

문제는 개발자가 항상 "이 타입은 몇 바이트인가?"를 정확히 알아야 한다는 점이다.

## C의 `p++`는 타입 크기만큼 이동한다

C에서는 다음처럼 쓸 수 있다.

```c
int arr[3] = {1, 2, 3};
int *p = arr;

p++;
```

`p++`는 주소를 `+1` 하는 것이 아니다. `p`가 `int *`이므로 `sizeof(int)`만큼 이동한다.

```text
int가 4바이트인 환경: p = p + 4바이트
int가 2바이트인 환경: p = p + 2바이트
```

즉 개발자가 바이트 수를 직접 박아 넣는 대신, 컴파일러가 해당 타입과 플랫폼에 맞는 주소 계산을 해준다.

## 구조체에서는 더 중요해진다

구조체는 필드 크기를 단순히 더한 값과 실제 크기가 다를 수 있다.

```c
struct User {
    int id;
    char grade;
};
```

겉으로 보면 `int` 4바이트 + `char` 1바이트라서 5바이트처럼 보인다. 하지만 실제로는 정렬과 패딩 때문에 8바이트가 될 수 있다.

```text
[ id 4바이트 ][ grade 1바이트 ][ padding 3바이트 ]
```

구조체 배열에서 다음 원소로 이동할 때는 이 실제 크기만큼 이동해야 한다.

```c
struct User users[10];
struct User *p = users;

p++;  // sizeof(struct User)만큼 이동
```

여기서 `p++`는 `sizeof(struct User)`만큼 이동한다. 구조체가 8바이트면 8바이트, 다른 ABI에서 12바이트가 되면 12바이트만큼 이동한다.

> [!summary]
> `p++`의 핵심은 "주소를 증가시킨다"가 아니라 "이 타입의 다음 원소로 이동한다"이다.

## 직접 계산이 가능한 이유와 위험한 이유

직접 계산은 가능하다. 주소가 숫자이기 때문이다.

```text
다음 원소 주소 = 시작 주소 + 인덱스 * 원소 크기
```

배열 접근도 개념적으로는 이런 계산이다.

```c
arr[i]
```

는 대략 다음 의미에 가깝다.

```text
*(arr + i)
```

여기서 `arr + i`도 바이트 단위 `+ i`가 아니라, `i * sizeof(원소 타입)`만큼 이동한다.

위험한 부분은 사람이 바이트 수를 직접 계산하면 타입 크기, 구조체 패딩, CPU/ABI 차이에 쉽게 묶인다는 점이다.

## CPU 아키텍처가 달라도 OS를 작성할 수 있다는 말

예전에는 8비트, 16비트, 32비트처럼 CPU가 자연스럽게 처리하는 폭이 더 다양했다. 지금도 임베디드까지 보면 다양하지만, 일반 PC/서버 환경은 64비트에 많이 수렴해 있다.

CPU 아키텍처가 달라지면 다음이 달라질 수 있다.

- 포인터 크기
- 정수 타입 크기
- 정렬 규칙
- 구조체 패딩
- 함수 호출 규약

어셈블리만으로 OS를 쓰면 이런 차이를 사람이 직접 많이 맞춰야 한다.

반면 C로 쓰면 다음처럼 타입 중심으로 표현할 수 있다.

```c
struct Page *page = pages;
page++;
```

컴파일러가 대상 CPU와 ABI에 맞춰 실제 주소 계산 코드를 만든다.

그래서 C는 하드웨어를 완전히 숨기는 언어는 아니지만, 어셈블리보다 훨씬 이식성 있게 시스템 소프트웨어를 쓸 수 있게 해줬다.

## CSAPP에서 본 레지스터 이름과의 연결

CSAPP에서 본 x86-64 레지스터 이름은 보통 이런 표다.

```text
64비트: %rax
32비트: %eax
16비트: %ax
8비트:  %al
```

이 표는 x86-64 안에서 같은 물리 레지스터의 하위 몇 비트를 어떤 이름으로 접근하는지 보여준다.

반면 "CPU 레지스터 길이가 다르다"는 말은 CPU 세대나 아키텍처마다 자연스럽게 다루는 폭이 8/16/32/64비트처럼 달랐다는 뜻에 가깝다.

둘은 연결되어 있지만 같은 말은 아니다.

## 백엔드 관점 비유

Java에서 다음 코드를 쓴다고 생각하면 된다.

```java
users.get(i + 1)
```

개발자는 내부 메모리 주소를 직접 계산하지 않는다. 리스트나 배열 구현이 다음 원소 위치를 찾아준다.

C의 `p++`도 비슷하게, 개발자가 바이트 오프셋을 직접 계산하지 않고 "다음 원소"라는 의도를 표현하게 해준다.

차이는 C에서는 그 결과가 실제 메모리 주소 이동으로 바로 이어진다는 점이다.

## 한 줄 회고
