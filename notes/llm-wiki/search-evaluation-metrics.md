# 검색 평가 지표 기초

> [!summary]
> 검색 평가 지표는 검색 시스템이 정답 문서를 얼마나 잘 찾아오는지 숫자로 확인하는 방법이다.
>
> LLM wiki 시스템에서는 LLM 답변 자체를 보기 전에, 먼저 LLM에게 넘겨줄 근거 문서가 제대로 검색됐는지 봐야 한다.

## 왜 먼저 봐야 하나

LLM wiki나 RAG 시스템은 보통 이런 흐름으로 동작한다.

```text
사용자 질문
-> 관련 문서 검색
-> 검색 결과를 LLM 프롬프트에 넣음
-> LLM이 답변 생성
```

여기서 검색 결과가 틀리면 LLM이 아무리 말을 잘해도 답변이 이상해질 수 있다.

예를 들어 사용자가 이렇게 물었다고 하자.

```text
회원 탈퇴 API는 어디에 있어?
```

그런데 검색 시스템이 회원 탈퇴 문서 대신 로그인 문서, 결제 문서, 공지사항 문서를 가져오면 LLM은 잘못된 근거를 보고 답하게 된다.

> [!info]
> 백엔드 관점에서 보면 검색 평가는 API 테스트와 비슷하다.
>
> ```text
> 입력: 질문
> 기대값: 관련 문서 목록
> 실제값: 검색 시스템이 가져온 문서 목록
> 평가: Precision, Recall, Precision@K, Recall@K
> ```

## Ground Truth와 Gold Set

`ground truth` 또는 `gold set`은 평가 기준으로 쓰는 정답지다.

예를 들면 다음처럼 질문별로 정답 문서를 미리 정해둔다.

```text
질문:
회원 탈퇴 로직은 어디에 있나?

정답 문서:
- UserController.deleteUser
- UserService.withdraw
- user-withdrawal-policy.md
```

검색 시스템이 어떤 결과를 가져왔을 때, 이 정답지와 비교해야 맞았는지 틀렸는지 판단할 수 있다.

> [!warning]
> gold set이 없으면 검색 품질을 감으로만 보게 된다.
>
> "좋아 보인다", "잘 찾는 것 같다"가 아니라, 질문별 정답지를 기준으로 숫자를 계산해야 PR 본문에 있는 평가 결과를 읽을 수 있다.

## Precision

`Precision`은 가져온 검색 결과 중에서 맞는 것이 얼마나 많은지 보는 지표다.

```text
Precision = 맞게 가져온 개수 / 전체 가져온 개수
```

예를 들어 검색 시스템이 문서 5개를 가져왔고, 그중 3개가 실제로 유용한 문서라면 다음과 같다.

```text
Precision = 3 / 5 = 0.6
```

Precision이 낮다는 것은 검색 결과에 쓸데없는 문서가 많이 섞였다는 뜻이다.

> [!example]
> 질문이 "회원 탈퇴 API는 어디에 있어?"인데 검색 결과에 로그인, 결제, 배포 문서가 많이 섞이면 Precision이 낮아진다.

## Recall

`Recall`은 전체 정답 중에서 검색 시스템이 얼마나 많이 찾아냈는지 보는 지표다.

```text
Recall = 맞게 가져온 개수 / 전체 정답 개수
```

예를 들어 실제 정답 문서가 총 6개인데, 검색 시스템이 그중 3개만 찾았다면 다음과 같다.

```text
Recall = 3 / 6 = 0.5
```

Recall이 낮다는 것은 중요한 정답 문서를 놓쳤다는 뜻이다.

> [!warning]
> LLM wiki에서는 Recall이 낮으면 답변 근거가 부족해질 수 있다.
>
> 특히 여러 문서를 함께 봐야 하는 질문에서 정답 문서 일부를 놓치면 답변이 반쪽짜리가 될 수 있다.

## Precision@K

`Precision@K`는 상위 K개 검색 결과 중에서 맞는 것이 얼마나 많은지 보는 지표다.

```text
Precision@5 = 상위 5개 중 맞는 개수 / 5
```

예를 들어 상위 5개 결과 중 3개가 유용하면 다음과 같다.

```text
Precision@5 = 3 / 5 = 0.6
```

검색 시스템에서는 상위 결과가 특히 중요하다. 사용자가 전체 결과를 다 보지 않고, LLM도 보통 상위 몇 개 문서만 context로 받기 때문이다.

> [!info]
> `Precision@5`는 "LLM에게 넣을 상위 5개 문서가 얼마나 깨끗한가"를 보는 지표라고 이해하면 된다.

## Recall@K

`Recall@K`는 전체 정답 중에서 상위 K개 안에 들어온 정답이 얼마나 많은지 보는 지표다.

```text
Recall@5 = 상위 5개 안에 들어온 정답 개수 / 전체 정답 개수
```

예를 들어 전체 정답이 6개이고, 상위 5개 안에 정답 3개가 들어왔다면 다음과 같다.

```text
Recall@5 = 3 / 6 = 0.5
```

> [!summary]
> `Precision@5`는 상위 5개가 얼마나 알짜인지 보는 지표다.
>
> `Recall@5`는 전체 정답 중 상위 5개 안에 얼마나 많이 담겼는지 보는 지표다.

## 예시로 한 번에 보기

질문과 정답지가 다음과 같다고 하자.

```text
질문:
회원 탈퇴 API는 어디에 있어?

gold set:
- 회원 탈퇴 API 설계 문서
- 회원 API 컨트롤러 설명
- 회원 탈퇴 DB 처리 문서
- 회원 탈퇴 정책 문서
```

검색 시스템의 상위 5개 결과가 다음과 같다고 하자.

```text
1. 회원 탈퇴 API 설계 문서       O
2. 회원 API 컨트롤러 설명        O
3. 로그인 API 문서               X
4. 회원 탈퇴 DB 처리 문서        O
5. 결제 취소 API 문서            X
```

그러면 상위 5개 중 맞는 결과는 3개다.

```text
Precision@5 = 3 / 5 = 0.6
Recall@5 = 3 / 4 = 0.75
```

이 결과는 이렇게 읽으면 된다.

```text
상위 5개 중 60%는 유용했다.
전체 정답 4개 중 75%를 상위 5개 안에서 찾았다.
```

## PR 본문에서 읽는 법

PR 본문에 이런 문장이 있다고 하자.

```text
Recall@5 improved from 0.62 to 0.81 on the gold set.
```

이 말은 다음 뜻이다.

```text
질문별 정답지 기준으로 봤을 때,
상위 5개 검색 결과 안에 정답 문서가 들어오는 비율이
0.62에서 0.81로 좋아졌다.
```

다른 예시도 보자.

```text
Precision@5 dropped slightly, but Recall@5 improved.
```

이 말은 다음처럼 읽을 수 있다.

```text
상위 5개 안에 정답을 더 많이 넣는 데는 성공했다.
대신 쓸데없는 문서도 조금 더 섞였을 수 있다.
```

검색 시스템에서는 Precision과 Recall이 서로 trade-off 관계가 될 때가 많다.

```text
결과를 넓게 가져오면:
정답을 놓칠 확률은 줄어듦 -> Recall 증가
노이즈가 섞일 확률은 늘어남 -> Precision 감소

결과를 빡세게 걸러내면:
노이즈는 줄어듦 -> Precision 증가
정답을 놓칠 수 있음 -> Recall 감소
```

## 실무에서 보는 포인트 

| 관점 | 확인할 것 | 왜 중요한가 |
| --- | --- | --- |
| 검색 품질 | Recall@K | 정답 문서를 상위 결과 안에 넣었는지 확인 |
| 검색 노이즈 | Precision@K | LLM에게 쓸데없는 근거가 많이 들어가는지 확인 |
| 평가 기준 | gold set | 감이 아니라 정답지 기준으로 평가하기 위해 필요 |
| LLM 답변 | 검색 결과 품질 | 답변 품질은 검색 품질에 크게 의존함 |

> [!tip]
> 짧게 말하면:
>
> 검색 평가는 "LLM이 답을 잘했나" 이전에 "LLM에게 줄 근거 문서를 잘 찾았나"를 보는 단계다. `Recall@5`는 상위 5개 안에 정답을 얼마나 담았는지, `Precision@5`는 상위 5개가 얼마나 쓸모 있는지 보는 지표다.

## 나중에 다시 볼 포인트

- `ground truth`와 `gold set`은 검색 평가용 정답지다.
- `Precision`은 가져온 것 중 정답 비율이다.
- `Recall`은 전체 정답 중 찾아낸 비율이다.
- `@K`는 상위 K개 결과만 보고 계산한다는 뜻이다.
- LLM wiki에서는 *==검색 결과가==* 나쁘면 LLM 답변도 같이 흔들릴 수 있다.
