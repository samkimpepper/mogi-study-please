# 가설 검정 아주 기초

> [!summary]
> 가설 검정은 실험에서 나온 차이가 진짜 의미 있는 차이인지, 아니면 우연히 그렇게 보인 것인지 판단하려는 통계 절차다.
>
> LLM wiki 검색 평가에서는 "새 검색 방식의 Recall@5가 높아졌는데, 이게 진짜 개선인가?"를 볼 때 필요하다.

## 왜 봐야 하나

검색 실험에서 이런 결과가 나왔다고 하자.

```text
기존 검색 Recall@5 = 0.65
새 검색 Recall@5 = 0.76
```

겉으로 보면 새 검색 방식이 좋아 보인다.

하지만 바로 이렇게 말하면 위험하다.

```text
새 검색 방식이 더 좋다는 것이 증명됐다.
```

gold task가 적거나, 문제 구성이 우연히 새 방식에 유리하거나, 여러 설정 중 좋아 보이는 숫자만 골랐을 수도 있기 때문이다.

> [!info]
> 백엔드 관점에서 보면 가설 검정은 "이번 성능 차이를 운영에 반영할 만큼 믿어도 되는가?"를 따져보는 절차에 가깝다.

## 귀무가설 H0

`H0`는 귀무가설이라고 부른다.

가설 검정에서는 보통 "차이가 없다"를 기본 가정으로 둔다.

```text
H0:
새 검색 방식은 기존 방식보다 실제로 좋아진 것이 아니다.
관측된 차이는 우연일 수 있다.
```

반대로 우리가 기대하는 가설은 보통 다음과 같다.

```text
H1:
새 검색 방식은 기존 방식보다 실제로 좋아졌다.
```

> [!warning]
> 가설 검정의 기본 태도는 "좋아 보이는 숫자가 나왔으니 바로 믿자"가 아니다.
>
> 먼저 "차이가 없다고 가정해도 이런 결과가 나올 수 있나?"를 따진다.

## p-value

`p-value`는 귀무가설이 맞다고 가정했을 때, 지금 관측한 결과만큼 또는 그보다 더 극단적인 결과가 나올 가능성을 보는 값이다.

쉽게 말하면 다음 질문에 가깝다.

```text
진짜 차이가 없다고 치자.
그런데도 이번처럼 새 방식이 더 좋아 보이는 결과가 나오는 게 얼마나 이상한가?
```

예를 들어 p-value가 다음처럼 나왔다고 하자.

```text
p-value = 0.03
```

그러면 보통 이렇게 해석한다.

```text
차이가 없다고 보기에는 이번 결과가 꽤 이상하다.
```

하지만 p-value를 다음처럼 이해하면 안 된다.

```text
p-value = 새 방식이 진짜 좋을 확률
p-value = 결과가 우연일 확률
```

> [!warning]
> p-value는 "새 방식이 맞을 확률"이 아니다.
>
> 정확히는 "H0가 참이라고 가정했을 때, 지금 같은 결과가 얼마나 나오기 어려운가"에 가깝다.

## Statistical Significance

`statistical significance`는 관측된 차이가 통계적으로 유의하다고 판단하는 것이다.

보통 실험 전에 기준을 정한다.

```text
p-value < 0.05 이면 통계적으로 유의하다고 본다.
```

예를 들어 다음 결과라면 통계적으로 유의하다고 말할 수 있다.

```text
p-value = 0.03
```

하지만 통계적으로 유의하다는 말이 곧 실무적으로 중요하다는 뜻은 아니다.

```text
Recall@5 0.801 -> 0.804
p-value는 작음
```

이런 경우 숫자 차이는 통계적으로 유의할 수 있지만, 실제 사용자 경험이나 운영 품질에는 거의 의미가 없을 수 있다.

반대로 표본이 너무 작으면 꽤 좋아 보이는 차이도 통계적으로 유의하지 않을 수 있다.

> [!info]
> 통계적 유의성은 "차이가 있다고 볼 근거가 충분한가"를 말한다.
>
> 실무적 중요성은 "그 차이가 실제로 쓸 만한가"를 따로 봐야 한다.

## Fail to Reject

`fail to reject`는 귀무가설을 기각하지 못했다는 뜻이다.

예를 들어 결과가 다음과 같다고 하자.

```text
기존 검색 Recall@5 = 0.65
새 검색 Recall@5 = 0.72
p-value = 0.18
```

보통 기준을 `0.05`로 잡았다면, 이 결과는 통계적으로 유의하다고 보기 어렵다.

이때 정확한 표현은 다음이다.

```text
귀무가설을 기각하지 못했다.
이번 데이터만으로는 새 방식이 좋아졌다고 말하기 어렵다.
```

하지만 이것을 다음처럼 말하면 안 된다.

```text
새 방식은 효과가 없다.
```

> [!summary]
> 증명 실패는 효과 없음 증명이 아니다.
>
> 데이터가 적어서 증명하지 못한 것일 수도 있고, gold task가 너무 적어서 검정력이 약했을 수도 있다.

백엔드 테스트로 비유하면 다음과 비슷하다.

```text
테스트가 실패하지 않았다
!= 버그가 절대 없다
```

마찬가지로 통계적으로 유의하지 않다는 말은 다음과 다르다.

```text
좋아졌다는 근거가 부족하다
!= 효과가 전혀 없다
```

## Multiple Comparisons

`multiple comparisons`는 비교를 많이 할수록 우연히 좋아 보이는 결과가 하나쯤 나올 확률이 커지는 문제다.

검색 실험에서 여러 설정을 비교한다고 하자.

```text
A: BM25 only
B: vector only
C: BM25 + vector
D: BM25 + reranker
E: BM25 + vector + reranker
F: chunk size 300
G: chunk size 500
H: chunk size 800
```

실제로는 별 차이가 없어도, 이렇게 많이 비교하면 어떤 하나는 우연히 좋아 보일 수 있다.

> [!example]
> 주사위 한 번 던져서 6이 나올 확률은 낮다.
>
> 하지만 주사위를 100번 던지면 6은 거의 반드시 나온다.
>
> 검색 실험도 비슷하게, 설정을 많이 바꿔보면 우연히 좋은 점수가 하나쯤 나올 수 있다.

PR에서 다음 패턴이 보이면 조심해야 한다.

```text
여러 검색 설정을 비교했다.
여러 K 값을 봤다.
여러 지표를 봤다.
여러 task subset을 봤다.
그중 제일 좋아 보이는 숫자만 PR에 적었다.
```

이러면 진짜 개선이 아니라, 많이 비교하다가 운 좋게 예쁜 숫자를 찾은 것일 수 있다.

그래서 비교를 많이 했을 때는 보정이 필요하다.

대표적인 보정 방식 이름은 다음 정도만 알아두면 된다.

```text
Bonferroni correction
False Discovery Rate
```

지금 단계에서 핵심은 방식의 수식이 아니라 다음 문장이다.

> [!warning]
> 비교를 많이 했으면 "우연히 좋아 보이는 결과"에 더 엄격해야 한다.

## LLM Wiki PR에서 읽는 법

PR 본문에 이런 문장이 있다고 하자.

```text
Recall@5 improved on 17 gold tasks.
```

이때 바로 "좋아졌구나"에서 끝내면 안 된다.

다음 질문을 같이 봐야 한다.

```text
1. 기존 방식과 같은 task로 비교했나?
2. task 17개에서 차이가 얼마나 났나?
3. 그 차이가 우연일 가능성을 봤나?
4. 여러 지표나 설정 중 좋아 보이는 것만 고른 건 아닌가?
5. 유의하지 않은 결과를 "효과 없음" 또는 "완전한 개선"처럼 과장하지 않았나?
```

gold task가 17개라면 표본이 작아서 통계적으로 강하게 말하기 어려울 수 있다.

좋은 표현은 다음에 가깝다.

```text
17개 gold task에서는 새 방식이 더 높은 Recall@5를 보였다.
다만 표본이 작아 통계적으로 확정적인 결론은 어렵다.
```

위험한 표현은 다음에 가깝다.

```text
새 검색 방식이 기존보다 우월하다는 것이 증명됐다.
```

## 한 번에 정리

| 개념 | 뜻 | 기억할 점 |
| --- | --- | --- |
| H0 | 차이가 없다는 기본 가정 | 먼저 차이가 없다고 두고 시작 |
| p-value | H0가 맞을 때 지금 같은 결과가 얼마나 이상한지 보는 값 | 새 방식이 맞을 확률이 아님 |
| statistical significance | 통계적으로 유의하다고 판단 | 실무적으로 큰 효과라는 뜻은 아님 |
| fail to reject | H0를 기각하지 못함 | 효과 없음 증명이 아님 |
| multiple comparisons | 비교를 많이 해서 우연히 좋아 보이는 결과가 나오는 문제 | 보정이나 사전 계획이 필요 |

> [!tip]
> 짧게 말하면:
>
> 가설 검정은 새 검색 방식이 좋아 보이는 숫자가 나왔을 때, 그 차이가 우연인지 아닌지를 따져보는 절차다. 통계적으로 유의하지 않다는 말은 "효과가 없다"가 아니라 "이번 데이터만으로는 좋아졌다고 말할 근거가 부족하다"에 가깝다.

## 나중에 다시 볼 포인트

- `H0`는 차이가 없다는 기본 가정이다.
- `p-value`는 새 방식이 진짜일 확률이 아니다.
- `statistical significance`와 실무적 중요성은 다르다.
- `fail to reject`는 효과 없음 증명이 아니다.
- 여러 설정을 많이 비교하면 우연히 좋아 보이는 결과가 나올 수 있다.
- 비교를 많이 했으면 보정이나 pre-registration이 중요해진다.
