# 브라우저 기본 UI와 HTML 의미

## 한 줄 정리

HTML은 화면을 그리는 코드이기도 하지만, 브라우저와 도구들에게 "이 요소가 무엇인지" 알려주는 의미 계약이기도 하다.

## 직접 만들지 말라는 원칙

`Don't Roll Your Own ...` 원칙은 원래 암호화에서 자주 쓰이는 말이다.

뜻은 "개발자가 아무것도 만들지 말라"가 아니라, 이미 검증된 기능을 어설프게 다시 구현해서 안전장치와 생태계의 도움을 깨지 말라는 쪽에 가깝다.

암호화에서는 직접 만든 알고리즘이나 자체 구현이 작은 실수 하나로 보안 전체를 무너뜨릴 수 있다. 그래서 실무에서는 TLS, 검증된 암호화 라이브러리, bcrypt, argon2 같은 도구를 사용한다.

웹 UI에서도 비슷한 문제가 생긴다. 브라우저는 이미 링크, 버튼, 폼, 스크롤, 텍스트 선택, 복사/붙여넣기, 비밀번호 입력, 날짜 입력 같은 기본 기능을 오래 다듬어 왔다. 이 기능을 직접 대체하면 겉보기는 비슷해도 브라우저가 제공하던 기능이 깨질 수 있다.

## 비밀번호 입력 예시

브라우저는 화면에 점이 보인다고 해서 비밀번호 필드라고 판단하지 않는다.

브라우저는 HTML 요소와 속성을 보고 의미를 판단한다.

```html
<input type="password" name="password">
```

이렇게 작성하면 브라우저는 이 입력칸을 비밀번호 필드로 인식한다.

그 결과 다음 기능들과 협력할 수 있다.

- 비밀번호 마스킹
- 비밀번호 저장과 자동완성
- 강한 비밀번호 생성
- 비밀번호 관리자 연동
- 모바일 키보드 최적화
- 접근성 도구 연동
- 안전하지 않은 연결에서의 경고

반대로 다음처럼 만들면 사람 눈에는 비밀번호처럼 보일 수 있어도 브라우저 입장에서는 일반 텍스트 입력칸이다.

```html
<input type="text" name="password">
```

여기에 CSS나 JavaScript로 글자를 점처럼 보이게 만들어도, HTML 의미는 여전히 `text`다.

즉 브라우저는 "점으로 보이네, 비밀번호구나"라고 판단하지 않는다. 브라우저는 `type="password"`라는 의미 정보를 보고 비밀번호라고 판단한다.

## HTML과 브라우저 내부 동작의 관계

브라우저 내부 코드는 HTML 표준에 정의된 태그와 속성을 기준으로 다르게 동작한다.

예를 들어 다음 요소들은 브라우저에게 각자 다른 의미를 전달한다.

```html
<a href="/posts/1">글 보기</a>
<button type="submit">저장</button>
<input type="password">
<input type="date">
```

`<a href>`는 링크 탐색이라는 의미를 가진다. 그래서 브라우저는 새 탭 열기, 링크 주소 복사, 방문 기록, 뒤로 가기, 로딩 표시 같은 기능을 제공할 수 있다.

`<button>`은 버튼이라는 의미를 가진다. 그래서 키보드 조작, 폼 제출, 접근성 도구와 자연스럽게 연결된다.

`<input type="password">`는 민감한 비밀번호 입력이라는 의미를 가진다. 그래서 브라우저와 비밀번호 관리자가 이 필드를 특별하게 다룰 수 있다.

## 백엔드 관점 비유

백엔드에서 HTTP 메서드를 의미 없이 쓰면 생태계와 어긋난다.

```http
GET /users
```

이 요청은 서버, 브라우저, 프록시, 캐시가 보통 "조회 요청"으로 이해한다.

그런데 삭제를 하면서 다음처럼 만들면 문제가 생긴다.

```http
GET /delete-user?id=1
```

사람은 URL을 보고 의도를 추측할 수 있지만, HTTP 생태계 입장에서는 여전히 `GET` 요청이다. 캐시, 크롤러, 프록시가 잘못된 방식으로 다룰 수 있다.

HTML도 비슷하다.

```html
<input type="password">
```

는 브라우저 생태계에 "이건 비밀번호다"라고 알려주는 약속이다.

```html
<input type="text">
```

에 CSS로 점만 찍는 것은 사람 눈에만 비밀번호처럼 보이게 하는 것이다. 브라우저 입장에서는 의미가 일반 텍스트 입력칸으로 남아 있다.

## 링크 탐색과 JavaScript 가로채기

링크 탐색은 브라우저가 가장 기본적으로 잘 처리하는 기능 중 하나다.

```html
<a href="/some/file">파일 보기</a>
```

사용자가 이런 링크를 클릭하면 브라우저는 해당 URL로 이동하고, 로딩 상태를 보여주고, 방문 기록을 남기고, 뒤로 가기와 새 탭 열기 같은 기능을 자연스럽게 제공한다.

그런데 GitHub 같은 복잡한 웹앱은 링크 클릭을 브라우저에 바로 맡기지 않고 JavaScript가 먼저 처리하는 경우가 있다.

개념적으로는 다음과 비슷하다.

```js
link.addEventListener("click", (event) => {
  event.preventDefault();

  loadPageWithJavaScript(event.target.href);
});
```

여기서 `event.preventDefault()`는 브라우저의 기본 링크 이동을 막는다.

그다음 JavaScript가 직접 다음 일을 처리한다.

- 클릭한 링크가 파일인지, 이슈인지, PR인지 판단한다.
- 필요한 데이터를 API로 가져온다.
- 현재 화면 일부를 바꾼다.
- URL만 변경한다.
- 기존 화면 상태나 캐시를 유지한다.
- 로딩 UI와 화면 전환을 직접 처리한다.

이 방식은 잘 만들면 전체 페이지를 새로 로딩하지 않고 필요한 부분만 바꿀 수 있어서 빠르고 부드럽게 느껴질 수 있다. 흔히 말하는 클라이언트 사이드 라우팅이나 SPA 라우팅이 이런 흐름을 사용한다.

하지만 구현이 무겁거나 네트워크 상황이 좋지 않으면 브라우저 기본 링크 이동보다 느릴 수도 있다.

현재 탭에서 링크를 클릭하면 다음 흐름을 기다릴 수 있다.

```text
클릭 발생
→ JavaScript가 클릭 이벤트를 가로챔
→ 내부 라우팅 로직 실행
→ 필요한 데이터 요청
→ 응답 가공
→ 화면 일부 업데이트
→ URL 변경
```

반대로 새 탭으로 열면 브라우저가 일반적인 문서 로딩을 바로 시작할 수 있다.

```text
새 탭 생성
→ 해당 URL 요청
→ 서버 응답 수신
→ 새 페이지 렌더링
```

그래서 GitHub 같은 사이트에서는 현재 탭에서 JavaScript 라우팅 처리를 기다리는 것보다, 링크를 새 탭으로 여는 편이 더 빠르게 느껴질 때가 있다.

개발자도구에서 `Event Listener Breakpoints > Mouse > click`을 켜고 GitHub 링크를 클릭하면, 클릭 순간 JavaScript 코드에서 디버거가 멈출 수 있다. 이것은 링크 클릭을 브라우저가 바로 처리하는 것이 아니라 사이트의 JavaScript 코드가 중간에서 처리하고 있다는 것을 확인하는 방법이다.

핵심은 링크를 JavaScript로 직접 처리하면 안 된다는 것이 아니다.

다만 단순한 링크 이동까지 직접 처리하려면, 브라우저가 원래 제공하던 새 탭 열기, 뒤로 가기, 주소 복사, 로딩 표시, 방문 기록, 접근성 같은 기본 기대를 깨지 않는지 확인해야 한다.

## 실무에서 가져갈 기준

기본 기능을 대체하기 전에 먼저 생각해야 한다.

- 단순 링크라면 `<a href>`를 쓴다.
- 클릭 가능한 명령이라면 `<button>`을 쓴다.
- 비밀번호 입력은 `<input type="password">`를 기본으로 쓴다.
- 날짜 입력은 네이티브 `<input type="date">`를 먼저 검토한다.
- 커스텀 UI가 필요하더라도 키보드 조작, 접근성, 모바일 입력, 자동완성, 복사/붙여넣기, 새 탭 열기 같은 기본 기대를 깨지 않는지 확인한다.
- 링크를 JavaScript로 가로채야 한다면, 기본 `<a href>`의 장점을 유지하면서 실제로 더 나은 사용자 경험을 주는지 확인한다.

## 오늘의 깨달음

좋은 개발은 모든 것을 직접 구현하는 것이 아니라, 플랫폼이 이미 잘하는 일과 우리가 책임져야 할 일을 구분하는 것이다.

HTML의 의미를 제대로 쓰면 브라우저가 도와준다. 반대로 의미를 무시하고 겉모양만 직접 만들면 브라우저가 도와줄 근거가 사라진다.
