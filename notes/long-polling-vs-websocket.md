# 롱폴링과 WebSocket 비교

## 한 줄 정리

롱폴링은 HTTP 요청을 잠시 열어두었다가 데이터가 생기면 응답하는 방식이고, WebSocket은 연결을 계속 유지하면서 양방향으로 메시지를 주고받는 방식이다.

실시간 업데이트가 필요하다고 해서 항상 WebSocket이 정답은 아니다.

필요한 실시간성이 "서버 상태 변경을 클라이언트에게 비교적 빠르게 알려주는 정도"라면 롱폴링도 충분히 현실적인 선택이 될 수 있다.

## 숏폴링과 롱폴링의 차이

숏폴링은 클라이언트가 정해진 시간마다 서버에 계속 물어보는 방식이다.

```text
클라이언트: 새 데이터 있어?
서버: 없어.

3초 뒤
클라이언트: 새 데이터 있어?
서버: 없어.

3초 뒤
클라이언트: 새 데이터 있어?
서버: 있어.
```

구현은 단순하지만, 데이터가 없어도 요청과 응답이 계속 생긴다.

롱폴링은 클라이언트가 요청을 보내면 서버가 바로 응답하지 않고, 데이터가 생기거나 타임아웃이 될 때까지 기다리는 방식이다.

```text
클라이언트: 새 데이터 생기면 알려줘.
서버: 기다려볼게.

데이터 생김
서버: 생겼어. 여기 데이터.
```

데이터가 안 생기면 서버는 일정 시간 뒤 빈 응답을 준다.

```text
클라이언트: 새 데이터 생기면 알려줘.
서버: 기다려볼게.

30초 동안 데이터 없음
서버: 아직 없어.
클라이언트: 알겠어. 다시 기다릴게.
```

## 기차 비유

숏폴링은 시간표대로 출발하는 기차와 비슷하다.

승객이 있든 없든 정해진 시간마다 출발한다.

```text
3초마다 출발
승객 없음 -> 그래도 출발
승객 있음 -> 태우고 출발
```

롱폴링은 승객이 오면 바로 출발하는 기차와 비슷하다.

다만 영원히 기다리지는 않고, 정해진 시간 안에 승객이 안 오면 빈 상태로 출발한다.

```text
승객이 오면 바로 출발
승객이 안 오면 TTL 이후 빈 상태로 출발
```

여기서 TTL은 롱폴링 요청을 최대 몇 초까지 잡아둘지 정한 시간으로 보면 된다.

## 롱폴링은 요청을 오래 잡고 있어서 위험하지 않나?

위험 요소는 있다.

롱폴링은 요청 하나가 20초, 30초씩 살아 있을 수 있다.

사용자가 많아지면 동시에 대기 중인 HTTP 요청 수가 늘어난다.

```text
사용자 10,000명
각자 롱폴링 요청 1개 유지
=> 대기 중 요청 10,000개
```

특히 요청 하나가 서버 스레드 하나를 오래 점유하는 구조라면 위험해질 수 있다.

```text
요청 1개 = 서버 스레드 1개 점유
롱폴링 요청 500개 = 스레드 500개 대기
```

그러면 실제 일을 많이 하지 않는데도 스레드 풀이 꽉 차서 다른 API까지 느려질 수 있다.

그래서 롱폴링은 다음을 신경 써야 한다.

```text
- 요청 타임아웃을 둔다.
- 사용자당 열린 롱폴링 요청 수를 제한한다.
- 클라이언트 재시도에 backoff를 둔다.
- 서버, 프록시, 로드밸런서 타임아웃을 맞춘다.
- 가능하면 비동기 I/O 기반으로 처리한다.
```

정리하면 롱폴링은 아무 API 서버에 대충 붙이면 위험할 수 있다.

하지만 요청을 오래 잡아도 괜찮게 설계된 서버라면 실무에서도 사용할 수 있다.

## WebSocket과의 차이

WebSocket은 한 번 연결을 열고 계속 유지하는 방식이다.

```text
클라이언트 <-> 서버
열린 연결 위에서 양방향 메시지 교환
```

서버도 클라이언트에게 바로 메시지를 보낼 수 있고, 클라이언트도 서버에게 바로 메시지를 보낼 수 있다.

그래서 다음처럼 양방향 실시간성이 중요한 기능에 잘 맞는다.

```text
- 실시간 채팅
- 온라인 게임
- 공동 편집
- 매우 잦은 상태 업데이트
```

반면 롱폴링은 HTTP 요청/응답 모델을 유지한다.

서버가 응답을 조금 늦게 줄 뿐, 겉보기에는 일반 HTTP API와 비슷하다.

```text
GET /events
서버가 이벤트가 생길 때까지 응답을 지연
```

## WebSocket 대신 롱폴링을 선택할 수 있는 이유

WebSocket은 기능적으로 강력하지만 운영 환경에서는 더 까다로울 수 있다.

예를 들어 다음 문제가 생길 수 있다.

```text
- 프록시가 WebSocket 업그레이드를 막을 수 있다.
- 회사 방화벽이 WebSocket 연결을 제한할 수 있다.
- 로드밸런서 설정을 따로 맞춰야 할 수 있다.
- 서버 배포나 스케일링 때 연결 관리가 복잡해질 수 있다.
- 인증, 세션 만료, 재연결 처리를 별도로 신경 써야 한다.
```

롱폴링은 기존 HTTP 인프라를 그대로 활용하기 쉽다.

```text
- 기존 API 인증 구조를 쓰기 쉽다.
- 기존 프록시와 로드밸런서 구성을 크게 바꾸지 않아도 된다.
- 방화벽이나 조직 인프라와 충돌할 가능성이 상대적으로 낮다.
- 클라이언트 구현도 WebSocket보다 단순할 수 있다.
```

따라서 필요한 기능이 완전한 양방향 실시간 통신이 아니라 "상태 변경을 비교적 빠르게 알려주는 것"이라면 롱폴링이 더 단순하고 호환성 높은 선택이 될 수 있다.

## 롱폴링 구현 예시: 작업 상태 조회

`getJobStatusSync` 같은 함수는 롱폴링이 서버 안에서 실제로 어떻게 동작하는지 보여주는 중요한 예시다.

핵심 흐름은 다음과 같다.

```text
클라이언트가 작업 상태를 물어봄
-> 서버가 바로 "아직 처리 중"이라고 끝내지 않음
-> 서버가 최대 ttl 시간 동안 DB를 계속 확인함
-> 작업이 끝나면 바로 응답
-> 끝까지 안 끝나면 타임아웃 에러
```

예를 들어 이미지 생성, 배치 작업, AI 실행 작업처럼 시간이 걸리는 작업이 있다고 하자.

```text
POST /jobs
-> jobId 발급

GET /jobs/{jobId}/status
-> 작업 상태 조회
```

일반적인 상태 조회 API라면 작업이 아직 끝나지 않았을 때 바로 응답할 수 있다.

```json
{
  "status": "running"
}
```

하지만 롱폴링에서는 서버가 바로 응답하지 않고 잠깐 기다린다.

```text
0ms: DB 조회 -> running
500ms: DB 조회 -> running
1000ms: DB 조회 -> running
1500ms: DB 조회 -> success
응답 반환
```

즉, `getJobStatusSync`는 특정 `jobId`의 상태가 끝날 때까지, 또는 `ttl`이 지날 때까지 500ms마다 DB를 확인하는 함수라고 볼 수 있다.

```text
getJobStatusSync =
특정 작업 상태를
정해진 시간 동안
반복해서 확인하다가
확정 상태가 되면 결과를 반환하는 함수
```

## 클라이언트 폴링과 서버 내부 반복 조회의 차이

이 구현에서 중요한 점은 클라이언트가 계속 요청하는 대신, 서버가 요청을 잡고 내부에서 반복 확인한다는 것이다.

```text
숏폴링:
클라이언트가 500ms마다 서버 호출

롱폴링:
클라이언트 요청 1번
서버가 500ms마다 내부에서 DB 조회
```

겉으로 보면 클라이언트 요청 수는 줄어든다.

하지만 서버 안에서는 여전히 상태 변화를 확인하기 위한 반복 작업이 일어난다.

그래서 롱폴링은 클라이언트와 네트워크 요청 부담을 줄일 수 있지만, 서버와 DB 쪽 부하는 따로 봐야 한다.

## ttl이 중요한 이유

롱폴링 요청은 무한정 기다리면 안 된다.

작업이 언제 끝날지 모르는 요청이 서버에 계속 남으면 서버 자원을 오래 붙잡게 된다.

```text
ttl 없음:
끝나지 않는 요청이 계속 서버에 남을 수 있음

ttl 있음:
최대 30초까지만 기다리고 요청을 정리함
```

그래서 롱폴링 구현에는 보통 타임아웃 시간이 필요하다.

작업이 `success` 또는 `failure`가 되면 바로 응답하고, `ttl`이 지나도 확정되지 않으면 타임아웃으로 끝낸다.

```text
success -> 결과 반환
failure -> 실패 결과 반환
ttl 초과 -> 타임아웃 에러
```

## 500ms 간격의 의미

500ms마다 DB를 조회한다는 것은 실시간성과 DB 부하 사이의 타협이다.

간격이 짧으면 작업 완료를 빠르게 알아차릴 수 있다.

하지만 동시 요청이 많으면 DB 조회 수도 빠르게 늘어난다.

```text
동시 롱폴링 요청 1,000개
각 요청이 500ms마다 DB 조회
=> 초당 약 2,000번 조회 가능
```

그래서 500ms라는 숫자는 그냥 디테일이 아니라 성능과 부하에 직접 영향을 주는 설정이다.

이런 구현을 보면 다음 질문을 같이 떠올려야 한다.

```text
- DB를 500ms마다 계속 조회해도 괜찮은가?
- 동시 요청이 많아지면 DB 부하가 커지지 않는가?
- ttl은 몇 초가 적당한가?
- success/failure 말고 cancelled 같은 상태는 없는가?
- DB 반복 조회 대신 이벤트 기반으로 깨우는 방식은 가능한가?
```

오늘 기준으로 기억할 포인트는 이것이다.

```text
롱폴링은 클라이언트가 계속 물어보는 대신,
서버가 요청을 잡고 기다리며 상태를 확인하는 방식이다.

그래서 클라이언트 요청 수는 줄일 수 있지만,
서버와 DB가 감당해야 하는 대기 요청과 반복 조회 비용을 반드시 봐야 한다.
```

## 롱폴링 구현 시 주의사항

롱폴링은 단순해 보이지만, 대충 구현하면 서버, DB, 인프라에 부담을 줄 수 있다.

핵심은 얼마나 오래 요청을 잡을지와 얼마나 자주 상태를 확인할지를 반드시 제한하는 것이다.

```text
롱폴링은 요청을 오래 잡는 방식이다.
따라서 최대 대기 시간과 반복 조회 간격을 정해야 한다.
```

### TTL 설정

TTL은 서버가 롱폴링 요청을 최대 몇 초까지 붙잡아둘지 정한 시간이다.

클라이언트가 요청한 TTL을 그대로 믿으면 안 된다.

클라이언트가 실수하거나 악의적으로 너무 큰 값을 보낼 수 있기 때문이다.

```text
ttl=999999
```

이런 요청을 그대로 받아들이면 서버 요청이 너무 오래 살아남을 수 있다.

그래서 서버는 반드시 최대 TTL을 강제해야 한다.

```text
serverMaxTTL = 30초
clientTTL = 요청으로 받은 ttl

실제 TTL = min(clientTTL, serverMaxTTL)
```

예를 들어 클라이언트가 5초를 요청하면 5초만 기다리고, 999초를 요청해도 서버는 최대 30초까지만 기다리는 식이다.

### 인프라 타임아웃 고려

롱폴링 요청은 서버에 바로 도착하는 것이 아니라 여러 장비를 지나간다.

```text
브라우저
-> 회사 프록시
-> CDN/엣지 서버
-> 로드밸런서
-> API 서버
```

각 장비에는 응답이 너무 오래 없으면 연결을 끊는 타임아웃 설정이 있을 수 있다.

예를 들어 다음처럼 설정되어 있다고 하자.

```text
로드밸런서 idle timeout: 60초
서버 롱폴링 TTL: 90초
```

이 경우 서버는 90초까지 기다리려고 하지만, 로드밸런서가 60초에 먼저 연결을 끊을 수 있다.

그러면 클라이언트 입장에서는 정상 타임아웃 응답이 아니라 네트워크 에러처럼 보일 수 있다.

그래서 서버의 롱폴링 TTL은 인프라 타임아웃보다 충분히 짧아야 한다.

```text
로드밸런서 timeout: 60초
서버 롱폴링 TTL: 25~30초
```

핵심은 TTL을 서버 코드만 보고 정하면 안 된다는 점이다.

로드밸런서, 프록시, 엣지 서버의 타임아웃보다 짧게 잡아야 서버가 먼저 정상 응답을 주고 요청을 정리할 수 있다.

### DB 폴링 간격

롱폴링은 클라이언트 요청 수를 줄일 수 있지만, 서버 내부에서는 상태를 확인해야 한다.

```text
0ms: DB 조회
500ms: DB 조회
1000ms: DB 조회
1500ms: DB 조회
```

간격 없이 계속 조회하면 DB에 큰 부담이 된다.

```text
while not done:
  query DB
```

그래서 반복 조회 사이에 딜레이를 둔다.

```text
while not done:
  query DB
  wait 500ms
```

500ms는 상태 변화를 너무 늦게 감지하지 않으면서도 DB를 너무 자주 조회하지 않기 위한 타협값으로 볼 수 있다.

하지만 동시 요청이 많으면 500ms도 큰 부하가 될 수 있다.

```text
동시 롱폴링 요청 1,000개
각 요청이 500ms마다 DB 조회
=> 초당 약 2,000번 DB 조회
```

따라서 DB 폴링 간격은 단순한 숫자가 아니라 성능과 부하를 좌우하는 설정이다.

### 백오프 전략

백오프는 시간이 지날수록 폴링 간격을 점점 늘리는 전략이다.

```text
0.5초 뒤 확인
1초 뒤 확인
2초 뒤 확인
4초 뒤 확인
```

작업이 바로 끝날 수도 있는 초반에는 자주 확인하고, 오래 걸리는 작업으로 보이면 점점 덜 자주 확인하는 방식이다.

예를 들어 영상 변환이나 긴 AI 작업처럼 오래 걸릴 가능성이 큰 작업을 500ms마다 계속 확인하면 낭비가 될 수 있다.

```text
초반:
빠르게 끝날 수도 있으니 자주 확인

시간이 오래 지나면:
아직도 안 끝났으면 긴 작업일 가능성이 높음
-> 확인 간격을 늘림
```

다만 백오프를 너무 크게 잡으면 완료 반영이 늦어질 수 있다.

```text
작업은 끝났는데 다음 확인이 10초 뒤
-> 사용자는 10초 늦게 알게 됨
```

그래서 백오프도 trade-off다.

```text
간격 짧음:
빠르게 반영
DB 부하 큼

간격 김:
DB 부하 작음
상태 반영 늦음
```

정리하면 롱폴링 구현 시 주의사항은 다음과 같다.

```text
TTL:
요청을 최대 얼마나 오래 잡을지 제한

인프라 타임아웃:
중간 장비가 먼저 끊지 않도록 TTL을 더 짧게 설정

DB 폴링 간격:
상태 확인을 너무 자주 하지 않도록 딜레이 추가

백오프:
오래 걸리는 작업일수록 점점 덜 자주 확인해서 부하 절약
```

백엔드 관점에서 가장 중요한 감각은 이것이다.

```text
롱폴링은 HTTP라서 단순하지만,
오래 기다리는 요청과 반복 DB 조회를 다루는 기능이다.

그래서 반드시 상한과 간격을 둬야 한다.
```

## 롱폴링의 운영상 이점

롱폴링의 장점은 WebSocket보다 성능이 항상 좋다는 뜻이 아니다.

핵심은 WebSocket을 새로 운영하는 것보다 기존 HTTP 시스템 위에 얹기 쉽다는 점이다.

```text
롱폴링은 HTTP의 모양을 유지한다.
그래서 기존 HTTP 인증, 로깅, 모니터링, 프록시, 로드밸런서, 클라이언트 코드를 재사용하기 쉽다.
```

### 모니터링 유지가 쉬운 이유

롱폴링은 결국 HTTP 요청이다.

```text
GET /job-status?jobId=123
```

그래서 기존 API 서버에서 쓰던 로깅과 모니터링 체계를 그대로 활용하기 쉽다.

예를 들어 기존 백엔드 모니터링은 보통 이런 지표를 본다.

```text
- 어떤 URL이 몇 번 호출됐는지
- 응답 시간이 몇 ms인지
- 4xx/5xx 에러가 얼마나 나는지
- 특정 API가 느려졌는지
- access log
- APM trace
```

롱폴링도 HTTP 요청이기 때문에 이런 지표에 자연스럽게 잡힌다.

반면 WebSocket은 처음 연결은 HTTP처럼 시작하지만, 연결이 업그레이드된 뒤에는 일반적인 요청/응답 단위가 아니다.

```text
HTTP:
요청 1번 -> 응답 1번 -> 로그 1줄

WebSocket:
연결 1번 열림 -> 그 안에서 메시지 여러 번 오감
```

그래서 WebSocket은 메시지 하나하나를 어떻게 로깅하고 추적할지 별도로 고민해야 한다.

### 인증이 단순한 이유

롱폴링은 일반 HTTP API처럼 기존 인증 방식을 그대로 쓰기 쉽다.

```http
GET /events HTTP/1.1
Authorization: Bearer token
Cookie: sessionId=...
```

서버는 평소 API처럼 토큰이나 쿠키를 검사하면 된다.

WebSocket도 인증을 할 수는 있지만, 연결을 오래 유지하기 때문에 고민이 늘어난다.

```text
- 연결할 때 토큰을 어디에 실을지
- 연결 중 토큰이 만료되면 어떻게 할지
- 권한이 바뀌면 기존 연결을 끊을지
- 재연결할 때 인증을 다시 어떻게 할지
```

롱폴링은 요청이 계속 새로 만들어지므로 매 요청마다 기존 HTTP 인증을 적용하면 된다.

### 인프라 호환성이 좋은 이유

롱폴링은 프록시, 방화벽, 로드밸런서 입장에서 일반 HTTP 요청에 가깝다.

```text
브라우저
-> 회사 방화벽
-> 프록시
-> 로드밸런서
-> 서버
```

이 경로를 일반 API 요청처럼 통과할 수 있다.

WebSocket은 HTTP에서 시작하지만 중간에 연결 업그레이드를 한다.

```text
HTTP 연결을 WebSocket 연결로 바꿔줘
```

이때 중간 장비가 WebSocket 업그레이드를 허용해야 한다.

조직 환경에 따라 다음 문제가 생길 수 있다.

```text
- 회사 프록시가 WebSocket을 막을 수 있다.
- 로드밸런서의 WebSocket idle timeout이 짧을 수 있다.
- 방화벽 정책이 긴 연결을 제한할 수 있다.
- reverse proxy 설정을 따로 해야 할 수 있다.
```

그래서 인프라 제약이 있는 환경에서는 롱폴링이 더 호환성 높은 선택이 될 수 있다.

### 운영이 단순한 이유

롱폴링은 요청을 잠시 오래 잡기는 하지만, 결국 요청/응답으로 끝난다.

```text
요청 들어옴
-> 최대 ttl 동안 기다림
-> 응답하고 끝남
```

서버를 재시작하면 열려 있던 롱폴링 요청은 실패할 수 있다.

하지만 클라이언트가 다시 요청하면 된다.

```text
요청 실패
-> 잠깐 기다림
-> 다시 GET /events
```

WebSocket은 연결 자체가 중요한 상태가 될 수 있다.

예를 들어 이런 것을 관리해야 할 수 있다.

```text
- 사용자 A가 어느 서버에 연결되어 있는가?
- 어떤 채널을 구독 중인가?
- 서버 재시작 시 구독 상태를 어떻게 복구할 것인가?
- 서버를 여러 대로 늘리면 메시지를 어떻게 브로드캐스트할 것인가?
```

그래서 WebSocket은 연결 관리, 재연결, 서버 간 메시지 전달 같은 운영 고민이 늘어난다.

### 클라이언트 구현이 간단한 이유

롱폴링 클라이언트는 기본적으로 HTTP 요청 반복이다.

```text
1. GET /events 요청
2. 응답 받음
3. 데이터가 있으면 화면 갱신
4. 다시 GET /events 요청
5. 실패하면 조금 기다렸다 재시도
```

브라우저에서는 `fetch`로 구현할 수 있고, 모바일이나 서버 클라이언트도 일반 HTTP 클라이언트를 사용할 수 있다.

WebSocket은 연결 생명주기를 따로 다뤄야 한다.

```text
- 연결 열기
- 메시지 수신
- 연결 끊김 감지
- 재연결
- ping/pong 또는 heartbeat
- 중복 메시지 처리
- 연결 중 인증 만료 처리
```

라이브러리를 쓰면 편해질 수 있지만, HTTP 요청 하나를 반복하는 롱폴링보다는 챙길 것이 많다.

정리하면 롱폴링의 운영상 이점은 다음 한 문장으로 볼 수 있다.

```text
롱폴링은 성능 최강 선택이라기보다,
기존 HTTP 운영 체계 안에서 실시간 비슷한 경험을 만들 수 있는 현실적인 선택지다.
```

## Raw Long Polling을 선택한 이유

작성자가 말하는 핵심은 "WebSocket이 어려워서 롱폴링을 썼다"가 아니다.

더 정확히는 메시지 전달이 제품의 핵심 기능이라서, 라이브러리나 외부 시스템에 맡기지 않고 직접 통제 가능한 Raw Long Polling을 선택했다는 뜻이다.

```text
메시지 전달이 제품 핵심이다.
-> 동작을 완전히 이해하고 제어해야 한다.
-> 외부 의존성을 줄이고 직접 구현한다.
-> 그래서 단순한 HTTP Long Polling을 선택한다.
```

### 메시지 전달은 단순 구현 세부사항이 아니다

메시지 전달은 단순한 기술 부품이 아니라 제품 경험 자체일 수 있다.

예를 들어 제품이 작업 실행 상태, 채팅 상태, 실행 로그, 응답 진행 상황을 실시간 비슷하게 보여준다고 하자.

이때 메시지 전달이 흔들리면 사용자는 바로 제품이 이상하다고 느낀다.

```text
작업은 끝났는데 화면이 안 바뀜
실패했는데 계속 실행 중으로 보임
채팅 메시지가 늦게 뜸
사용자가 새로고침해야 상태가 맞음
```

이런 문제는 단순한 인프라 문제가 아니라 제품 품질 문제가 된다.

그래서 작성자는 메시지 전달 메커니즘을 제품의 핵심 요소로 본다.

### 핵심 기능을 타사 라이브러리에 맡기지 않으려는 이유

라이브러리를 쓰면 빠르게 만들 수 있다.

하지만 핵심 동작이 라이브러리의 설계와 정책에 묶일 수 있다.

예를 들어 다음 동작을 제품 요구사항에 맞게 조정해야 할 수 있다.

```text
- 재연결 타이밍
- 타임아웃 처리
- 에러 처리 방식
- 메시지 중복 처리
- 서버 재시작 시 동작
- 클라이언트 fallback 방식
```

이런 부분이 라이브러리 내부 정책에 묶이면 제품이 원하는 방식으로 세밀하게 바꾸기 어려울 수 있다.

그래서 작성자는 "이건 제품 핵심이므로 내부 동작을 우리가 이해하고 제어해야 한다"고 판단한 것이다.

### 셀프 호스팅을 단순하게 만들기 위한 선택

셀프 호스팅 제품은 사용자가 자기 서버에 직접 설치해서 돌릴 수 있어야 한다.

메시지 전달을 위해 외부 시스템이 많이 필요해지면 설치와 운영 난이도가 올라간다.

단순한 구성은 다음과 같다.

```text
앱 서버
```

하지만 메시지 전달 구조가 복잡해지면 다음처럼 될 수 있다.

```text
앱 서버
+ Redis
+ WebSocket gateway
+ 별도 message broker
+ reverse proxy WebSocket 설정
```

이렇게 되면 사용자가 설치하고 운영하기 어려워진다.

그래서 외부 의존성을 줄이고 HTTP만으로 처리하는 구조는 셀프 호스팅 제품에 큰 장점이 될 수 있다.

### 저수준 제어가 필요한 이유

Raw Long Polling은 추상화된 라이브러리에 맡기지 않고 직접 구현하는 방식에 가깝다.

직접 구현하면 다음 세부 동작을 마음대로 조정할 수 있다.

```text
- 폴링 TTL
- 재시도 간격
- 동적 폴링 간격
- 에러 응답 형태
- 작업 상태별 응답 규칙
- owner 검증 방식
- jobId별 조회 방식
```

예를 들어 작업 상태에 따라 폴링 간격을 다르게 가져갈 수 있다.

```text
초반: 1초마다 확인
진행률 90% 이후: 300ms마다 확인
실패 후: 재시도 간격 늘림
```

이런 세밀한 제어는 범용 라이브러리보다 직접 구현에서 더 단순할 수 있다.

### 코드 단순성이 중요한 이유

코드 단순성은 기능이 적다는 뜻이 아니다.

코드베이스를 읽는 사람이 메시지 전달 흐름을 쉽게 이해하고 수정할 수 있게 만든다는 뜻이다.

셀프 호스팅이나 오픈소스 제품이라면 사용자가 코드를 직접 보거나 수정할 수 있다.

이때 메시지 전달이 복잡한 라이브러리, 브로커, 추상화 계층에 숨어 있으면 이해하기 어렵다.

Raw Long Polling은 흐름이 비교적 눈에 보인다.

```text
요청 받음
DB에서 상태 확인
끝났으면 응답
아직이면 기다림
ttl 지나면 타임아웃
```

백엔드 개발자가 보기에도 이 API가 어떻게 기다리고 언제 응답하는지 따라가기 쉽다.

### 이 선택의 핵심

이 문단은 WebSocket과 Long Polling만 비교한 것이 아니다.

제품 제어권, 외부 의존성, 셀프 호스팅, 코드 이해 가능성까지 본 선택이다.

```text
Raw Long Polling을 선택한 이유:
- 메시지 전달이 제품 핵심이라 직접 제어하고 싶음
- 외부 의존성을 줄여 셀프 호스팅을 쉽게 만들고 싶음
- 폴링 간격, TTL, 재시도 같은 세부 동작을 직접 조정하고 싶음
- 코드베이스를 사용자가 이해하고 수정하기 쉽게 유지하고 싶음
```

따라서 이 선택은 롱폴링이 WebSocket보다 항상 좋다는 뜻이 아니다.

이 제품의 요구사항에서는 단순한 HTTP 기반 직접 구현이 더 맞았다는 판단이다.

## 댓글 관점: 숏폴링이 더 안전할 수 있는 이유

WebSocket이나 SSE는 연결을 오래 유지하는 방식이다.

```text
클라이언트 ---- 계속 열린 연결 ---- 서버
```

하지만 현실에서는 연결이 언제든 끊길 수 있다.

```text
- 모바일 네트워크 전환
- 브라우저 탭 sleep
- 노트북 절전
- 회사 프록시 idle timeout
- 서버 재배포
- 로드밸런서 연결 종료
```

연결이 끊기면 클라이언트는 다시 연결해야 한다.

문제는 재연결 자체보다, 끊긴 동안 발생한 이벤트를 어떻게 복구할지다.

### 재연결 복구가 어려운 이유

예를 들어 서버가 다음 이벤트를 보낸다고 하자.

```text
1번: 작업 시작
2번: 로그 A
3번: 로그 B
4번: 성공
```

클라이언트가 2번까지 받고 연결이 끊겼다.

그 사이 서버에서는 3번과 4번이 발생했다.

재연결 후에는 다음 질문이 생긴다.

```text
- 클라이언트가 3번부터 다시 받아야 하나?
- 이미 받은 2번이 중복으로 오면 어떻게 하나?
- 4번 성공 이벤트만 받으면 3번 로그는 없어도 되나?
- 서버는 이벤트를 얼마나 오래 저장해야 하나?
```

이 문제를 제대로 처리하려면 보통 다음 장치들이 필요하다.

```text
- 이벤트 ID
- 마지막으로 받은 이벤트 위치 저장
- 재연결 시 Last-Event-ID 또는 cursor 전달
- 서버의 이벤트 버퍼/저장소
- 중복 이벤트 제거
- 순서 보장
- 오래된 이벤트를 못 찾았을 때 fallback
```

SSE는 `Last-Event-ID` 같은 표준 기능이 있지만, 그것만으로 자동 해결되는 것은 아니다.

서버가 해당 ID 이후의 이벤트를 다시 보낼 수 있도록 이벤트를 저장하고 있어야 한다.

WebSocket은 이런 복구 프로토콜을 직접 구현해야 하는 경우가 많다.

```text
클라이언트: 나 eventId=2까지 받았어. 3번부터 줘.
서버: 3번부터 다시 보내줄게.
```

이 설계가 없으면 재연결 후 이벤트 유실이나 중복이 생길 수 있다.

### 숏폴링의 실패 모델이 단순한 이유

숏폴링은 연결을 오래 유지하지 않는다.

정해진 간격마다 현재 상태를 다시 조회한다.

```text
3초마다 현재 상태 조회
GET /job-status/123
```

요청 하나가 실패해도 다음 요청이 현재 상태를 다시 가져오면 된다.

```text
3초: running
6초: 요청 실패
9초: success
```

중간 이벤트 하나를 놓쳐도 최종 상태만 중요하다면 큰 문제가 되지 않는다.

그래서 작업 상태 조회 같은 기능에서는 숏폴링이 오히려 안전할 수 있다.

```text
WebSocket/SSE 실패:
연결 끊김
-> 재연결
-> 어디까지 받았는지 복구
-> 놓친 이벤트 재전송
-> 중복 제거
-> 순서 보장

숏폴링 실패:
요청 한 번 실패
-> 다음 요청에서 현재 상태 다시 조회
```

여기서 "숏폴링이 더 안전하다"는 말은 성능이 더 좋다는 뜻이 아니다.

실패했을 때 복구 모델이 단순하다는 뜻에 가깝다.

### 이벤트 스트림이 필요한지, 현재 상태 조회면 충분한지

핵심은 모든 이벤트를 빠짐없이 받아야 하는 문제인지, 현재 상태만 알면 되는 문제인지 구분하는 것이다.

채팅 메시지처럼 모든 이벤트가 중요하다면 이벤트 유실이 치명적이다.

```text
메시지 A
메시지 B
메시지 C
```

중간의 메시지 B를 놓치면 안 된다.

이런 경우에는 WebSocket이나 SSE를 쓰더라도 재연결 복구 설계가 필요하다.

반면 작업 상태는 보통 현재 상태가 더 중요하다.

```text
queued -> running -> success
```

중간의 `running` 이벤트를 한 번 놓쳐도 나중에 `success` 상태를 조회하면 화면을 맞출 수 있다.

정리하면 다음과 같다.

```text
이벤트 하나하나가 중요함:
채팅 메시지, 주문 체결, 로그 스트림
-> WebSocket/SSE + 재연결 복구 설계 필요

현재 상태만 중요함:
작업 실행 상태, 배치 완료 여부, 생성 결과 준비 여부
-> 숏폴링/롱폴링이 더 단순하고 안전할 수 있음
```

댓글의 핵심은 WebSocket이나 SSE를 쓰지 말라는 뜻이 아니다.

재연결 복구까지 제대로 만들 자신이 없거나, 현재 상태 조회만으로 충분한 기능이면 숏폴링이 운영상 더 안전할 수 있다는 뜻이다.

## 반박 관점: 롱폴링이 hacky하게 느껴질 수 있는 이유

롱폴링에 대한 반박 중 하나는 HTTP 요청을 일부러 끝내지 않고 붙잡는 방식이 부자연스럽게 느껴질 수 있다는 점이다.

이 반박은 롱폴링이 기술적으로 불가능하다는 뜻이 아니다.

브라우저, 디버깅, 서버 대기 모델 관점에서 찝찝함이 생길 수 있다는 뜻이다.

### HTTP 요청/응답 모델과 살짝 어긋나는 느낌

보통 HTTP 요청은 다음 흐름으로 이해한다.

```text
요청 보냄
-> 서버가 처리함
-> 응답 받음
-> 끝
```

하지만 롱폴링은 일부러 응답을 늦춘다.

```text
요청 보냄
-> 서버가 바로 응답하지 않음
-> 이벤트가 생기거나 TTL이 끝날 때까지 기다림
-> 응답
```

그래서 HTTP 요청/응답 모델을 이용해 실시간처럼 보이게 만드는 우회 방식처럼 느껴질 수 있다.

이런 이유로 롱폴링이 `hacky`하게 보일 수 있다.

### 브라우저에서 Pending 요청으로 보일 수 있음

롱폴링 요청은 개발자 도구 Network 탭에서 오랫동안 `Pending` 상태로 보일 수 있다.

```text
GET /events      Pending...
GET /job/status  Pending...
```

기술적으로는 정상 동작이지만, 보는 사람 입장에서는 헷갈릴 수 있다.

```text
이거 요청이 멈춘 건가?
페이지 로딩이 덜 끝난 건가?
서버가 느린 건가?
```

특히 사용자가 브라우저 하단이나 탭 로딩 표시에서 계속 로딩 중처럼 느낀다면 UX가 나빠질 수 있다.

다만 `fetch`나 XHR 백그라운드 요청이 항상 페이지 전체 로딩 표시를 계속 돌리는 것은 아니다.

구현 방식에 따라 다르지만, 개발자 도구에서 Pending으로 보이는 것은 자연스러운 현상이다.

### 페이지 로딩과 롱폴링은 분리해야 함

사용자는 보통 로딩 중 상태를 다음처럼 해석한다.

```text
아직 페이지가 준비 안 됐나?
뭔가 덜 받아졌나?
인터넷이 느린가?
```

롱폴링 요청이 페이지 전체 로딩 상태와 연결되면 사용자는 불안해할 수 있다.

그래서 롱폴링은 초기 페이지 로딩과 분리해서 백그라운드 상태 업데이트로 다뤄야 한다.

```text
초기 페이지 로딩:
필요한 컨텐츠를 다 받은 뒤 완료

롱폴링:
백그라운드에서 상태 업데이트만 기다림
```

롱폴링 때문에 앱 전체가 계속 로딩 중처럼 보이면 설계가 좋지 않은 것이다.

### 서버에도 기다리는 요청이 생김

롱폴링은 서버 어딘가에 기다리는 요청을 만든다.

```text
요청 스레드 또는 async task
-> 이벤트나 timeout까지 대기
```

blocking 구조라면 요청이 서버 스레드를 오래 점유할 수 있다.

async/non-blocking 구조라 하더라도 대기 중인 요청이 많아지면 관리해야 할 자원이 생긴다.

그래서 롱폴링은 서버가 요청을 정상 처리하고 바로 끝내는 방식이 아니라, 일부러 붙잡고 있는 구조라는 어색함이 있다.

이것은 롱폴링의 본질적인 단점 중 하나다.

### 이 반박이 맞는 경우

다음 상황에서는 롱폴링이 어색하거나 부적절할 수 있다.

```text
- 요청이 Pending으로 보이는 것이 디버깅에 혼란을 줌
- 사용자에게 페이지가 계속 로딩 중처럼 보임
- 서버가 blocking 방식이라 요청 대기가 스레드를 잡아먹음
- 동시 접속자가 많아서 대기 요청 수가 크게 늘어남
- 양방향 실시간성이 진짜 필요함
```

따라서 롱폴링을 쓸 때는 다음을 신경 써야 한다.

```text
- 페이지 초기 로딩과 롱폴링 요청을 분리
- 백그라운드 상태 업데이트 요청으로 다루기
- 짧은 TTL로 주기적으로 정상 종료되게 만들기
- 서버에서는 async/non-blocking 방식으로 대기 처리
- 모니터링에서 롱폴링 endpoint의 긴 응답시간을 별도로 해석
```

### 모니터링에서도 별도 해석이 필요함

롱폴링 endpoint는 응답시간이 긴 것이 정상일 수 있다.

```text
일반 API 응답시간 10초:
문제일 가능성 큼

롱폴링 API 응답시간 10초:
이벤트가 없어서 기다린 정상 동작일 수 있음
```

그래서 일반 API와 같은 기준으로 응답시간을 해석하면 안 된다.

롱폴링 endpoint는 긴 응답시간, Pending 요청, timeout 응답이 정상 동작의 일부일 수 있다는 점을 팀과 운영 도구가 이해해야 한다.

이 반박의 핵심은 다음과 같다.

```text
롱폴링은 HTTP 요청을 일부러 오래 열어두는 방식이라
브라우저, 디버깅, 서버 대기 모델 관점에서 어색하고 hacky하게 느껴질 수 있다.

따라서 페이지 로딩과 분리하고,
서버 대기를 안전하게 처리하고,
모니터링에서도 별도 기준으로 봐야 한다.
```

## 오늘 기준으로 기억할 결론

```text
숏폴링:
정해진 시간마다 계속 물어본다.
단순하지만 낭비가 많다.

롱폴링:
요청을 보내고 서버가 이벤트 생길 때까지 기다린다.
HTTP 기반이라 WebSocket보다 도입이 쉬울 수 있다.

WebSocket:
연결을 계속 열고 양방향 통신한다.
실시간성은 좋지만 인프라와 운영 복잡도가 올라갈 수 있다.
```

실시간 업데이트가 필요하다고 무조건 WebSocket을 선택할 필요는 없다.

요구사항이 서버 상태 변경 알림 중심이고, 조직 환경이나 인프라 제약이 있다면 롱폴링이 더 실용적인 선택일 수 있다.
