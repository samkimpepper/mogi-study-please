# 🐱 유니코드와 인코딩 — "냐옹" 으로 끝까지 따라가기

> 글자가 화면의 문자에서 → 디스크의 바이트까지 가는 전 과정을 "냐옹" 한 단어로 추적하는 노트.

---

## 🎯 큰 그림 — 사실 서로 다른 두 질문이다

헷갈리는 이유: **서로 다른 두 질문**을 "문자코드" 한 단어로 뭉쳐서 그렇다.

| 개념                 | 답하는 질문                       | 한마디     |
| ------------------ | ---------------------------- | ------- |
| **유니코드 (코드포인트)**   | "이 글자한테 무슨 **번호**를 줄까?"      | 번호 매기기  |
| **인코딩 (UTF-8/16)** | "그 번호를 실제 **바이트**로 어떻게 적을까?" | 바이트로 적기 |

➡️ **번호 매기기**와 **바이트로 적기**는 별개의 단계. 이걸 분리하는 게 전부다.

---

## 1️⃣ 유니코드 — 글자에 번호 붙이기

유니코드는 세상 모든 글자에 고유 번호(**코드포인트**)를 부여한 거대한 번호표. 표기는 `U+` 뒤에 16진수.

|글자|코드포인트|10진수|
|---|---|---|
|`냐`|**U+B0D0**|45264|
|`옹`|**U+C639**|50745|

이 단계에선 "냐는 45264번, 옹은 50745번" 이라는 **추상적 번호**만 정해졌다. 아직 컴퓨터에 _어떻게 저장_되는지는 안 정해진 상태.

> 🏛️ **비유**: 도서관에서 책마다 청구기호 붙이기. "이 책은 813.6번" 이라고 정한 거지, 어느 책장에 어떻게 꽂을지는 아직 안 정함.

---

## 2️⃣ 인코딩 — 번호를 바이트로 적기

번호를 디스크에 저장하려면 **바이트(0~255 숫자들의 나열)** 로 바꿔야 함. 그 변환 규칙이 인코딩. **같은 번호라도 규칙이 다르면 다른 바이트가 나온다.**

### UTF-8 로 적으면 (한글 보통 3바이트)

```
냐 (U+B0D0) → EB 83 90
옹 (U+C639) → EC 98 B9
```

➡️ "냐옹" = 총 **6바이트**: `EB 83 90 EC 98 B9`

### UTF-16 으로 적으면 (2바이트, 번호 거의 그대로)

```
냐 (U+B0D0) → B0 D0
옹 (U+C639) → C6 39
```

➡️ "냐옹" = 총 **4바이트**: `B0 D0 C6 39`

### 💥 핵심 포인트

> 똑같은 "냐옹", 똑같은 코드포인트(B0D0 / C639) 인데 **UTF-8 → 6바이트, UTF-16 → 4바이트.** **번호는 같고 바이트는 다르다.** 이게 "유니코드 ≠ 인코딩" 의 실체.

---

## 3️⃣ 읽을 때는 거꾸로 (+ 함정)

파일 읽기는 반대 방향: **바이트 → 번호 → 글자**

⚠️ **함정**: 디스크엔 바이트만 있고 "나 UTF-8이야" 라는 꼬리표가 안 붙는다. 그래서 **어떤 인코딩으로 읽을지 잘못 고르면** 글자가 깨진다.

`EB 83 90 EC 98 B9` (UTF-8로 적힌 "냐옹") 를 실수로 EUC-KR 같은 다른 인코딩으로 읽으면 → 엉뚱한 글자.

> 이게 흔히 보는 `ë\x83`... 같은 **글자 깨짐 (mojibake)** 의 정체. 바이트는 멀쩡한데 **해석 규칙을 틀린** 것.

---

## 4️⃣ surrogate 로 연결 (이슈 #61670 과 이어짐)

"냐옹" 은 번호가 작아서(U+B0D0, U+C639) UTF-16 의 2바이트 한 칸에 쏙 들어간다.

근데 이모지처럼 번호가 큰 글자는 한 칸을 넘침:

```
🐱 = U+1F431 (10진수 128049) → 2바이트 한 칸 초과
```

그래서 UTF-16 에서 **두 조각(surrogate pair)** 으로 쪼갠다:

- **high surrogate** (앞조각, `\uD800`~`\uDBFF`)
- **low surrogate** (뒷조각)
- ⚠️ 이 둘은 **반드시 짝**

만약 "냐옹🐱" 을 쓰는데 🐱 의 **앞조각만 남고 뒷조각이 사라지면**:

➡️ **외톨이 high surrogate** = 이슈 #61670 의 `\uD83A` 상황! ➡️ 유효하지 않은 유니코드 → JSON 파서가 400 으로 거부 → 세션 brick

> 🔗 관련 노트: 외톨이 surrogate 버그 (Claude Code #61670)

## 🧾 정리 — "냐옹" 으로 한눈에

|층위|"냐옹" 의 경우|설명|
|---|---|---|
|**유니코드 (번호)**|냐=U+B0D0, 옹=U+C639|글자에 매긴 추상 번호|
|**인코딩 (바이트)**|UTF-8 → 6바이트 / UTF-16 → 4바이트|같은 번호의 다른 저장 방식|
|**깨짐 ①: mojibake**|UTF-8 바이트를 EUC-KR로 읽음|번호 _해석_ 을 틀림|
|**깨짐 ②: surrogate**|🐱 의 조각이 짝 잃음|번호 _자체_ 가 불완전|

## 🐍 직접 확인 (파이썬 한 줄)

```python
"냐옹".encode("utf-8")        # b'\xeb\x83\x90\xec\x98\xb9'  (6바이트)
"냐옹".encode("utf-16-be")    # b'\xb0\xd0\xc69'             (4바이트)
[hex(ord(c)) for c in "냐옹"] # ['0xb0d0', '0xc639']         (코드포인트)
```

## 🤔 더 팔 거리

- UTF-8 이 왜 하필 한글을 3바이트로 쪼개는가 → 비트 패턴 규칙 (`1110xxxx 10xxxxxx 10xxxxxx`)
- BOM (Byte Order Mark) 과 엔디언 (UTF-16-BE vs LE)
- 정규화 (NFC / NFD) — 같은 글자가 코드포인트 조합이 다를 수 있음 (한글 자모 분리 등)
