# Boolean을 숫자로 더하는 검증 코드 안티패턴

## 한 줄 정리

검증 조건은 계산 문제처럼 보이게 만들기보다, 읽는 사람이 바로 의도를 알 수 있게 표현하는 것이 좋다.

## 문제 코드

```js
const invalid =
  Number(!hasName) +
  Number(!hasEmail) +
  Number(!isAdult);

if (invalid === 0) {
  // OK
}
```

이 코드는 동작 자체는 맞다.

- `!hasName`이 `true`면 이름이 없다는 뜻이다.
- `Number(true)`는 `1`이다.
- `Number(false)`는 `0`이다.
- 그래서 `invalid === 0`은 실패한 조건이 하나도 없다는 뜻이다.

하지만 읽는 사람 입장에서는 조건식을 바로 이해하지 못하고, boolean이 숫자로 바뀌는 과정을 머릿속으로 한 번 더 계산해야 한다.

## 왜 안티패턴처럼 보이나

검증 코드는 "무엇이 유효한지"가 바로 보여야 한다.

그런데 boolean을 숫자로 바꿔 더하면 코드의 중심이 조건의 의미가 아니라 계산 방식으로 이동한다.

```text
이름이 있는가?
이메일이 있는가?
성인인가?
```

를 보고 싶은데, 실제 코드는 다음 질문을 하게 만든다.

```text
true가 1이었나?
false가 0이었나?
invalid가 실패 개수인가?
0이면 성공인가?
```

즉 동작은 짧지만, 의도가 숨는다.

## 더 읽기 쉬운 표현

성공 조건만 필요하면 이렇게 쓰는 편이 낫다.

```js
const isValid = hasName && hasEmail && isAdult;

if (isValid) {
  // OK
}
```

또는 바로 조건문에 넣을 수도 있다.

```js
if (hasName && hasEmail && isAdult) {
  // OK
}
```

검증 실패 이유가 필요하다면 숫자 합보다 에러 목록이 더 명확하다.

```js
const errors = [];

if (!hasName) errors.push('name');
if (!hasEmail) errors.push('email');
if (!isAdult) errors.push('age');

if (errors.length === 0) {
  // OK
}
```

이 방식은 단순히 성공 여부만 보는 것이 아니라, 나중에 사용자에게 어떤 값이 빠졌는지 보여주기도 쉽다.

## 실무 기준

boolean을 숫자로 바꾸는 코드가 항상 틀린 것은 아니다. 예를 들어 통계나 점수 계산처럼 "개수" 자체가 목적이면 쓸 수 있다.

하지만 검증 로직에서는 보통 다음 기준이 더 중요하다.

- 조건의 의미가 이름으로 드러나는가?
- 성공 조건과 실패 조건이 바로 읽히는가?
- 나중에 검증 항목이 늘어났을 때 수정하기 쉬운가?
- 실패 이유를 보여줘야 할 때 자연스럽게 확장되는가?

> [!summary]
> 검증 코드는 짧은 계산식보다 의도가 드러나는 조건식이 더 낫다. `Number(boolean)`을 더하는 방식은 동작은 맞아도 읽는 사람이 의미를 한 번 더 해석해야 한다.
