# 직접 구현하지 말라는 말의 진짜 의미

## 한 줄 정리

직접 구현이 항상 나쁜 것은 아니지만, 플랫폼이나 생태계가 제공하던 안전장치를 깨면서 직접 구현하는 것은 위험하다.

## 직접 만들지 말라는 말이 헷갈리는 이유

`Don't Roll Your Own`은 "아무것도 직접 만들지 말라"는 뜻으로만 받아들이면 너무 거칠다.

실무에서는 직접 구현이 필요한 순간도 있다. 기존 도구가 너무 무겁거나, 문제에 맞지 않거나, 선택지가 너무 적을 수도 있다. 그래서 중요한 질문은 "직접 만들었는가?"가 아니라 "무엇을 직접 만들었고, 어떤 안전장치를 잃었는가?"에 가깝다.

> [!summary]
> 직접 구현 자체보다 위험한 것은 이미 검증된 의미, 프로토콜, 보안 경계, 생명주기 관리 방식을 어설프게 대체하는 것이다.

## 웹 UI에서의 예시

웹 폼에서는 브라우저에 기대는 것이 가장 단순하고 빠르며, 거의 항상 가장 일관적인 방법이다.

```html
<form method="post" action="/login">
  <input type="email" name="email" required>
  <input type="password" name="password" autocomplete="current-password">
  <button type="submit">로그인</button>
</form>
```

이렇게 만들면 브라우저가 폼 제출, Enter 키 처리, 필수값 검증, 비밀번호 관리자 연동, 자동완성, 모바일 키보드 최적화, 접근성 도구 연결을 도와준다.

반대로 단순 폼과 링크까지 JavaScript로 전부 대체하면 애플리케이션 코드가 브라우저의 기본 책임을 다시 떠안게 된다.

```text
클릭 처리
폼 상태 관리
검증 표시
라우팅
로딩 상태
뒤로 가기
새 탭 열기
접근성
```

여기서 위험한 것은 "자체 웹 프레임워크를 만들었다"는 사실 하나가 아니다. 브라우저와 HTML이 이미 제공하던 의미와 동작을 깨면서, 그 복잡성을 직접 책임지는 것이 위험하다.

> [!note]
> 브라우저 기본 폼과 링크를 그대로 살린다면, 서버 쪽의 얇은 라우터나 템플릿 렌더링 계층을 직접 만드는 것은 상대적으로 단순한 문제가 된다.

## 암호학에서의 예시

암호학에서는 검증된 알고리즘과 라이브러리를 쓰는 것이 보통 안전하다. 자체 암호 알고리즘이나 자체 프로토콜은 작은 실수 하나로 전체 보안이 무너질 수 있다.

하지만 "새 암호 코드는 내부 집단에 속한 사람만 작성할 수 있다"는 식으로 말하면 판단 기준이 흐려진다. 중요한 것은 학위나 소속 그 자체가 아니라, 코드와 설계가 공개 검증, 테스트, 리뷰, 공격 모델 분석을 견딜 수 있는지다.

> [!warning]
> 학습용 구현과 운영용 구현은 다르다. 운영에서 암호 코드를 다룰 때는 직접 구현의 필요성, 검증 방식, 실패 비용을 훨씬 엄격하게 봐야 한다.

## 메모리 안전 논쟁

메모리 안전하지 않은 언어에서 치명적 취약점이 많이 나오는 것은 맞다. 하지만 그 안에서도 특히 위험한 코딩 스타일이 있다.

대표적으로 작은 객체마다 `malloc()`이나 `new`를 남발하면 소유권과 생명주기 관리가 복잡해진다.

```c
User *user = malloc(sizeof(User));
Profile *profile = malloc(sizeof(Profile));
Session *session = malloc(sizeof(Session));
```

이 방식에서는 계속 이런 질문이 생긴다.

```text
누가 소유하지?
언제 해제하지?
다른 곳에서 아직 참조 중인가?
이미 해제한 뒤 다시 쓰는 건 아닌가?
두 번 해제하는 건 아닌가?
```

이 질문 하나라도 틀리면 `use-after-free`, `double free`, 메모리 누수 같은 문제가 생긴다.

반대로 생명주기가 짧고 명확한 데이터는 스택이나 아레나 방식으로 다룰 수 있다.

```text
요청 시작
→ request arena 생성
→ 요청 처리 중 필요한 작은 객체들을 arena에 할당
→ 응답 끝
→ arena 전체를 한 번에 해제
```

백엔드 관점으로 보면 요청 하나를 처리하는 동안 생긴 임시 DTO, 파싱 결과, 검증 중간값을 요청 스코프에 묶어두고 요청이 끝나면 한 번에 버리는 느낌이다.

> [!tip]
> 아레나는 개별 객체마다 `free`하는 부담을 줄여서 생명주기 실수를 줄인다. 대신 아레나보다 오래 살아야 하는 데이터를 잘못 넣으면 다른 종류의 버그가 생길 수 있다.

## 널 종료 문자열과 슬라이스

C 스타일 문자열은 보통 끝에 `\0`이 있어야 문자열의 끝을 알 수 있다.

```c
char name[5] = "mogi";
```

메모리에는 대략 이렇게 들어간다.

```text
m o g i \0
```

문제는 길이 정보가 문자열과 함께 다니지 않는다는 점이다. 문자열 함수는 `\0`을 만날 때까지 계속 읽는다.

```text
버퍼 크기: 5
복사하려는 문자열: "hello"
필요한 공간: h e l l o \0 = 6칸
```

이런 계산을 틀리면 버퍼 경계 밖을 읽거나 쓰는 문제가 생긴다.

슬라이스는 보통 시작 위치와 길이를 함께 들고 다니는 방식이다.

```text
ptr: 문자열 시작 위치
len: 문자열 길이
```

즉 널 종료 문자열이 "읽다가 `\0`이 나오면 끝"이라면, 슬라이스는 "여기서 시작해서 정확히 이 길이만큼"이라고 말하는 방식이다.

> [!note]
> 슬라이스는 경계를 명시적으로 들고 다니기 때문에 문자열과 버퍼 처리에서 실수할 여지를 줄인다.

## 실무 판단 기준

직접 구현을 볼 때는 다음처럼 나눠서 생각하면 좋다.

직접 구현해도 비교적 괜찮은 영역:

- 단순 라우터
- HTML 템플릿 렌더링
- 정적 파일 서빙
- 브라우저 기본 폼을 살린 서버 사이드 폼 처리
- 특정 프로젝트에 맞춘 얇은 접착 코드

직접 구현을 조심해야 하는 영역:

- 암호 알고리즘
- 비밀번호 해싱 방식
- 세션 토큰 설계
- CSRF 방어
- HTML escaping
- SQL escaping
- 인증과 인가 모델
- 파일 업로드 보안
- 신뢰할 수 없는 입력을 처리하는 파서
- 수동 메모리 관리가 필요한 고위험 코드

> [!summary]
> 플랫폼 기본 기능을 살리는 얇은 구현은 괜찮을 수 있다. 하지만 보안, 파싱, 인증, 암호, 메모리 관리처럼 실패 비용이 큰 영역은 직접 구현보다 검증된 도구와 설계를 먼저 봐야 한다.
