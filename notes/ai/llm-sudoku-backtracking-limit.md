# LLM이 스도쿠 같은 백트래킹 문제에 약한 이유

> [!summary]
> LLM은 솔버를 짜는 것은 잘할 수 있지만, 솔버 자체가 되는 것은 구조적으로 약하다.
>
> 스도쿠는 수정 가능한 작업 메모리와 백트래킹 루프가 필요한 문제인데, 트랜스포머의 기본 추론은 고정 깊이 forward pass에 가깝기 때문이다.

## 처음 헷갈렸던 질문

> [!question]
> LLM은 글도 쓰고 코드도 짜는데, 왜 스도쿠처럼 규칙이 명확한 문제에서는 이상하게 약해질까?
>
> 사람과 LLM 사이에 결정적인 구조 차이가 있는 걸까?

요즘 추론 모델은 스도쿠를 어느 정도 풀 수 있다. 그리고 어떤 LLM이든 파이썬으로 스도쿠 솔버를 짜서 실행하면 빠르게 풀 수 있다.

중요한 차이는 "어떻게 푸느냐"다.

```text
LLM이 직접 머릿속으로 풀기:
약함

오래 추론 과정을 텍스트로 펼쳐서 풀기:
조금 나아짐

파이썬 솔버를 짜서 실행하기:
강함
```

이 차이는 LLM 자체가 실행 엔진이라기보다 패턴과 언어를 다루는 모델이라는 점에서 나온다.

## 스도쿠가 요구하는 것

스도쿠는 두 가지를 강하게 요구한다.

```text
1. 수정 가능한 작업 메모리
2. 백트래킹 루프
```

사람은 종이에 숫자를 써보고 지우거나, 머릿속에서 후보를 바꾸면서 상태를 갱신한다.

```text
이 칸은 5인가?
-> 아니네
-> 지우고 7을 넣어보자
-> 모순이 생기면 이전 가정으로 돌아가자
```

스도쿠는 단순히 "그럴듯한 숫자"를 채우는 문제가 아니다. 81칸의 상태를 일관되게 유지하면서, 모순이 생기면 되돌아가야 한다.

## 수정 가능한 작업 메모리가 없다

트랜스포머 LLM의 기본 메모리는 주어진 컨텍스트와 이미 출력한 토큰이다.

그런데 이미 출력한 토큰은 사실상 append-only다.

```text
출력한 토큰:
뒤에 덧붙일 수 있음

이미 출력한 토큰:
진짜로 지우거나 수정할 수 없음
```

LLM이 "아까 5라고 했는데 7이 맞다"라고 말할 수는 있다.

하지만 그것은 기존 상태를 실제로 수정한 것이 아니라, 뒤에 정정 텍스트를 덧붙인 것이다.

> [!warning]
> LLM은 출력 스트림 안에서 정정할 수는 있지만, 일반 프로그램처럼 같은 변수나 격자 상태를 안정적으로 덮어쓰는 구조가 아니다.

스도쿠에서는 81칸짜리 격자와 각 칸의 후보 상태를 계속 갱신해야 한다.

```text
cell[3][4] = {2, 5, 7}
-> 5 제거
-> 7 확정
-> 같은 행/열/박스 후보 갱신
```

이런 작업은 mutable state가 있으면 자연스럽지만, LLM이 텍스트만으로 직접 유지하기에는 불안정하다.

## 백트래킹 루프가 없다

스도쿠 솔버의 핵심은 백트래킹이다.

```text
가정
-> 제약 전파
-> 모순 발견
-> 이전 상태로 되돌아감
-> 다른 값 시도
```

일반 코드로 쓰면 재귀나 `while` 루프가 들어간다.

```text
solve(board):
    빈 칸을 고른다
    가능한 숫자를 하나 넣는다
    모순이 없으면 다음 칸으로 간다
    막히면 이전 칸으로 돌아간다
```

하지만 트랜스포머의 한 번의 forward pass는 입력이 정해진 층을 통과해 출력 분포를 만드는 구조에 가깝다.

```text
input
-> layer1
-> layer2
-> ...
-> layerN
-> output
```

여기에는 일반 프로그램의 `while`, 재귀, mutable grid 같은 실행 구조가 기본적으로 들어 있지 않다.

> [!info]
> 백엔드 비유로 보면, 트랜스포머 한 번의 추론은 루프 금지, 변수 재할당 금지, 호출 깊이 고정인 순수 함수에 가깝다.
>
> 반면 스도쿠 솔버는 루프와 mutable state가 핵심이다.

## 토큰당 계산량이 고정된다

트랜스포머는 토큰 하나를 생성할 때 대체로 정해진 계산량을 쓴다.

쉬운 토큰이든 어려운 토큰이든, 같은 모델에서는 정해진 층을 통과한다.

```text
쉬운 토큰:
N개 layer 통과

어려운 토큰:
N개 layer 통과
```

단일 forward pass 안에서 "이 문제는 어려우니 루프를 더 돌자"처럼 계산 시간을 마음대로 늘리는 구조가 아니다.

> [!tip]
> 모든 요청에 CPU 시간이 똑같이 배정되는 서버라고 생각하면 된다.
>
> 어려운 작업이라고 단일 패스 안에서 더 오래 돌려주지 않는다.

LLM이 더 많이 계산하는 현실적인 방법은 토큰을 더 생성하는 것이다.

```text
더 많은 중간 추론 텍스트 생성
-> 더 많은 forward pass
-> 더 많은 계산
```

추론 모델이 오래 생각하는 것처럼 보이는 이유도 이 구조와 관련 있다. 내부에 진짜 mutable board와 루프를 가진다기보다, 텍스트 스트림으로 중간 과정을 외부화하면서 부족한 작업 메모리와 반복을 흉내 내는 것이다.

## 왜 스도쿠가 함정인가

LLM이 잘하는 많은 작업은 부분적으로 틀려도 그럴듯하게 보일 수 있다.

```text
글쓰기:
조금 어색해도 읽힘

코드 초안:
조금 틀려도 테스트와 리뷰로 고칠 수 있음

요약:
핵심이 대체로 맞으면 유용함
```

하지만 스도쿠는 다르다.

```text
79칸 맞고 2칸 틀림
-> 실패
```

근사치가 거의 의미 없다. 정답이 딱 맞아야 한다.

또한 스도쿠는 중간 피드백이 약하다.

코드 작업은 테스트나 컴파일 에러가 바로 알려준다.

```text
코드 작성
-> 컴파일
-> 테스트 실패
-> 수정
```

하지만 머릿속 스도쿠는 초반 작은 실수가 조용히 전파될 수 있다.

```text
초반 잘못된 숫자 선택
-> 한동안 그럴듯하게 진행
-> 후반에 모순 발생
-> 어디서 틀렸는지 되돌아가야 함
```

이 되돌아가기와 상태 복원이 바로 LLM에게 약한 부분이다.

## 사람과 LLM의 차이

사람도 어려운 스도쿠를 항상 잘 푸는 것은 아니다.

차이는 "사람은 똑똑하고 LLM은 부족하다"가 아니라, 문제를 푸는 구조에 있다.

| 구분 | 사람 | LLM |
| --- | --- | --- |
| 작업 메모리 | 지우고 다시 쓸 수 있음 | 출력은 append-only에 가까움 |
| 루프 | 실제로 되돌아가며 반복 가능 | 기본 추론은 고정 깊이 forward pass |
| 계산 시간 | 어려우면 더 오래 생각 가능 | 토큰을 더 생성해야 계산이 늘어남 |
| 상태 추적 | 격자, 후보, 가정을 유지 가능 | 긴 상태를 텍스트로 유지해야 함 |
| 강한 영역 | 느린 알고리즘 실행, 검산 | 패턴 인식, 언어화, 솔버 코드 생성 |

사람 인지는 느리고 의식적인 알고리즘 실행을 할 수 있다.

반면 트랜스포머는 기본적으로 빠른 패턴 처리 엔진에 가깝다.

추론 모델은 이 패턴 엔진 위에 토큰을 더 많이 생성하게 해서, 느린 사고처럼 보이는 과정을 얹은 것으로 볼 수 있다.

## LLM은 솔버를 짜는 것은 잘한다

중요한 점은 LLM이 스도쿠에 완전히 무력하다는 뜻이 아니다.

LLM은 스도쿠 솔버 코드를 짜는 데는 꽤 강할 수 있다.

```text
스도쿠 규칙 설명
백트래킹 알고리즘 설명
Python 코드 작성
테스트 방법 제안
```

이런 것은 인터넷과 코드 데이터에 많이 있는 패턴이고, 언어와 코드 생성 능력이 잘 작동하는 영역이다.

하지만 솔버 코드가 되는 것은 다르다.

> [!summary]
> LLM은 솔버를 짜는 것은 잘해도, 솔버가 되는 것은 약하다.

솔버가 되는 일은 다음 능력이 필요하다.

```text
정확한 상태 유지
반복
백트래킹
검증
모순 발견
이전 상태 복원
```

이것은 일반 프로그램이나 파이썬 실행기에 맡기는 것이 맞다.

## 실무적 결론

스도쿠 사례는 LLM을 어떻게 써야 하는지 잘 보여준다.

```text
나쁜 사용:
LLM아, 머릿속으로 정확히 풀어줘.

좋은 사용:
LLM아, 스도쿠 솔버를 짜줘.
그 솔버를 실행해서 답을 검증하자.
```

이 구조는 앞에서 정리한 "확률적 LLM을 결정론적 레이어로 감싼다"는 원칙과 같다.

```text
LLM:
솔버 코드 생성, 설명, 접근 방법 제안

결정론적 도구:
코드 실행, 검증, 정답 판정
```

백엔드 개발에서도 같은 원리가 적용된다.

```text
LLM:
코드 초안 작성

테스트/타입체커/린터/CI:
정확성 검증
```

> [!tip]
> LLM은 직관과 패턴 엔진에 가깝고, 실행이 필요하면 코드, 계산기, 검증기 같은 도구와 짝지어야 제값을 한다.

## 나중에 다시 볼 포인트

- 스도쿠는 수정 가능한 작업 메모리와 백트래킹 루프가 필요하다.
- LLM의 출력은 append-only 텍스트에 가까워서 상태를 안정적으로 지우고 다시 쓰기 어렵다.
- 트랜스포머의 단일 forward pass는 고정 깊이 계산에 가깝다.
- 추론 모델은 토큰을 많이 생성해서 부족한 작업 메모리와 루프를 텍스트로 흉내 낼 수 있다.
- LLM은 솔버를 짜는 데는 강하지만, 솔버 자체가 되는 데는 약하다.
- 정확한 실행과 검증은 결정론적 도구에 맡기는 것이 좋다.
