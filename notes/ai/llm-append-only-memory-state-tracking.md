# 🧠 LLM 추가형 메모리와 최신 상태 추적 한계

> "LLM도 메모리를 추가할 수 있는데, 왜 최신 상태를 정확히 못 읽지?" 라는 질문에서 출발한 노트. 핵심은 저장 공간이 아니라 **읽기 방식**이다.

---

## 📌 한 줄 요약

LLM은 메모리를 못 쓰는 게 아니다. 출력 텍스트와 컨텍스트에 계속 추가할 수 있다.

진짜 한계는 쌓인 기록에서 최신 상태를 `board["47"]` 처럼 정확히 조회하는 구조가 아니라, 관련도 높은 정보들을 섞어서 읽는 구조라는 점이다.

---

## 1. 스도쿠 비유 — 지울 수 없어도 추적은 된다

너와 LLM이 둘 다 스도쿠를 푼다고 하자.

규칙은 하나다:

> 종이에 한번 쓴 글자는 절대 못 지운다. 줄 긋고 옆에 다시 쓰는 것만 된다.

예를 들어 47번 칸을 계속 수정하면 기록은 이렇게 쌓인다.

```text
47번 칸 = 5
47번 칸 = 5 -> 7로 수정
47번 칸 = 7 -> 다시 3
```

사람은 이걸 보고 "맨 아래가 최신이니까 지금 47번 칸은 3"이라고 판단할 수 있다.

즉, **못 지워도 상태 추적 자체는 가능하다.** 여기까지는 네 직관이 맞다.

---

## 2. 문제는 "추가"가 아니라 "조회"다 🔍

LLM도 뒤에 수정 내용을 계속 추가할 수 있다.

문제는 최신 값을 찾을 때다.

기록이 짧으면 잘 찾는다.

```text
47번 칸 = 5
47번 칸 = 7
47번 칸 = 3
```

그런데 81칸이 있고 각 칸이 수십 번 수정되면 기록은 금방 수백 줄이 된다.

```text
1번 칸 후보 수정
2번 칸 후보 수정
47번 칸 = 5
...
47번 칸 = 7
...
12번 칸 후보 수정
...
47번 칸 = 3
...
```

이때 LLM은 "47번 칸의 최신값"을 정확한 주소로 찍어 가져오는 게 아니라, 컨텍스트 전체에서 관련 있어 보이는 줄들을 다시 훑는다.

그래서 오래된 `47번 = 5`, 중간 수정인 `47번 = 7`, 최신값인 `47번 = 3`이 모두 비슷하게 중요해 보일 수 있다.

> 핵심 한계는 "기록을 못 추가한다"가 아니라, 쌓인 기록에서 현재 상태를 안정적으로 조회하는 능력이 약해진다는 점.

---

## 3. 해시맵 조회 vs attention 조회 🧩

백엔드 코드라면 최신 상태를 해시맵에서 바로 꺼낼 수 있다.

```text
board["47"] -> 3
```

이건 키 `"47"`로 47번 슬롯에 직접 점프하는 방식이다. `47`과 `74`는 완전히 다른 키라서 헷갈리지 않는다.

LLM의 기본 읽기 방식은 이와 다르다.

```text
"47번 칸의 값을 찾고 싶다"
-> 컨텍스트에 있는 여러 줄을 함께 본다
-> 각 줄이 지금 찾는 내용과 얼마나 관련 있는지 점수를 매긴다
-> 관련도가 높은 정보들을 섞어서 다음 토큰을 만든다
```

attention은 해시맵 조회보다는 유사도 기반 검색에 가깝다.

키로 하나를 딱 찍어 가져오는 게 아니라, 여러 토큰과의 관련도를 계산하고 그 비중에 따라 정보를 섞는다.

---

## 4. 왜 흐릿해지는가 🌫️

스도쿠 기록에는 비슷하게 생긴 줄이 많이 쌓인다.

```text
47번 = 5
47번 = 7
47번 = 3
```

사람이나 프로그램은 "마지막 47번 값은 3"이라고 규칙적으로 판단할 수 있다.

하지만 LLM 입장에서는 세 줄 모두 "47번 칸 이야기"라서 관련도가 높다. 그래서 오래된 값인 `5`나 `7`이 최신값 `3`과 섞여 들어올 수 있다.

비슷한 칸이 많아지면 더 헷갈린다.

```text
47번 = 3
41번 = 8
74번 = 6
```

사람이 보기에는 `47`, `41`, `74`가 다른 키다. 하지만 LLM에게는 모두 숫자와 칸 값에 대한 비슷한 문맥이다.

LLM이 흐릿하게 읽는 것은 게을러서가 아니라, 기본 작동 방식이 관련도 기반으로 정보를 섞는 구조이기 때문이다.

---

## 5. 이 흐릿함이 언어에는 강점이다 💬

정확한 해시맵 조회는 키가 딱 맞아야 한다.

하지만 자연어는 그렇게만 동작하지 않는다.

```text
강아지
반려견
멍멍이
그거
어제 말한 동물
```

이 표현들은 글자가 다르지만, 문맥에 따라 같은 대상을 가리킬 수 있다.

정확한 키 매칭만으로는 이런 애매한 참조를 처리하기 어렵다. LLM은 유사도와 문맥을 기반으로 관련된 표현을 이어 붙일 수 있기 때문에 언어를 잘 다룬다.

즉, 흐릿하게 읽는 능력은 LLM의 버그가 아니라 언어 이해를 가능하게 하는 핵심 기능이다.

다만 그 같은 특성이 스도쿠 보드처럼 정확한 최신 상태를 관리해야 하는 문제에서는 약점이 된다.

---

## 6. 백엔드 관점 — 이벤트 로그와 현재 상태 테이블

이 차이는 이벤트 로그와 현재 상태 테이블의 차이에 가깝다.

| 방식 | 의미 | LLM에게 생기는 문제 |
| --- | --- | --- |
| 수정 로그 계속 추가 | 모든 변경 이력을 append-only로 남김 | 최신 상태를 매번 재구성해야 함 |
| 현재 상태 전체 재출력 | 매번 깨끗한 최신 스냅샷을 남김 | 바로 위 최신본만 보면 됨 |
| 외부 코드 실행 | 파이썬 같은 도구가 상태를 직접 갱신 | 정확한 계산과 검증이 쉬움 |

이벤트 소싱에서도 이벤트만 계속 쌓으면 조회할 때 현재 상태를 재구성해야 한다.

그래서 실무에서는 projection, snapshot, materialized view 같은 방식으로 "현재 상태"를 따로 만든다. LLM에게도 비슷한 전략이 필요하다.

---

## 7. 더 나은 프롬프트 구조 🧾

수정 기록을 계속 쌓게 하기보다, 한 번 고칠 때마다 현재 보드 전체를 다시 쓰게 하는 편이 낫다.

```text
[현재 보드 전체 81칸]
...
47번 = 3
...
```

그러면 LLM은 지저분한 변경 이력을 뒤질 필요가 줄어든다.

바로 위에 있는 깨끗한 최신본만 보면 되기 때문이다.

> 더러운 변경 로그를 계속 읽게 하는 것보다, 매번 정리된 현황판을 새로 만들어 주는 편이 LLM에게 안정적이다.

---

## 8. 메모리만 더 주면 해결되는 문제가 아니다 ⚠️

"LLM에게 RAM 같은 메모리를 하나 더 붙여주면 끝나는 문제일까?"

아니다.

문제는 저장 공간이 전혀 없다는 것이 아니다. 출력한 텍스트와 컨텍스트도 일종의 메모리이고, 필요하면 외부 DB나 파일도 붙일 수 있다.

백엔드로 비유하면 디스크 자체는 이미 있는 셈이다.

진짜 문제는 저장된 내용을 읽는 방식이다. LLM은 저장된 내용을 일반 프로그램처럼 정확한 키로 조회하는 것이 아니라, 관련도 기반으로 여러 정보를 섞어서 다음 토큰을 만든다.

그래서 메모리 용량만 늘린다고 정확한 조회가 자동으로 생기지는 않는다. 오히려 메모가 더 많이 쌓이면 비슷한 정보가 많아져서 섞일 대상이 늘어날 수도 있다.

---

## 9. 현실적인 우회로 🛠️

LLM 머릿속에 정확한 작업판이 없다면, 정확한 조회가 필요한 일은 바깥 도구로 빼면 된다.

백엔드에서 앱 서버가 불안정한 상태를 직접 들고 있기보다 Redis나 DB에 맡기는 것과 비슷하다.

| 우회로 | 역할 | 효과 |
| --- | --- | --- |
| 코드 실행 | 파이썬 딕셔너리나 배열에 실제 상태를 저장 | `board["47"] = 3` 같은 정확한 조회 가능 |
| 외부 메모리/DB | 최신 상태를 구조화해서 저장하고 조회 | LLM이 아니라 시스템이 정확한 검색 담당 |
| 깨끗한 현황판 재출력 | 현재 상태 전체를 매번 스냅샷으로 정리 | 유사도 기반 혼동을 줄임 |

스도쿠라면 LLM이 직접 머릿속으로 풀기보다, 파이썬 솔버를 만들고 실행하게 하는 것이 더 안정적이다.

```text
LLM 내부 텍스트 메모리:
흐릿한 유사도 기반 추론

Python dict:
정확한 키 기반 조회

DB/Redis:
구조화된 상태 저장과 조회
```

LLM 혼자 정확한 작업판을 갖고 있는 것은 아니지만, 정확한 도구를 호출하게 만들면 그 약점을 우회할 수 있다.

---

## 10. 현재 상황을 솔직하게 보면

이 문제는 "메모리 하나 붙이면 끝"도 아니고, "영원히 해결 불가능한 한계"도 아니다.

지금 실무적으로 잘 되는 방향은 도구로 우회하는 것이다.

```text
LLM
+ 코드 실행기
+ 파일
+ 검색
+ DB
+ 검증기
```

한편으로는 모델 구조 자체에 더 정확한 작업 메모리나 상태 갱신 능력을 넣으려는 연구도 계속되고 있다.

아직 "이 구조가 정답"이라고 말할 만큼 끝난 문제는 아니다.

> 현실적인 결론은 **"우회로는 잘 돌아가고, 정공법은 연구 중"** 이다.

---

## 🎯 나중에 다시 볼 포인트

- LLM도 메모리를 쓸 수 있고, 뒤에 수정 내용을 추가할 수 있다.
- append-only 기록만으로도 짧은 상태 추적은 가능하다.
- 기록이 길어지면 LLM은 최신 값을 정확히 찾는 데 약해진다.
- 한계는 "못 지움" 자체보다 "쌓인 기록에서 현재 상태를 흐릿하게 읽음"에 있다.
- 메모리 용량을 늘린다고 정확한 키 기반 조회가 자동으로 생기지는 않는다.
- 긴 상태 추적은 변경 로그보다 최신 스냅샷을 반복 출력하는 방식이 낫다.
- 정확한 계산, 조회, 검증은 파이썬이나 DB 같은 외부 도구에 맡기는 편이 좋다.
- 현재의 실용적 해법은 도구로 우회하는 것이고, 모델 내부에 정확한 작업판을 넣는 정공법은 아직 연구 중이다.

- [ ] 해시맵 조회와 attention의 유사도 기반 조회 차이를 설명할 수 있다.

- [ ] 변경 로그와 최신 스냅샷의 차이를 백엔드 관점에서 설명할 수 있다.
