# LLM이 같은 실수를 반복하는 이유

> [!summary]
> LLM에게 실수를 한 번 명시했다고 해서 모델 자체가 고쳐지는 것은 아니다.
>
> 지적은 컨텍스트에 텍스트로 추가될 뿐이고, 실제 토큰 생성을 만드는 모델 가중치는 바뀌지 않는다.

## 처음 헷갈렸던 질문

> [!question]
> LLM이 어떤 실수를 반복할 때, "너 이 실수 자꾸 해"라고 한 번 명시해도 다음에 또 같은 실수를 할 수 있을까?

할 수 있다.

핵심은 다음 문장이다.

> "명시했다"가 "고쳐졌다"는 뜻은 아니다.

LLM은 지적을 이해하고 사과하고 "다음부터 안 그러겠다"고 말할 수 있다. 하지만 그 말 자체가 모델의 실제 동작 방식을 바꾸는 것은 아니다.

## 지적과 실제 동작은 다르다

LLM이 "맞습니다. 다음부터 조심하겠습니다"라고 말하는 것도 텍스트 생성이다.

하지만 다음 답변을 만들 때도 모델은 여전히 같은 가중치로 다음 토큰을 예측한다.

```text
사용자 지적:
"너 자꾸 X 실수해"

LLM 반응:
"맞습니다. 다음부터 조심하겠습니다"

실제 다음 출력:
같은 가중치 + 현재 컨텍스트를 바탕으로 다시 토큰 생성
```

여기서 중요한 차이는 다음과 같다.

```text
컨텍스트:
대화에 추가된 텍스트

가중치:
모델이 학습으로 갖게 된 깊은 패턴
```

지적은 컨텍스트에 들어가지만, 가중치를 바꾸지는 않는다.

## 백엔드 비유

코드 리뷰에서 누군가 이렇게 코멘트했다고 하자.

```text
여기 N+1 쿼리 나요.
```

이 코멘트는 문제를 알려주지만, 실제 코드를 고치지는 않는다.

```text
리뷰 코멘트:
문제 지적

코드 수정:
실제 동작 변경
```

LLM도 비슷하다.

```text
사용자의 지적:
컨텍스트에 텍스트 추가

모델 가중치 변경:
일어나지 않음
```

> [!info]
> LLM에게 "이 실수 하지 마"라고 말하는 것은 코드에 리뷰 코멘트를 다는 것에 가깝다.
>
> 실제 코드를 고치거나 테스트를 추가하거나 린터를 거는 것과는 다르다.

## 지적이 잘 먹히는 경우

지적이 항상 무의미한 것은 아니다. 컨텍스트에 살아 있는 동안에는 다음 토큰 생성에 영향을 줄 수 있다.

예를 들어 다음처럼 단순하고 명확한 제약은 같은 세션 안에서 꽤 잘 지켜질 수 있다.

```text
코드에 주석 달지 마.
출력은 JSON으로만 해.
파일을 수정하기 전에 먼저 설명해.
X 라이브러리는 v2 API만 써.
```

이런 지시는 모델이 다음 토큰을 고를 때 참고하기 쉽다.

```text
현재 컨텍스트에 강한 지시가 있음
-> 다음 토큰 확률 분포에 영향
-> 행동이 바뀔 수 있음
```

## 지적이 잘 안 먹히는 경우

반복 실수가 모델의 깊은 경향성에 가까우면, 컨텍스트의 지적 한 줄로는 충분하지 않을 수 있다.

예를 들면 다음과 같다.

- 특정 라이브러리의 옛날 API를 자꾸 쓴다.
- 프로젝트 스타일과 다른 일반적인 예시 코드를 자꾸 쓴다.
- 코드를 고치지 말고 설명만 하라고 했는데 자꾸 구현하려 한다.
- 긴 작업 중 앞에서 받은 제약을 잊고 원래 패턴으로 돌아간다.

이런 경우는 학습 데이터나 모델의 일반 패턴에 강하게 끌리는 상황이다.

> [!warning]
> 컨텍스트에 있는 지시는 영향력을 줄 수 있지만, 모델의 깊은 경향성을 항상 이기지는 못한다.
>
> 특히 컨텍스트가 길어지면 앞의 지적은 묻히거나, 요약 과정에서 약해지거나, 새 세션에서 사라질 수 있다.

## 인간 습관과 비슷한 점

사람도 "그거 안 좋은 거 알아"라고 말한다고 바로 습관이 고쳐지지는 않는다.

예를 들어 누군가 담배가 몸에 나쁘다는 것을 알아도, 그 지식만으로 바로 끊기는 어렵다.

```text
명시적 인지:
나쁜 걸 안다

실제 행동:
습관, 충동, 환경의 영향을 받는다
```

LLM도 비슷하게 볼 수 있다.

```text
명시적 반응:
"그 실수 알아요. 안 할게요."

실제 출력:
가중치와 현재 컨텍스트를 바탕으로 토큰 생성
```

이 비유의 핵심은 LLM이 사람처럼 의식이나 습관을 가진다는 뜻이 아니다. "알고 있다고 말하는 것"과 "실제 동작이 바뀌는 것"이 분리되어 있다는 점을 이해하기 위한 비유다.

## 실무에서 막는 방법

반복 실수는 더 세게 말한다고 해결되는 경우보다, 구조적으로 못 하게 만드는 쪽이 더 잘 먹힌다.

### 반복 주입

중요한 제약은 한 번 말하고 끝내지 말고, 항상 보이게 만들어야 한다.

```text
CLAUDE.md
시스템 프롬프트
커스텀 인스트럭션
프로젝트 README
작업 체크리스트
```

이런 곳에 넣어두면 매 작업 때 컨텍스트에 다시 들어갈 가능성이 높다.

> [!tip]
> "한 번 말함"을 "항상 떠 있음"으로 바꾸는 것이 중요하다.

### 구체적인 지시

모호한 지시는 잘 안 먹힌다.

```text
나쁜 지시:
꼼꼼하게 해.
잘 맞춰서 해.
실수하지 마.
```

구체적인 지시는 더 잘 먹힌다.

```text
좋은 지시:
모든 함수에 타입힌트를 붙여.
X 라이브러리는 v2 API만 써.
응답은 findings, assumptions, summary 순서로 써.
파일 수정 전에는 변경할 파일명을 먼저 말해.
```

모델이 실제 출력에서 확인할 수 있는 형태로 제약을 주는 것이 좋다.

### 밖에서 강제

가장 확실한 방법은 모델을 설득하는 것이 아니라 검증 단계를 두는 것이다.

```text
린터
테스트
타입체커
CI
pre-commit hook
코드 리뷰 체크리스트
```

예를 들어 컨벤션을 지켜달라고 부탁하는 것보다, 린터가 통과하지 못하게 하는 편이 훨씬 안정적이다.

```text
부탁:
"이 스타일 지켜줘"

강제:
lint 실패 -> merge 불가
test 실패 -> CI 실패
```

> [!info]
> LLM의 경향성이 어떻든, 외부 검증 단계가 있으면 같은 실수가 그대로 통과하기 어렵다.

## 반복 주입도 100% 방지는 아니다

반복 주입은 실수를 줄이는 데 도움이 되지만, 100% 방지는 아니다.

LLM은 다음 토큰을 만들 때 확률 분포를 사용한다.

```text
현재 컨텍스트
-> 다음 토큰 후보들의 확률 계산
-> 그중 하나를 선택
```

프롬프트, 시스템 지시, `CLAUDE.md` 같은 반복 주입은 특정 행동의 확률을 높이거나 낮춘다.

하지만 이것은 규칙을 100% 강제하는 것과 다르다.

```text
프롬프트:
"이 실수를 하지 마"
-> 실수할 확률을 낮춤

컴파일러/린터:
규칙 위반
-> 실패 처리
```

> [!warning]
> 프롬프트 엔지니어링은 확률을 기울이는 작업이지, 규칙을 완전히 강제하는 작업이 아니다.

코드 한 번을 생성할 때는 토큰이 수백 개에서 수천 개까지 나올 수 있다.

토큰 하나에서 실수 확률이 매우 낮아도, 생성 길이가 길어지면 어딘가에서 실수할 가능성이 커진다.

```text
작은 실수 확률
* 많은 토큰 생성
* 긴 작업 시간
=> 언젠가 실수가 나올 수 있음
```

temperature를 낮추면 출력이 더 안정될 수 있지만, 그것도 완전한 보장은 아니다.

## 컴파일과 다른 점

컴파일러, 타입체커, 린터는 결정론적 규칙에 가깝다.

같은 입력과 같은 환경에서는 같은 결과가 나온다.

```text
if 규칙 위반:
    reject
```

규칙을 위반하면 통과하지 못한다.

반면 LLM 프롬프트는 모델의 출력 경향을 바꾸는 방식이다.

| 구분 | 컴파일러/린터/테스트 | LLM 프롬프트 |
| --- | --- | --- |
| 동작 방식 | 결정론적 | 확률적 |
| 같은 입력 | 보통 같은 결과 | 설정과 상황에 따라 달라질 수 있음 |
| 규칙 적용 | 위반 시 실패 | 위반 확률을 낮춤 |
| 성격 | 강제 | 경향 조정 |
| 목표 | 통과/실패 판정 | 좋은 출력 유도 |

> [!tip]
> 컴파일러의 타입 체크는 방화벽 규칙에 가깝다.
>
> 프롬프트로 LLM 행동을 제어하는 것은 작업자에게 지침을 주는 것에 가깝다.

## 실무 황금률

LLM 출력은 절대 신뢰의 대상이 아니라 검증의 대상이다.

LLM에게서 100% 보장을 얻으려 하기보다, 결정론적인 바깥 레이어로 감싸서 보장을 만드는 편이 현실적이다.

```text
LLM:
확률적 출력 생성

검증 레이어:
결정론적 통과/실패 판단
```

구조로 보면 다음과 같다.

```text
LLM 출력
-> 린터
-> 타입체커
-> 테스트
-> 스키마 검증
-> CI
-> 사람 리뷰
```

각 레이어의 역할은 다르다.

| 검증 레이어 | 막는 것 |
| --- | --- |
| 린터 | 스타일, 금지 패턴, 포맷 위반 |
| 타입체커 | 타입 불일치, 잘못된 API 사용 |
| 테스트 | 기능 회귀, 엣지 케이스 실패 |
| 스키마 검증 | 잘못된 JSON, 누락 필드, 형식 오류 |
| CI | 통합 검증 실패 |
| 사람 리뷰 | 의도, 설계, 제품 판단 문제 |

예를 들어 LLM이 JSON을 생성해야 한다면, 프롬프트만 믿지 말고 스키마 검증을 둔다.

```text
LLM:
JSON 생성

Zod/Pydantic/JSON Schema:
파싱과 검증

검증 실패:
reject 또는 재요청
```

코드도 마찬가지다.

```text
LLM:
코드 생성

lint/test/typecheck:
통과 여부 확인

실패:
수정 요청 또는 merge 차단
```

> [!summary]
> 중요한 프레임 전환은 이것이다.
>
> "LLM이 실수 안 하게 만들기"가 아니라, "LLM이 실수해도 시스템 밖으로 못 나가게 만들기"가 실무적으로 맞다.

### 새 세션에는 다시 넣기

같은 세션에서 효과가 있던 지시도 새 대화창에서는 사라진다.

모델 가중치가 바뀐 것이 아니기 때문이다.

중요한 규칙은 개인 기억, 프로젝트 지침, 설정 파일, 문서 같은 영구적인 위치에 두는 것이 좋다.

## 정리

LLM이 반복 실수를 하는 이유는 지적이 모델 자체를 고치지 않고, 컨텍스트에만 추가되기 때문이다.

```text
한 번 지적
-> 컨텍스트에 텍스트 추가
-> 다음 답변에 영향은 줄 수 있음
-> 하지만 가중치는 그대로
-> 같은 실수가 반복될 수 있음
```

해결 방향은 다음 두 가지다.

```text
항상 보이게 만들기:
시스템 프롬프트, CLAUDE.md, 커스텀 인스트럭션

밖에서 강제하기:
린터, 테스트, CI, pre-commit hook
```

## 나중에 다시 볼 포인트

- "명시했다"는 "모델이 고쳐졌다"와 다르다.
- 지적은 컨텍스트에 들어가지만, 모델 가중치는 바뀌지 않는다.
- 컨텍스트 지시는 짧고 명확할수록 잘 먹힌다.
- 깊은 모델 경향성이나 긴 작업에서는 같은 실수가 반복될 수 있다.
- 반복 주입은 실수 확률을 낮추지만, 100% 보장하지는 않는다.
- 반복 실수는 프롬프트만으로 해결하기보다 문서화, 반복 주입, 외부 검증으로 막는 것이 좋다.
- LLM 출력은 신뢰의 대상이 아니라 검증의 대상이다.

- [ ] 지적이 잘 먹히는 경우와 잘 안 먹히는 경우를 구분할 수 있다.
- [ ] 반복 실수를 막기 위한 프롬프트 전략과 외부 강제 전략을 말할 수 있다.
- [ ] 새 세션에서 같은 지시를 다시 넣어야 하는 이유를 설명할 수 있다.
- [ ] 반복 주입과 컴파일/린터 검증의 차이를 설명할 수 있다.

## 한 줄 회고

- 헷갈렸던 점: LLM이 "알겠다"고 말하는 것과 실제 출력 경향이 바뀌는 것은 다르다는 점.
