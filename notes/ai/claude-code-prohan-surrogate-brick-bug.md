# 🐛 외톨이 surrogate `\uD83A` 가 세션을 brick 시킨 버그

> Claude Code Issue #61670 을 공부용으로 정리한 노트. 긴 Claude Code 세션 (Opus 4.7, 1M context) 에서 모델이 깨진 유니코드 한 조각을 뱉어서 세션 전체가 먹통이 된 사건.

---

## 📌 한 줄 요약

모델이 `AskUserQuestion` 툴 호출 안에 **짝 없는 high surrogate `\uD83A`** 하나를 뱉었고, 그게 세션 기록에 박히면서 이후 모든 turn 이 같은 400 에러로 죽어버린 사건. `/clear` 로만 복구됐다.

---

## 🧠 0. 사전지식 — 이거 모르면 안 읽힘

본문 들어가기 전에 딱 4개만 감 잡고 가자.

### 0.1 유니코드 vs 인코딩 (제일 중요 ⭐)

이 둘은 **다른 층**이다. 분리해서 봐야 함.

|개념|정체|비유|
|---|---|---|
|**유니코드 / 코드포인트**|글자마다 붙은 번호표 (`A`=65, `가`=44032)|추상적 번호|
|**인코딩 (UTF-8 / UTF-16)**|그 번호를 실제 바이트로 저장하는 방식|물리적 저장|

같은 글자라도 UTF-8 과 UTF-16 은 저장하는 바이트가 다르다. **"번호표(코드포인트) ↔ 저장방식(인코딩) 은 별개의 층"** 이 한 문장이 이 이슈 전체를 푸는 열쇠.

### 0.2 surrogate 는 UTF-16 의 사정

UTF-16 은 한 글자를 16비트 (0~65535) 한 칸으로 표현한다. 근데 이모지처럼 65535 를 넘는 큰 글자가 생기면서 한 칸으로 부족해짐.

그래서 큰 글자를 **두 조각**으로 쪼개 저장하는 트릭이 `surrogate pair`:

- **high surrogate** = "다음 조각이랑 합쳐야 함" 앞조각
- **low surrogate** = "앞조각이랑 합쳐지는" 뒷조각

⚠️ 이 둘은 **반드시 짝**이어야 한다. high 혼자 덩그러니 있으면 (low 가 없으면) → **유효하지 않은 유니코드**. 손 내밀었는데 아무도 안 잡아준 깨진 상태.

> 이번 사건의 `\uD83A` 가 바로 그 **외톨이 high surrogate**. (high 범위 = `\uD800` ~ `\uDBFF`)

### 0.3 한글의 유니코드 범위

완성형 한글은 `U+AC00` ~ `U+D7AF` 구간에 쭉 박혀 있다. 그래서 `완좌` 같은 글자는:

- 조합은 이상해도 → **각 글자는 합법적인 한글 코드**

라고 말할 수 있다. (나중에 "코드포인트 층은 멀쩡한데 단어 층은 깨졌다" 분석의 근거)

### 0.4 JSON 에서 글자 저장 방식 2가지

같은 글자라도 JSON 에는 두 방식으로 저장될 수 있다:

1. **날것 바이트** 그대로 → `가`
2. **이스케이프 형태** → `\uAC00` (백슬래시-u-네자리, 6글자 ASCII)

➡️ 둘 다 같은 글자. 이게 나중에 **파일을 두 가지 방식으로 스캔한** 이유가 된다.

## 1. 증상 — 세션이 brick 됐다

세션이 갑자기 먹통. 뭘 입력해도 똑같은 에러:

```
API Error: 400 The request body is not valid JSON: no low surrogate in string:
line 1 column 703863 (char 703862)
```

에러 메시지 해독:

- `not valid JSON` → JSON 이 깨졌다
- `no low surrogate` → 유니코드 짝꿍이 없다 (← 핵심 키워드 🔑)
- `char 703862` → 정확한 위치까지 알려줌

---

## 2. 🔍 추적 과정 — 천재가 아니라 "체계적"이다

핵심은 영감이 아니라 **변수 통제 + 소거법**. 순서대로 따라가면 누구나 할 수 있다.

### 2.1 "내 입력이 문제인가?" 부터 의심

일부러 **서로 완전히 다른 입력**들을 쳐봤다:

- 한국어: `뭐야`
- 영어: `wtf`
- 글자 하나: `?`
- 깨끗한 ASCII 파이썬 코드

**결정적 관찰** 💡: 뭘 치든 에러 위치가 **항상 `char 703862` 로 똑같았다.**

논리 전개:

```
내 입력이 범인이면 → 입력 바뀔 때마다 에러 위치도 바뀌어야 함
근데 위치가 안 바뀜       → 범인은 내 입력이 아님
그럼 범인은              → 이미 고정돼서 쌓여있는 무언가
```

> 탐정으로 치면 "용의자를 다 바꿔봤는데 범행 시각이 똑같다 → 범인은 입력이 아니라 고정된 과거 기록" 으로 좁힌 것.

### 2.2 "쌓여있는 거" 의 정체

Claude Code 구조 지식이 들어온다:

- Claude Code 는 **매 turn 마다 과거 대화 전체를 다시 API 로 보낸다** (히스토리 replay)
- 그 기록은 `~/.claude/projects/<hash>/*.jsonl` 에 저장됨

➡️ 결론: "`char 703862` 자리에 깨진 글자가 박혀있고, 매번 그걸 다시 보내니까 매번 같은 자리에서 터진다."

✅ `/clear` 로 복구된 게 이걸 증명 — 기록을 버리니 폭탄도 사라짐.

### 2.3 범인을 직접 찾으러 감 (파일 스캔 2종)

깨진 글자가 파일에 저장되는 방식이 **2가지** (§0.4) 라서 둘 다 뒤졌다:

|스캔|찾는 것|결과|
|---|---|---|
|**UTF-8 raw 바이트**|`ED A0~BF` 패턴 (바이트로 직접 깨진 surrogate)|깨끗 ✅|
|**JSON 이스케이프 정규식**|`\uD800`~`\uDBFF` 짝없는 high surrogate (6글자 ASCII)|**`\uD83A` 2개 발견 🎯**|

> "도둑이 창문으로 들어왔나 문으로 들어왔나" 둘 다 확인 — 가능한 침입 경로를 빠짐없이 본 것.

### 2.4 "왜 2개지?" 까지 파고듦

각 줄의 `role` (누가 말한 줄인지) 확인:

```
LINE 757 | role=assistant | ts=2026-05-23T05:23:54.040Z
LINE 758 | role=user      | ts=2026-05-23T05:24:47.296Z
```

- **757** = assistant turn (모델이 원본 생성한 곳)
- **758** = Claude Code 가 그 답변을 "User answered Claude's questions" UI 블록으로 **다시 기록한 복사본**

➡️ 원본 하나가 두 군데 적힌 것. `paired=0, orphan=2` 인 이유. (서로 다른 사고 2건이 아니라 단일 원본)

### 2.5 깨진 글자를 현미경으로 봄 — 3개 층위 🔬

LINE 757 의 실제 `description` 필드 (깨진 부분):

```
"wiki-lint.sh § 4 의 dec-* 검사 로직을 D-* 로 정정.
 안 완좌 단은 \uD83A는 이세 이될 — 크지는 5 분"
```

이걸 **3개 층**으로 분해한 게 분석의 백미:

|층위|상태|설명|
|---|---|---|
|**코드포인트 층**|✅ 정상|`완좌`, `이세 이될`, `크지는` — 각 글자는 합법적 한글 (`U+AC00~U+D7AF`)|
|**단어/형태소 층**|❌ 깨짐|그 조합이 실제 한국어 단어가 아님 (모델 추정 의도: "정정 작업, 약 5분")|
|**surrogate `\uD83A`**|💥 인코딩 자체 깨짐|짝없는 high surrogate → API 파서가 거부하는 진범|

> 같은 자리에서 **언어 층(의미 깨짐) + 인코딩 층(유니코드 깨짐) 이 동시에** 무너졌다.

---

## 3. 🤔 두 층이 동시에 깨진 게 왜 희귀한가

각각이 **다른 메커니즘**에서 나오기 때문.

### 헛소리 (언어 깨짐) — 흔하다

- 모델이 긴 세션에서 횡설수설 / 스크립트 드리프트 (한국어 → 일본어 한자 등)
- "다음 글자 뭐 쓸까" 를 확률로 고르다 길을 잃는 것
- **중요**: 이래도 고르는 글자 하나하나는 **다 유효한 유니코드** → JSON 안 깨지고 API 안 거부 → 그냥 읽기 이상할 뿐

### 인코딩 깨짐 (외톨이 surrogate) — 희귀하다

- 모델은 보통 "유효한 글자 단위" 로 출력
- 짝없는 high surrogate 덜렁 뱉는 건 정상 경로에 거의 없음
- 글자를 고르는 문제가 아니라, **글자 → 바이트 변환/저장 더 아래 단계**에서 어긋나야 생김
- → 희귀 + API 가 단호히 400 거부

### 비유 🎨

||정체|시스템 영향|
|---|---|---|
|**헛소리**|맞춤법 틀린 문장|읽기 거북해도 글자는 글자, 종이에 쓸 수 있음|
|**외톨이 surrogate**|반쪽만 인쇄돼 잉크 번진 글자|글자 인식 자체가 안 됨, 스캐너가 거부|

➡️ 깨지는 **층이 완전히 다르다.** 보통 하나만 터지지, 둘이 같은 지점에서 겹치는 건 드물다. 그래서 작성자도 "같은 뿌리인지 우연히 겹친 건지 단정 못 한다, 같은 지역에 동시 등장했다는 사실만 기록" 이라고 솔직하게 적음.

## 4. 🧪 같은 turn 의 다른 선택지는 멀쩡했다

`AskUserQuestion` 에 선택지 4개가 있었는데 **딱 1개만 깨짐**:

- **E (Recommended)** ✅ 멀쩡한 한국어 — "8 commits push 후 PR --base dev ..."
- **B** ❌ 깨짐 + orphan (위의 그놈)
- **A** ✅ 멀쩡 — "postmortem 과 부속 파일에 v2 필드 추가 ..."
- **F** ✅ 멀쩡 — "Phase A + audit + ... 충분한 세션."

> 같은 추론 run, 같은 turn 인데 1개만 깨짐 → "전체 turn 이 아니라 한 생성 옵션에 국소적인 문제" 라는 신호.

### turn 메타데이터 (`message.usage`)

|항목|값|해석|
|---|---|---|
|`model`|`claude-opus-4-7`||
|`stop_reason`|`tool_use`|깨끗하게 종료, truncation 아님|
|`cache_read_input_tokens`|385,045 (~38%)|1M 컨텍스트의 약 38% 지점|
|`output_tokens`|1,566|평범한 크기 출력|

➡️ 메타데이터상 컨텍스트 고갈/출력 폭주 징후 **없음**.

## 5. ✂️ 소거법 — 확실히 아닌 것 지우기

원인을 못 정해도 **아닌 건 지울 수 있다** = 과학적 태도.

|가설|검증|판정|
|---|---|---|
|정적 파일이 원래 깨져있었나|전체 `~/.claude` 바이트 스캔 → 다른 파일 다 깨끗|❌ 아님|
|네트워크 전송 중 잘렸나|재시도 **전에** 이미 디스크에 깨진 채로 있었음|❌ 아님|
|harness 쪽 문자열 잘림|시스템 프롬프트에 `[truncated]` 마커 채널은 있으나, 증거는 모델 emission 을 가리킴|❌ 가능성 낮음|

➡️ 남는 결론: **"모델 생성 단계에서 깨진 것"** (범인을 못 잡아도 용의선상을 좁힘)

## 6. 💡 작성자의 수정 제안 (겸손하게 제시)

> "내부 구조 모르니 추천이 아니라 제안으로 받아달라" 는 태도.

|위치|방법|트레이드오프|
|---|---|---|
|**Claude Code CLI**|툴 호출 직렬화 **전에** 모든 string 필드에서 외톨이 surrogate 스캔 → 그 turn 만 빠르게 실패|세션 전체 대신 **한 turn 만** 잃음|
|**API 게이트웨이**|서버에서 정화 (sanitize)|모든 클라이언트 한 번에 보호 ✅ / 단 텔레메트리 신호를 가릴 수 있음 ⚠️|

→ 둘 다 말 되고, 둘 다 적용도 가능. layer 우선순위는 Anthropic 판단.

## 7. 🔁 재현성 — 못 했다 (솔직한 한계)

- 무거운 세션 1회성 발생 (body ~200KB, 1M 컨텍스트 깊이, 큰 skill/MCP 세트 로드)
- 비슷한 프롬프트로 fresh 세션에서 **재현 실패**
- 컨텍스트 크기가 원인인지도 **증명 못 함**
- ✅ 확실한 것 단 하나: **재시도 전에 이미 디스크에 깨진 채로 있었다 → 네트워크 artifact 아님**

## 8. 🎓 백엔드 개발자가 가져갈 교훈

> 이거 사실 백엔드 직접 교훈이 빵빵하다.

- **경계에서 검증하라 (validate at the boundary)**: 외부 입력 (여기선 모델 출력) 을 신뢰하고 그대로 영속 저장하면, 깨진 데이터가 박혀서 매 replay 때마다 시스템 전체를 마비시킨다.
- **오염 격리 (fail one record, not the whole system)**: 작성자가 제안한 "그 turn 만 빨리 실패" 가 정확히 이 원칙. 한 레코드 오염이 전체를 죽이면 안 됨.
- **변수 통제 디버깅**: "무엇을 바꿨을 때 무엇이 변하고 무엇이 안 변하는가." 안 변하는 건 내 변경과 무관한 고정 원인이라는 신호. (offset 이 안 움직인 관찰 → 입력 아님 → 누적 상태로 좁힘)
- **"데이터 이상 (읽기 거북)" vs "인코딩 깨짐 (파싱 실패)" 구분**: 이걸 구분할 줄 알면 디버깅 속도가 확 빨라진다. 예) 외부 API 응답을 DB 저장하는데 가끔 깨진 바이트가 섞여 들어와 그 레코드 읽을 때마다 터지는 케이스.

### 실전 체크리스트 ✅

- [ ] 에러 메시지를 **끝까지** 읽었는가 (위치/원인 키워드 추출)
- [ ] **무엇을 바꿔야 증상이 변하는가** 테스트했는가
- [ ] 안 변하는 요소 = 고정 원인 후보로 좁혔는가
- [ ] 데이터가 저장되는 **모든 경로/형식**을 스캔했는가
- [ ] 확실히 **아닌 가설**을 소거했는가
