# LLM 컨텍스트와 튜링 머신 메모리 차이

> [!summary]
> LLM은 계산 도중에 외부 메모리를 마음대로 읽고 쓰는 구조가 아니라, 주어진 컨텍스트 안에서 self-attention으로 관계를 계산해 다음 토큰을 생성하는 구조다.
>
> 그래서 긴 상태 추적, 정확한 카운팅, 스택 처리, 절차적 알고리즘 실행 같은 계산에는 구조적으로 약한 경우가 있다.

## 처음 헷갈렸던 질문

> [!question]
> "LLM은 컨텍스트를 동적으로 재설정할 수 없음"이라는 말은 무슨 뜻일까?
>
> 튜링 머신은 테이프를 메모리로 쓰고, 트랜스포머는 self-attention으로 중간 계산을 추적한다는 말은 왜 중요한가?

LLM의 입력은 보통 다음 요소들이 합쳐진 것이다.

```text
시스템 메시지
+ 이전 대화
+ 사용자 질문
+ 참고 문서
+ 지금까지 생성한 토큰
```

이 전체를 컨텍스트라고 보면 된다.

LLM은 답변을 만들 때 이 컨텍스트 안의 토큰들을 보고 다음 토큰을 예측한다. 하지만 계산 중간에 컨텍스트를 일반 프로그램의 메모리처럼 마음대로 수정하거나 재배치하지는 못한다.

## 프로그램의 상태 갱신

일반 프로그램은 명시적인 변수와 메모리를 사용한다.

```java
int count = 0;

for (Item item : items) {
    if (item.isValid()) {
        count += 1;
    }
}
```

여기서 `count`는 실제로 갱신되는 변수다.

```text
count = 0
count = 1
count = 2
count = 3
```

프로그램은 메모리에 값을 쓰고, 다시 읽고, 조건에 따라 갱신한다.

> [!info]
> 일반 프로그램의 계산은 "명시적 상태를 읽고 쓰는 과정"으로 볼 수 있다.

## 튜링 머신의 테이프

튜링 머신은 이론적인 계산 모델이다. 핵심은 테이프라는 메모리를 가진다는 점이다.

튜링 머신은 다음 과정을 반복할 수 있다.

```text
현재 칸을 읽음
-> 상태에 따라 값을 씀
-> 왼쪽 또는 오른쪽으로 이동
-> 다음 상태로 바꿈
-> 반복
```

중요한 점은 테이프에 값을 쓸 수 있다는 것이다.

예를 들어 괄호 짝을 검사한다고 하자.

```text
((())())
```

일반 알고리즘은 스택이나 카운터를 사용한다.

```text
( 를 보면 count += 1
) 를 보면 count -= 1
중간에 count < 0 이면 실패
마지막 count == 0 이면 성공
```

이런 계산은 중간 상태를 계속 갱신해야 한다.

## 트랜스포머의 self-attention

트랜스포머는 입력 토큰들 사이의 관계를 self-attention으로 계산한다.

예를 들어 다음 문장이 있다고 하자.

```text
철수는 영희에게 책을 줬다. 그는 기뻤다.
```

모델은 `그`라는 토큰이 앞의 어떤 토큰과 관련이 있는지 계산한다.

```text
"그"
-> "철수", "영희", "책" 등을 참고
-> 문맥상 누구를 가리키는지 추론
```

self-attention은 이미 컨텍스트에 있는 토큰들 사이의 관련도를 계산하는 방식이다.

> [!info]
> 트랜스포머의 내부 상태는 명시적 변수, 스택, 힙처럼 값을 직접 읽고 쓰는 메모리와 다르다.
>
> 현재 컨텍스트 안의 토큰들을 보고, 어떤 토큰을 얼마나 참고할지 계산하는 방식에 가깝다.

## 그래서 약해지는 계산

LLM은 짧은 예시는 잘 맞힐 수 있지만, 길고 정확한 상태 추적이 필요한 계산에서는 실수하기 쉽다.

### 긴 카운팅

```text
이 문서에서 조건 A를 만족하는 항목이 정확히 몇 개인가?
```

짧은 목록에서는 잘할 수 있지만, 항목이 길어지면 누락하거나 중복해서 셀 수 있다.

명시적인 `count` 변수를 안전하게 갱신하는 것이 아니라, 컨텍스트를 보고 추론하기 때문이다.

### 스택 추적

```text
(((()())(()))) 이 괄호가 정확히 맞는가?
```

이런 문제는 스택이 있으면 단순하지만, 길어질수록 LLM은 헷갈릴 수 있다.

괄호, 중첩된 JSON, XML, 복잡한 블록 구조처럼 중간 상태를 정확히 유지해야 하는 작업에서 비슷한 문제가 생긴다.

### 긴 절차적 계산

```text
A를 한다
-> 그 결과로 B를 한다
-> B 결과가 7보다 크면 C
-> 아니면 D
-> 그다음 다시 E
```

일반 프로그램은 중간 결과를 변수에 저장한다.

LLM은 중간 결과를 텍스트로 유지하거나 attention 패턴으로 따라가야 하므로, 단계가 길어질수록 오류가 생기기 쉽다.

### 정렬, 집계, 정확한 알고리즘 실행

```text
숫자 100개를 오름차순 정렬해줘
```

사람 눈에는 단순해 보여도, 컴퓨터적으로는 상태를 계속 바꾸는 작업이다.

LLM은 그럴듯한 결과를 만들 수는 있지만, 정확성 보장이 약하다.

## 백엔드 관점으로 이해하기

일반 프로그램은 상태를 명시적으로 저장하고 갱신하는 서버에 가깝다.

```text
요청 받음
-> 변수 갱신
-> DB 조회
-> 상태 저장
-> 다음 로직 실행
```

LLM은 기본적으로 다음 흐름에 가깝다.

```text
요청 전체를 읽음
-> 문맥 패턴을 계산함
-> 다음 텍스트를 생성함
```

그래서 정확한 계산은 외부 도구에 맡기는 편이 좋다.

```text
LLM 단독:
숫자 100개 합산

LLM + Python:
Python으로 합산 실행
LLM은 결과를 설명
```

> [!tip]
> 실무 감각으로 말하면, LLM은 판단과 설명을 잘하고 프로그램은 정확한 상태 계산을 잘한다.
>
> 둘을 붙이면 계산은 도구가 하고, 해석과 다음 행동 결정은 LLM이 맡는 구조를 만들 수 있다.

## 컨텍스트 재설정의 의미

LLM은 컨텍스트 윈도우 안에서만 본다. 그리고 그 안의 내용을 DB처럼 자유롭게 수정, 삭제, 정렬, 압축하면서 계산하지 못한다.

물론 실제 LLM 애플리케이션은 요약, 검색, 파일 읽기, 도구 실행으로 컨텍스트를 새로 구성할 수 있다.

하지만 이것은 LLM 자체가 내부에서 자유롭게 메모리를 편집하는 것이 아니라, 바깥 시스템이 새 입력을 만들어주는 것이다.

```text
LLM 자체:
주어진 컨텍스트를 보고 다음 토큰 생성

LLM 애플리케이션:
요약, 검색, 파일 읽기, 도구 실행으로 컨텍스트를 새로 구성
```

이 차이 때문에 에이전트 구조가 중요해진다.

```text
LLM
+ 파일시스템
+ 검색
+ DB
+ 코드 실행기
+ 메모리 저장소
+ 도구 호출
```

LLM 혼자 모든 계산을 하게 하지 않고, 상태 저장과 정확한 계산은 외부 도구에 맡기는 방식이다.

## 맡기기 좋은 일과 맡기면 위험한 일

| 구분 | LLM에게 잘 맞는 일 | 외부 도구에 맡기는 게 좋은 일 |
| --- | --- | --- |
| 언어 처리 | 요약, 설명, 문맥 해석 | 원문 전체 정합성 검증 |
| 코드 작업 | 코드 초안, 리팩터링 후보, 리뷰 관점 | 테스트 실행, 빌드, 정확한 diff 계산 |
| 데이터 처리 | 결과 해석, 패턴 설명 | 정렬, 집계, 카운팅, 통계 계산 |
| 검색 | 검색어 생성, 결과 요약 | 실제 파일 검색, DB 조회 |
| 추론 | 후보 생성, 판단 근거 설명 | 긴 상태 추적, 엄격한 알고리즘 실행 |

## 나중에 다시 볼 포인트

- LLM의 컨텍스트는 일반 프로그램의 메모리와 다르다.
- 튜링 머신은 테이프에 값을 읽고 쓸 수 있다.
- 트랜스포머는 self-attention으로 컨텍스트 안의 토큰 관계를 계산한다.
- 긴 카운팅, 스택 추적, 정렬, 집계처럼 명시적 상태 갱신이 필요한 작업은 LLM 단독으로 약할 수 있다.
- 실무에서는 LLM에 도구, 파일시스템, DB, 코드 실행기를 붙여 약점을 보완한다.
