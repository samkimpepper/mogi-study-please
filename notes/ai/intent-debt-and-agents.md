# AI 에이전트와 의도 부채

> [!summary]
> AI 에이전트는 코드를 고치고 설명할 수는 있지만, 시스템을 왜 그렇게 만들었는지는 사람이 기록해두지 않으면 알 수 없다.
>
> 그래서 에이전트 기반 개발에서는 기술부채보다 *==의도 부채==*가 더 비싸질 수 있다.

## 세 가지 부채

소프트웨어의 부채는 한 종류만 있는 것이 아니다. 글에서는 `technical debt`, `cognitive debt`, `intent debt`를 나눠서 설명한다.

| 부채 | 어디에 쌓이는가 | 의미 |
| --- | --- | --- |
| `technical debt` | 코드 | 급하게 만든 구조, 꼬인 모듈, 깨지기 쉬운 테스트처럼 나중에 변경을 어렵게 만드는 구현 선택 |
| `cognitive debt` | 사람 머릿속 | 시스템은 존재하지만 팀이 그 구조와 흐름을 충분히 이해하지 못하는 상태 |
| `intent debt` | 문서, ADR, PR, 이슈 같은 산출물 | 왜 그렇게 만들었는지, 어떤 목표와 제약이 있었는지 외부화되어 있지 않은 상태 |

> [!note]
> 세 부채는 서로 독립적이다.
>
> 코드가 깔끔해서 기술부채가 낮아도, 왜 그렇게 만들었는지 기록이 없으면 의도 부채는 높을 수 있다. 내가 완전히 이해하고 있어도 그 지식이 내 머릿속에만 있으면 팀과 에이전트 입장에서는 여전히 부채다.

## 기술부채와 인지부채는 AI가 어느 정도 갚을 수 있다

기술부채는 코드 안에 있다. 에이전트는 코드를 읽을 수 있으므로 꼬인 모듈을 리팩터링하거나, 중복을 줄이거나, 테스트를 추가하는 식으로 어느 정도 도와줄 수 있다.

인지부채도 코드가 남아 있으면 회복 가능하다.

```text
이 API 요청이 들어오면 어떤 서비스와 DB를 거쳐?
이 클래스가 왜 필요한지 설명해줘.
이 테스트가 어떤 정책을 검증하는지 알려줘.
```

이런 식으로 물으면 에이전트가 코드를 읽고 설명해줄 수 있다. 즉 잃어버린 멘탈 모델을 필요할 때 다시 만들 수 있다.

> [!info]
> 백엔드 관점에서 보면 기술부채는 "코드 구조가 변경을 방해하는 문제"이고, 인지부채는 "팀이 요청 흐름, 트랜잭션 경계, 상태 전이, 장애 포인트를 머릿속에 복원하지 못하는 문제"에 가깝다.

## 의도 부채는 AI가 대신 갚기 어렵다

의도는 코드만 보고 확정할 수 없다. 에이전트는 코드에서 그럴듯한 이유를 추측할 수 있지만, 그 추측은 실제 의도가 아니다.

예를 들어 코드에 다음 값이 있다고 하자.

```text
debounceTime = 300ms
```

에이전트는 이렇게 설명할 수 있다.

```text
300ms는 사용자 입력 반응성과 요청 빈도 제한 사이의 균형을 맞추기 위한 값입니다.
```

하지만 실제 이유는 다를 수 있다.

- UX 테스트 결과 300ms가 가장 좋았다.
- 서버 부하 벤치마크 때문에 300ms로 정했다.
- 누군가 임시로 넣은 값을 아무도 다시 보지 않았다.

> [!warning]
> 의도에 대한 그럴듯한 추측은 실제 의도가 아니다.
>
> 에이전트가 자신감 있게 틀린 이유를 만들어내면, 모른다고 말하는 것보다 더 위험할 수 있다.

## 에이전트가 많아질수록 의도 부채 비용이 커진다

예전 팀에서는 문서가 부족해도 어느 정도 버틸 수 있었다. 오래 있던 개발자가 기억하고 있었고, 신규 입사자는 회의, 코드 리뷰, PR 댓글, 잡담을 통해 천천히 맥락을 배웠다.

```text
그 API 응답 구조는 모바일 구버전 때문에 바꾸면 안 돼요.
그 필드는 이상해 보여도 PG사 정산 파일이랑 맞춰야 해요.
그 순서는 2023년에 장애가 나서 일부러 그렇게 둔 거예요.
```

이런 지식은 비싸고 손실이 많은 방식이지만, 사람끼리는 어느 정도 전달됐다.

에이전트는 다르다. 에이전트는 대부분의 세션을 차갑게 시작한다. 오래된 장애 경험도 없고, 팀의 암묵지도 없고, 복도 대화도 모른다. 읽을 수 있는 산출물에 없으면 모르는 것이다.

```text
문서에 없는 의도
-> 에이전트가 모름
-> 사람이 매번 프롬프트로 다시 설명해야 함
-> 설명이 빠지면 에이전트가 추측함
-> 추측 기반 코드가 쌓임
```

> [!summary]
> 예전에는 의도 부채 비용을 온보딩이나 담당자 퇴사 때 가끔 냈다.
>
> 에이전트 기반 개발에서는 새 세션을 열 때마다, 작업을 나눌 때마다, 병렬 에이전트를 늘릴 때마다 반복해서 낸다.

## 백엔드 예시

주문 도메인에 이런 정책이 있다고 하자.

```text
결제 성공 전에는 재고를 확정 차감하지 않는다.
결제 성공 후 취소는 단순 취소가 아니라 환불 플로우를 탄다.
쿠폰 할인 금액의 반올림 규칙은 정산 시스템과 맞아야 하므로 바꾸면 안 된다.
주문 취소 이벤트는 알림 이벤트보다 정산 이벤트가 먼저 발행되어야 한다.
```

이 의도가 문서에 없고 코드에만 흩어져 있으면, 에이전트는 "더 깔끔한 코드"를 만들면서 실제 비즈니스 규칙을 깨뜨릴 수 있다.

예를 들어 쿠폰 로직이 복잡해 보여서 중복 쿠폰 예외를 제거하거나, 재고 차감 시점을 결제 전으로 옮기거나, 이벤트 발행 순서를 단순화할 수 있다. 코드만 보면 리팩터링처럼 보이지만, 시스템의 의도는 망가질 수 있다.

> [!tip]
> 에이전트에게 중요한 문서는 "기능 목록"만이 아니다.
>
> 목표, 제약, 금지사항, 예외가 생긴 이유, 과거에 버린 선택지를 함께 남겨야 한다.

## llm-wiki와의 연결

카파시가 말한 `llm-wiki` 관점과도 연결된다. LLM이나 에이전트가 읽을 수 있는 위키는 사람이 반복해서 설명하지 않기 위한 외부 기억장치다.

```text
사람 머릿속 의도
-> 문서, ADR, README, PR 설명으로 외부화
-> 에이전트가 읽을 수 있는 컨텍스트가 됨
-> 의도 부채 감소
```

좋은 LLM용 문서는 단순히 "이 클래스는 무엇이다"만 적지 않는다.

```markdown
# 주문 정책

- 주문은 CREATED -> PAID -> SHIPPING -> COMPLETED 순서로 진행된다.
- CANCELLED는 PAID 이전에는 즉시 가능하다.
- PAID 이후 취소는 환불 요청 플로우를 타야 한다.
- 재고 차감은 결제 성공 시점에 한다.
- 이 정책은 결제 실패 후 재고 잠김 문제를 줄이기 위해 정했다.
```

이런 문서가 있으면 에이전트는 주문 코드를 고칠 때 지켜야 할 방향을 알 수 있다. 반대로 문서가 없으면 에이전트는 코드에서 그럴듯한 규칙을 추측할 뿐이다.

## 나중에 다시 볼 포인트

- 기술부채는 코드에 있고, 인지부채는 사람의 이해에 있고, 의도 부채는 외부화된 산출물의 부재에 있다.
- 에이전트는 코드 리팩터링과 코드 설명은 도와줄 수 있다.
- 에이전트는 실제 의도를 생성할 수 없다. 문서가 없으면 추측만 할 수 있다.
- 병렬 에이전트를 많이 쓸수록 문서화되지 않은 의도를 매번 다시 설명하는 비용이 커진다.
- LLM wiki, ADR, README, PR 설명은 에이전트에게 반복 공급할 의도를 외부화하는 장치다.

## 한 줄 회고

- 헷갈렸던 점: AI가 코드를 설명해줄 수 있다고 해서 시스템의 진짜 의도까지 아는 것은 아니다. 의도는 사람이 문서로 외부화해야 에이전트도 읽을 수 있다.
