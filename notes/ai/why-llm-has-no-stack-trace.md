# 왜 LLM 사고과정은 스택트레이스처럼 찍기 어려운가

> [!summary]
> LLM 내부에도 수치 계산의 흐름은 있지만, 백엔드 코드처럼 이름 붙은 함수 호출 경계가 없다.
>
> 그래서 "스택트레이스를 찍자"는 방향은 맞지만, 찍을 수 있는 함수형 스택트레이스가 구조적으로 존재하지 않는다.

## 내가 물어본 질문

> [!question]
> LLM의 사고과정을 스택트레이스처럼 찍어서 출력하면 해결되는 거 아님?

## 먼저 잡을 한 줄 정의

> [!info]
> 스택트레이스는 의미 있는 함수 경계와 호출 기록이 있을 때 가능하다.
>
> LLM 내부는 사람이 이름 붙인 함수 호출이 아니라, 거대한 벡터와 행렬 연산으로 이루어져 있다.

## 스택트레이스가 존재하려면 필요한 것

일반 백엔드 코드에서 스택트레이스가 나오는 이유는 코드가 의미 단위로 쪼개져 있기 때문이다.

```text
calculateTax()
-> getUserRegion()
-> fetchFromDB()
```

각 함수에는 이름이 있고, 책임이 있고, 호출 경계가 있다. 개발자가 "여기는 세금 계산", "여기는 사용자 지역 조회", "여기는 DB 접근"처럼 의미 단위로 나눠 설계했기 때문이다.

스택트레이스는 이 의미 단위들이 어떤 순서로 호출됐는지 보여주는 기록이다.

> [!info]
> 스택트레이스는 그냥 실행 흔적이 아니라, 사람이 설계한 의미 경계 위에 찍히는 실행 흔적이다.

## LLM 내부에는 함수 경계가 없다

LLM 내부는 대략 이런 식으로 흘러간다.

```text
레이어 1: [0.3, -1.2, 0.7, ...]
   -> 행렬 곱과 비선형 계산
레이어 2: [-0.8, 0.1, 2.3, ...]
   -> 행렬 곱과 비선형 계산
레이어 3: ...
```

여기에는 `calculateTax()`나 `validateUser()` 같은 함수 이름이 없다. 수천~수만 차원의 숫자 벡터가 여러 레이어를 지나면서 다음 숫자 벡터로 바뀐다.

문제는 특정 칸 하나가 "세금 계산 담당", "인증 담당", "날짜 파싱 담당"처럼 깔끔하게 나뉘어 있지 않다는 점이다. 하나의 개념은 여러 숫자에 분산될 수 있고, 숫자 하나도 여러 개념에 동시에 관여할 수 있다.

이런 현상을 `superposition`이라고 부른다.

> [!warning]
> LLM 내부에는 계산 흐름은 있지만, 사람이 읽을 수 있는 의미 단위의 호출 스택은 없다.

## 백엔드 비유로 보면

LLM 내부를 억지로 코드에 비유하면 이런 상황에 가깝다.

```text
float[12000] state;

state = hugeMatrixMultiply(state);
state = hugeMatrixMultiply(state);
state = hugeMatrixMultiply(state);
```

변수명도 의미가 없고, 함수 분리도 거의 없고, 주석도 없다. 게다가 `state[381]` 하나가 인증, 세금 계산, 날짜 파싱, 문장 톤 조절에 동시에 조금씩 관여할 수도 있다.

이런 구조에서는 일반적인 의미의 스택트레이스를 찍기 어렵다. 찍을 수 있는 것은 "몇 번째 레이어를 지났는가", "어떤 활성화 값이 나왔는가" 같은 수치 기록이지, "지금 세금 계산 함수가 호출됐다" 같은 의미 기록이 아니다.

## Mechanistic Interpretability가 하는 일

이 문제를 풀려는 연구 분야가 `mechanistic interpretability`다.

이 분야에서는 모델 내부의 활성화 값, attention head, 뉴런, 회로를 분석해서 어떤 패턴이 어떤 개념이나 행동과 연결되는지 역공학한다.

예를 들면 다음과 같은 질문을 던진다.

- 어떤 활성화 패턴이 특정 사실을 떠올릴 때 반복해서 켜지는가?
- 어떤 attention head가 문장 안의 특정 관계를 추적하는가?
- 어떤 회로를 조작하면 모델의 출력이 특정 방향으로 바뀌는가?

> [!example]
> Anthropic은 특정 feature를 강하게 활성화했을 때 모델이 Golden Gate Bridge와 관련된 말을 과도하게 하도록 만드는 실험을 공개한 적이 있다.
>
> 이런 실험은 "모델 안에 의미 있는 회로를 일부 찾을 수 있다"는 가능성을 보여준다. 다만 전체 사고과정을 스택트레이스처럼 다 읽을 수 있다는 뜻은 아니다.

## 왜 아직 해결이 아닌가

| 어려움 | 의미 |
| --- | --- |
| 규모 | 파라미터와 활성화가 너무 많아서 사람이 전부 라벨링할 수 없다 |
| Superposition | 하나의 숫자나 회로가 여러 개념에 동시에 관여할 수 있다 |
| 의미 경계 부재 | 사람이 짠 함수처럼 책임이 깔끔하게 나뉘어 있지 않다 |
| 창발적 조합 | 작은 회로들이 합쳐져 만드는 고차원 행동은 더 추적하기 어렵다 |

그래서 "스택트레이스를 찍자"는 발상은 틀린 게 아니라, 해석가능성 연구가 가고 싶은 방향에 가깝다. 다만 이것은 `console.log`를 하나 추가하는 문제가 아니다.

> [!danger]
> LLM 내부 값을 모두 출력한다고 해서 사람이 이해 가능한 사고과정이 바로 나오지는 않는다.
>
> 로그가 많아지는 것과 의미 있는 디버깅 정보가 생기는 것은 다르다.

## 컴파일된 바이너리 비유

LLM 해석은 소스코드도 없고, 디버그 심볼도 없는 컴파일된 바이너리를 리버스 엔지니어링하는 일과 비슷하다.

그런데 더 어려운 점은, 이 바이너리가 사람이 명확한 함수 구조로 짠 코드가 아니라 학습 과정에서 저절로 만들어졌다는 점이다.

> 사람이 짠 코드의 디버그 심볼을 잃어버린 것이 아니라, 애초에 사람이 이름 붙인 함수 구조가 없는 시스템을 해석하는 문제에 가깝다.

## 면접에서 말하면

> [!tip]
> LLM에도 레이어별 활성화나 attention 같은 내부 계산 흔적은 있습니다. 하지만 일반 코드의 스택트레이스처럼 의미 있는 함수 호출 기록이 있는 것은 아닙니다.
>
> 스택트레이스는 사람이 만든 함수 경계와 호출 구조가 있어야 가능한데, LLM 내부는 학습으로 만들어진 거대한 수치 연산이고 개념도 여러 차원에 분산되어 있습니다. 그래서 내부 값을 찍는 것만으로는 "왜 그렇게 답했는지"를 바로 읽을 수 없습니다.

> [!note]-
> 조금 더 길게 말하면
>
> Mechanistic interpretability는 이런 숫자 덩어리 안에서 의미 있는 회로나 feature를 역공학하는 분야입니다. 부분적으로는 성공 사례가 있지만, 모델 전체의 추론 과정을 사람이 읽을 수 있는 스택트레이스로 출력하는 수준과는 아직 거리가 있습니다.

## 나중에 다시 볼 포인트

- 스택트레이스는 의미 있는 함수 경계와 호출 기록이 있을 때 가능하다.
- LLM 내부는 함수 호출이 아니라 벡터와 행렬 연산의 연속이다.
- 개념은 특정 변수 하나에 깔끔하게 저장되지 않고 여러 활성화에 분산될 수 있다.
- `superposition` 때문에 하나의 내부 요소가 여러 기능에 동시에 관여할 수 있다.
- Mechanistic interpretability는 내부 회로와 feature를 역공학하는 연구 분야다.
- 내부 값을 많이 출력하는 것과 사람이 이해할 수 있는 디버그 정보가 생기는 것은 다르다.
