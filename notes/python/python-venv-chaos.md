# 파이썬 환경 관리, 왜 카오스였나 — 자바 개발자를 위한 정리 🐍☕

> [!info] 이 문서는 뭐냐 "uv 나오기 전엔 파이썬 환경 어떻게 관리했냐?" 라는 질문에서 시작해서, 자바(Maven/Gradle) 쓰던 백엔드 개발자 입장에서 파이썬 환경 관리가 왜 유난히 빡셌는지 정리한 글. 핵심 결론: **파이썬의 특이점은 "버전 관리"가 아니라 "의존성 격리를 언어가 안 챙겨주고 사람한테 떠넘긴 것"이다.**

---

## 1. uv 이전 시대 — 도구 분업 체제

uv가 지금 혼자 다 하는 일을, 옛날엔 **여러 도구가 나눠서** 했다. 이게 카오스의 근원.

|역할|옛날 도구|하는 일|
|---|---|---|
|파이썬 버전 관리|`pyenv`|3.10, 3.11, 3.12 여러 개 깔고 프로젝트마다 전환|
|가상환경 생성|`venv` / `virtualenv`|프로젝트별 독립된 패키지 공간 만들기|
|패키지 설치|`pip`|`pip install requests`|
|의존성 잠금(lock)|`pip-tools` / `poetry` / `pipenv`|"정확히 같은 버전" 보장|

> [!note] 주니어가 잘 모르는 포인트 — lock `requirements.txt`만으론 "정확히 같은 버전"을 보장 못 한다. 그래서 `pip-compile`(pip-tools)로 잠금 파일을 만들거나, `poetry`/`pipenv` 같은 통합 도구를 따로 썼다.

### 전형적인 옛날 워크플로우

```bash
pyenv install 3.11.5              # 파이썬 버전 깔고
pyenv local 3.11.5               # 이 프로젝트는 3.11.5 쓸게
python -m venv .venv             # 가상환경 만들고
source .venv/bin/activate        # 활성화하고
pip install -r requirements.txt  # 패키지 설치
```

### uv는 이걸 한 줄로

```bash
uv sync
```

`pyproject.toml` + `uv.lock` 있으면 → 파이썬 버전 설치, venv 생성, 패키지 설치까지 전부 자동.

## 2. 자바랑 뭐가 다른가 (핵심)

> [!tip] 오해 정리 "자바도 JDK 1.5 / 1.7 / 1.8 여러 개 깔아두는데?" → **맞다. 그건 파이썬 특이점 아니다.** 그건 카오스의 **1단계(버전 관리)**일 뿐이고, `pyenv`랑 같은 영역의 일이다. 파이썬이 추가로 빡센 건 아래 두 가지 때문.

### 차이점 ① 자바엔 없는 "전역 vs 가상환경" 레이어

자바는 **Maven/Gradle이 프로젝트 단위로 의존성을 격리**해준다. JAR을 시스템 전역에 깔아서 공유하는 짓을 안 한다. 라이브러리는 `~/.m2/repository`에 캐시되고, 뭘 쓸지는 `pom.xml`이 결정.

근데 파이썬 `pip`은 **기본 동작이 전역 설치**였다.

```bash
pip install requests   # 시스템 파이썬에 그냥 깔림. 모든 프로젝트가 공유.
```

→ 프로젝트 A는 `django 3.x`, B는 `django 4.x` 필요하면 충돌. → 이걸 막으려고 **가상환경(venv)** 이라는 레이어를 사람이 직접 만들고 활성화해야 함.

> [!example] 비유
> 
> - **자바/Maven**: 프로젝트마다 자기 서랍이 기본으로 있음. 알아서 자기 서랍에 넣음.
> - **파이썬/pip**: 기본이 공용 책상. 안 겹치게 하려면 _네가_ 칸막이(venv)를 매번 세워야 함.

### 차이점 ② "런타임 자체가 의존성을 들고 있는" 구조

- **자바**: JVM(실행기)이랑 라이브러리가 깔끔하게 분리. 의존성은 클래스패스로 주입. JDK 자체에 프로젝트 라이브러리를 안 깐다.
- **파이썬**: 인터프리터에 `site-packages` 폴더가 딱 붙어있음. 즉 **"파이썬 = 런타임 + 거기 깔린 패키지들"이 한 덩어리.**

→ "어느 파이썬을 쓰느냐" = "어느 패키지 묶음을 쓰느냐"가 직결됨. → 그래서 _"venv가 터미널에선 되는데 VSCode에선 다른 Python을 가리킴"_ 같은 일이 자주, 조용히 발생. (자바로 치면 IDE가 다른 클래스패스를 보는 것과 비슷)

### 한눈에 비교

||버전 여러 개|의존성 격리|누가 함|
|---|---|---|---|
|**자바**|있음 (JDK 1.5/1.8...)|기본 제공 (Maven/Gradle)|빌드툴이 자동|
|**파이썬(옛날)**|있음 (pyenv)|**수동** (venv 직접)|사람이 매번|

## 3. "그냥 venv 쓰면 되는 거 아냐?" — 함정

venv는 **1번 문제(의존성 격리)만** 해결한다. "칸막이 세우기" 딱 한 칸. 근데 실무에서 귀찮은 건 칸막이 자체가 아니라 주변 일들이다.

### ① 매번 손으로 활성화해야 함

```bash
source .venv/bin/activate   # 이거 까먹으면 전역에 깔림 💀
```

터미널 새로 열 때마다 해야 함. 까먹고 `pip install` 하면 시스템 전역을 오염시킴. → 자바로 치면 "프로젝트 열 때마다 클래스패스를 손으로 export 해야 하는데, 까먹으면 남의 프로젝트 JAR이 섞임". Maven은 이런 거 시킨 적 없다.

### ② venv는 버전 관리(pyenv)랑 별개

venv는 "지금 활성화된 파이썬"으로 환경을 만들 뿐, **파이썬 버전 자체는 못 고른다.** → 결국 **pyenv + venv** 조합이 됨. 도구 2개.

### ③ 진짜 핵심 — venv엔 lock 개념이 없음

venv는 빈 방만 만들어주고, 패키지는 `pip install`로 직접 채움. 뭘 깔았는지 기록은?

```bash
pip freeze > requirements.txt   # 이걸 손으로 해야 함
```

그리고 이 `requirements.txt`는 자바 `pom.xml`처럼 똑똑하지 않다.

> [!warning] "내 컴에선 되는데 서버에선 터짐"의 진원지
> 
> - `requests`만 적고 버전 안 박으면 → 사람마다 다른 버전 깔림
> - 버전 박아도 → `requests`가 끌고 오는 **하위 의존성(transitive)** 은 안 잠겨서, 어제랑 오늘 깔리는 게 미묘하게 달라짐
> 
> 자바 `pom.xml`은 "나는 `spring-boot 3.2` 원해"라고 _의도_를 적으면 하위 의존성은 Maven이 풀어준다. 옛날 파이썬 `requirements.txt`는 보통 평평한 목록이라 이걸 제대로 잠그려면 또 `pip-tools`나 `poetry`를 얹어야 했다.

### 정리: venv "만"으로 끝나려면 결국

```text
파이썬 버전 고르기   → pyenv (별도)
격리 공간 만들기     → venv (이건 함) ✅
활성화               → 매번 수동
패키지 설치          → pip (별도)
설치 목록 기록       → pip freeze (수동)
정확한 버전 잠그기   → pip-tools/poetry (또 별도)
```

→ venv는 이 중 **딱 한 칸**. 자바 개발자 입장에선 **"Maven이 알아서 해주던 걸 6단계로 쪼개서 사람이 손으로 하는 느낌"** 이 정확하다.

## 4. 결론 — uv가 한 일

uv는 위 6칸을 다시 **하나의 도구 + 하나의 lock 파일**로 합쳤다.

> [!success] 한 줄 요약 uv는 파이썬을 **Maven/Gradle처럼** 만든 것. `pyproject.toml`에 의존성 적으면 알아서 격리 환경 만들고 lock 걸어줌. 자바에서 Gradle 한 번 설정하면 끝인 것처럼, uv도 `uv sync` 한 줄.
> 
> 거꾸로 말하면 — **자바가 처음부터 갖고 있던 걸, 파이썬은 20년 걸려 외부 도구로 따라잡은 셈.**

## 5. 역사적 배경 — 왜 OS Python이 문제였나

> [!summary] 한 줄 정리
> Python은 앱 개발 언어이기도 했지만, macOS/Linux의 **시스템 관리·자동화 스크립트 언어**로도 오래 쓰였다. 그래서 한 컴퓨터 안에 "OS가 쓰는 Python"과 "개발자가 쓰는 Python"이 섞이기 쉬웠고, 전역 설치가 위험해졌다.

OS 커널 같은 핵심 부분은 보통 C/C++로 만든다. 그런데 OS에는 커널 말고도 주변 관리 도구가 많다.

```text
패키지 설치/업데이트
설정 파일 생성
시스템 상태 점검
로그 처리
빌드/설치 스크립트
사용자 계정/네트워크 설정 보조 도구
```

이런 작업은 초고성능보다 **빠른 개발, 유지보수, 문자열·파일 처리**가 중요하다. Python은 bash보다 구조적이고, C보다 가볍게 만들 수 있어서 이 영역에 잘 맞았다.

### 왜 굳이 Python이었나

|이유|의미|
|---|---|
|bash보다 복잡한 로직을 쓰기 좋음|조건 분기, 예외 처리, JSON/XML 파싱, 파일 처리 등을 읽기 좋게 작성 가능|
|C보다 개발 비용이 낮음|컴파일, 메모리 관리, 배포 부담이 작음|
|표준 라이브러리가 풍부함|파일, 경로, 프로세스, 네트워크, 압축 같은 OS 주변 작업을 기본 라이브러리로 처리 가능|
|Unix/Linux 문화와 잘 맞음|작은 관리 도구와 자동화 스크립트를 조합하는 방식에 적합|
|기본 설치되어 있으면 배포가 쉬움|OS 도구가 Python에 의존하고, 그 의존성 때문에 Python이 더 기본 구성처럼 굳어짐|

> [!note] 순환 구조
>
> ```text
> Python이 기본 설치됨
> → OS 도구를 Python으로 만듦
> → 그 도구 때문에 Python이 더 필수가 됨
> → 시스템 Python을 함부로 못 건드리게 됨
> ```

대표적으로 과거 Fedora/RHEL 계열의 `yum`은 Python 기반이었다. 그래서 시스템 Python이나 그 패키지를 망가뜨리면 패키지 매니저 자체가 깨질 수 있었다.

```bash
sudo pip uninstall some-library
```

개발자는 "내 프로젝트 패키지 하나 지운다"고 생각했는데, 그 라이브러리를 OS 관리 도구가 쓰고 있었다면 패키지 설치·업데이트 도구가 같이 망가질 수 있다. 그래서 **시스템 Python은 건드리지 말고, 개발용 Python과 프로젝트 venv를 따로 두라**는 문화가 강해졌다.

### 다른 언어는 이런 문제가 없나?

전혀 없는 건 아니다. 대부분의 언어는 버전 관리와 의존성 격리 문제가 있다.

|언어|비슷한 문제|특징|
|---|---|---|
|Ruby|system Ruby, 전역 gem 설치|macOS 기본 Ruby와 프로젝트 Ruby가 섞일 수 있음|
|Node.js|전역 `npm install -g`, Node 버전 충돌|`nvm`, `fnm`, `volta`, 로컬 `node_modules`로 관리|
|Java|JDK 버전, `JAVA_HOME` 충돌|Maven/Gradle이 의존성 격리를 많이 맡아줌|
|C/C++|시스템 라이브러리, 헤더 경로, ABI|환경 문제가 매우 강하지만 성격이 다름|
|Go/Rust|상대적으로 적음|`go`, `rustup`, `cargo` 생태계가 비교적 일관됨|

Python이 유독 복잡하게 느껴지는 이유는 다음이 겹쳤기 때문이다.

```text
OS가 Python을 시스템 도구로 많이 씀
pip 전역 설치가 쉬웠음
Python 버전과 패키지 버전이 둘 다 중요함
python, python3, pip, pip3가 서로 다른 환경을 가리킬 수 있음
```

백엔드 관점으로 보면 **OS Python은 운영 서버의 시스템 패키지**, `.venv`는 **애플리케이션 컨테이너 안의 의존성 공간**에 가깝다. 운영체제가 쓰는 공용 런타임에 앱 의존성을 막 설치하면 장애가 날 수 있으니, 앱별로 격리하는 것이다.

## 보너스 — Dev Containers는 또 다른 처방

처음 트친이 붙인 글의 주제(Dev Containers)는 사실 **별개의 접근**이다.

- **uv** = "도구를 통일"해서 정리정돈 🧹
- **Dev Containers** = 아예 **Docker 컨테이너 안에 환경 전체를 격리**해서 호스트(맥북) 환경과 분리 📦

|용어|의미|
|---|---|
|호스트(host)|내 맥북 본체. 컨테이너의 바깥쪽|
|컨테이너/게스트(guest)|Docker가 띄운 격리된 작은 리눅스 박스|
|호스트 Python|시스템/Homebrew/uv/pyenv 등 맥북에 깔린 파이썬|
|컨테이너 Python|devcontainer 이미지 안의 파이썬. 호스트와 물리적으로 분리|

> [!quote] 비유 호스트 = 아파트 건물, 컨테이너 = 그 안의 한 호실. 호실 안에서 라면 끓이는 데 건물 1층 로비 부엌(호스트 Python)은 필요 없다 — devcontainer도 같은 구조.

둘 다 같은 고통("환경마다 빠개짐")을 푸는 **다른 처방**이다.

- uv = 정리정돈
- devcontainer = 아예 다른 방에서 작업

## TL;DR 🎯

- 자바도 버전 여러 개 까는 건 똑같음. 그건 카오스 1단계일 뿐.
- 파이썬이 빡센 진짜 이유: **의존성 격리를 언어가 안 챙겨주고 사람한테 떠넘김.**
- `pip`은 기본이 전역 설치 → venv로 사람이 직접 칸막이.
- venv는 격리 딱 한 칸만 해결. 활성화·버전관리·lock은 다 수동/별도 도구.
- Python은 OS 관리 스크립트에도 많이 쓰여서 **시스템 Python**과 **프로젝트 Python**이 섞이면 위험했다.
- **uv = 이 모든 걸 합쳐서 파이썬을 Maven/Gradle처럼 만든 것.**
- Dev Containers = 통일이 아니라 격리. 같은 문제의 다른 처방.
