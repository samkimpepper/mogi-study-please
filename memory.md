# Memory

이 레포는 모기(사용자)의 백엔드 이직 준비용 공부 레포지토리다.

## 기록 방식

- `TIL.md`는 매일 공부한 주제를 날짜별로 한 줄 체크리스트 형태로만 기록한다.
- 사용자가 붙여넣은 문단이나 원문 요약을 `TIL.md`에 그대로 넣지 않는다.
- 공부한 심층 내용은 필요할 때 별도 디렉토리와 별도 Markdown 파일로 정리한다.
- 단, 심층 내용도 사용자가 명시적으로 "기록해줘"라고 요청했을 때만 파일에 저장한다.
- 공부 노트 Markdown은 Obsidian에서 보기 좋게 summary, note, warning, tip 같은 callout을 적극 사용한다.
- 공부 노트 Markdown에는 태그만 YAML frontmatter로 추가한다. `title`, `date`, `status` 같은 메타데이터는 기본적으로 넣지 않는다.
- 코드블록은 실제 코드 예시나 명령어처럼 그대로 실행/복사할 수 있는 내용에만 주로 사용한다. 일반 개념 흐름, 비유, 단계 설명은 코드블록으로 감싸지 않는 편을 선호한다.
- 중요한 문장, 사용자가 붙여넣은 원문 일부, 핵심 비유는 필요할 때 일반 인용문 `>`로 정리한다. 다만 인용문도 남발하지 않고, 읽는 흐름을 잡아주는 용도로만 쓴다.
- 공부 노트 Markdown 하단에는 `## 복습 질문` 섹션을 추가한다. 질문은 노트 하나당 3~5개 정도로, 본문을 다시 읽기보다 기억에서 꺼내 답할 수 있게 만든다.
- 복습 질문 아래에는 `## 한 줄 회고` 섹션을 추가하고, 사용자가 헷갈렸던 점만 짧게 남긴다.
- 각 `notes/` 주제 디렉토리에는 백지복습과 오답노트용 파일로 `blank-review.md`를 둘 수 있다. 파일명만 고정하고, 내부 형식은 사용자가 직접 써보며 정한다.
- 사용자가 문단을 붙여넣으면 우선 대화로 같이 이해하고 정리한다.
- 파일 기록은 사용자가 깨달은 점, 나중에 다시 보고 싶은 점, 또는 직접 기록해달라고 한 내용 위주로 한다.

```markdown
---
tags:
  - backend
  - spring
---
```

```markdown
## 복습 질문

- [ ] ...
- [ ] ...
- [ ] ...

## 한 줄 회고

- 헷갈렸던 점:
```

## 사용자 맥락

- 사용자는 주니어 백엔드 개발자이며 백엔드 이직을 준비하고 있다.
- 설명은 백엔드 개발자 관점에서 실무적인 비유와 구조 중심으로 해주면 좋다.
- 사용자는 가볍고 편한 말투를 쓰지만, 내용은 제대로 이해하고 싶어 한다.

## 답변 스타일

- 어려운 개념은 먼저 한 줄 정의로 잡고, 그다음 백엔드 관점의 흐름이나 예시로 풀어 설명한다.
- 사용자가 "이게 뭔 소리야?", "왜?", "무슨 뜻?"처럼 묻는 경우에는 용어 정의보다 실제 동작 흐름과 차이를 먼저 설명한다.
- RAG, 검색, 인프라 같은 주제는 `무엇인지`, `왜 쓰는지`, `무엇과 다른지`, `실무에서는 어떻게 시작할지` 순서로 설명하면 잘 맞는다.
- 사용자가 붙여넣은 긴 문단은 모두 요약하려 하기보다, 사용자가 질문한 부분 중심으로 필요한 만큼만 해석한다.
- 사용자가 "기록해줘"라고 하면 대화 중 깨달은 내용과 나중에 참고할 실무 포인트를 Markdown으로 정리한다.

## Obsidian 스타일 취향

- 사용자는 `notes/ai/claude-code-prohan-surrogate-brick-bug.md`처럼 이모지 섹션 제목, `---` 구분선, 일반 문단 중심의 읽는 맛이 있는 노트 스타일을 선호한다.
- `> [!summary]`, `> [!info]`, `> [!warning]`, `> [!tip]` 같은 callout은 완전 금지는 아니지만 남발하지 않는 편을 선호한다. 핵심 요약, 주의점, 면접 답변처럼 진짜 구분이 필요한 곳에만 쓴다.
- 노트 전체가 callout 박스 위주로 쌓이면 딱딱해 보이므로, 일반 문단과 짧은 인용문, 표, 코드블록을 적당히 섞어 정리하는 편을 선호한다.
- callout은 각진 기본 스타일보다 둥근 모서리를 선호한다.
- callout 외곽 border는 없는 스타일을 선호한다.
- callout은 기본보다 위쪽 padding이 넉넉한 스타일을 선호한다.
- 일반 인용문 `>`를 적당히 활용해 핵심 문장이나 원문 맥락이 본문에서 잘 드러나는 스타일을 선호한다.
- 코드블록이 많으면 노트가 딱딱해 보이므로, 실제 코드/명령어/출력 예시가 아니면 일반 문단이나 인용문으로 풀어 쓰는 편을 선호한다.
- 표는 줄무늬처럼 행마다 배경색이 다른 스타일을 싫어한다.
- 표는 외곽 border 없이, 내부 행 구분선만 있는 깔끔한 스타일을 선호한다.
- 표 셀은 위아래 padding이 넉넉해서 행끼리 답답하게 붙어 보이지 않는 스타일을 선호한다.
- Markdown 하이라이트 `==텍스트==`는 연한 핑크 배경과 파란 글자 스타일을 선호한다.

## Primary 테마 사용 루틴

### 필기

- `[I]` 아이디어
- `[?]` 질문
- `[!]` 중요

### 강조 규칙

- `==중요==`
- `**==위험/반박==**`
- `*==정의/개념==*`
- `***==아이디어/응용==***`

### 프로젝트

```html
<progress max="100" value="40"></progress>
```

```markdown
> [!tip] 다음 행동
> 오늘 할 것 1개만 적기
```
